<summary>1. Що таке Django і які його ключові можливості?</summary>

#### Django

Django - це високорівневий Python-фреймворк для розробки вебзастосунків, який
працює за принципом "batteries included": більшість типових задач уже
реалізовані в екосистемі фреймворку.

#### Ключові можливості Django:

1. **Швидкий старт розробки:** генерація проєкту та додатків, готова структура і
   зрозумілі конвенції.
2. **Вбудована адмін-панель:** автоматично створювана Django Admin для керування
   даними через браузер.
3. **Потужний ORM:** робота з базою даних через Python-моделі без ручного SQL у
   більшості кейсів.
4. **Система міграцій:** контроль і версіонування змін схеми БД.
5. **URL-роутинг і view-рівень:** чітке розділення маршрутизації, бізнес-логіки
   та представлення.
6. **Шаблонізатор:** вбудовані templates для рендерингу HTML з підтримкою
   наслідування шаблонів.
7. **Безпека за замовчуванням:** захист від CSRF, XSS, SQL injection,
   clickjacking та безпечні практики аутентифікації.
8. **Вбудована аутентифікація:** користувачі, групи, дозволи, сесії.
9. **Робота з формами:** Django Forms/ModelForms із валідацією на рівні сервера.
10. **Масштабованість та екосистема:** кешування, middleware, сигнали,
    інтеграція з Celery, DRF, Redis, PostgreSQL тощо.

#### Коли Django особливо доречний:

- CRM/ERP і внутрішні бізнес-системи.
- Контентні платформи, маркетплейси, кабінети користувачів.
- API + вебчастина в одному проєкті.
- Продукти, де важливі швидкість реалізації, безпека і підтримуваність коду.

#### Короткий приклад "що дає Django з коробки":

```bash
django-admin startproject config .
python manage.py startapp users
python manage.py migrate
python manage.py createsuperuser
python manage.py runserver
```

Після цього ти вже маєш базовий каркас проєкту, підключену БД, систему міграцій
та доступ до адмінки.

#### Висновок:

Django - практичний вибір для швидкої та безпечної серверної розробки, особливо
коли потрібні зріла архітектура, стабільний стек і швидкий запуск продукту.

<summary>2. Які переваги використання Django?</summary>

#### Django

Django часто обирають для комерційної розробки, коли потрібні швидкий time-to-market,
стабільна архітектура і безпечний бекенд без зайвого "склеювання" десятків
сторонніх бібліотек.

#### Ключові переваги Django:

1. **Висока швидкість розробки:** багато типових задач уже реалізовано
   (auth, admin, forms, ORM, sessions, cache).
2. **Безпека "з коробки":** вбудовані механізми проти CSRF, XSS, SQL injection,
   clickjacking, безпечна робота з паролями.
3. **Чітка структура проєкту:** зрозумілі конвенції спрощують онбординг команди
   та підтримку коду в довгостроковій перспективі.
4. **Потужний ORM і міграції:** швидка робота з БД, контроль еволюції схеми,
   хороша підтримка PostgreSQL та інших реляційних СУБД.
5. **Адмін-панель для бізнесу:** Django Admin дозволяє швидко дати команді
   контенту/операцій робочий інтерфейс без окремої фронтенд-розробки.
6. **Гарна масштабованість архітектури:** додатки, middleware, signals, celery,
   кеші й черги легко інтегруються в production-сценарії.
7. **Зріла екосистема:** DRF, django-allauth, django-filter, django-celery-beat
   та інші перевірені інструменти.
8. **Придатність до enterprise-проєктів:** передбачуваність релізів, велика
   спільнота, якісна документація і стабільні практики.

#### Практична цінність для команди:

- Менше інфраструктурного коду на старті.
- Менше ризиків безпекових помилок у базовій реалізації.
- Швидший запуск MVP і зрозуміший шлях до масштабування.
- Краща підтримуваність при рості команди та коду.

#### Коли переваги Django особливо відчутні:

- B2B SaaS, CRM/ERP, кабінети користувачів, маркетплейси.
- Проєкти з великою кількістю бізнес-логіки й ролей доступу.
- Команди, яким важлива стабільність і прогнозований розвиток продукту.

#### Висновок:

Перевага Django в тому, що він дає швидкий бізнес-результат без компромісів у
безпеці та підтримуваності коду.

<summary>3. У чому різниця між MVC і MVT архітектурою?</summary>

#### Django

У Django частіше говорять про **MVT (Model-View-Template)**, тоді як у багатьох
інших фреймворках використовують термін **MVC (Model-View-Controller)**.
Концептуально це дуже близькі підходи, різниця здебільшого в ролях і назвах.

#### MVC (класично):

1. **Model** - робота з даними та бізнес-логікою.
2. **View** - відображення даних (UI / presentation layer).
3. **Controller** - обробляє вхідний запит, керує потоком між Model і View.

#### MVT у Django:

1. **Model** - моделі даних та доступ до БД через ORM.
2. **View** - обробка HTTP-запиту, виконання бізнес-логіки, підготовка відповіді.
3. **Template** - шар відображення (HTML + template syntax).

#### Як співвідносяться MVC і MVT:

- `Model (MVC)` ≈ `Model (Django MVT)`
- `View (MVC)` ≈ `Template (Django MVT)`
- `Controller (MVC)` ≈ `View + URL dispatcher (Django MVT)`

Тобто в Django роль "контролера" розділена між `urls.py` (маршрутизація) і
`views.py` (обробка запиту).

#### Простий приклад потоку в Django (MVT):

1. Запит приходить на URL, наприклад `/articles/`.
2. `urls.py` направляє його у відповідний `view`.
3. `view` отримує дані з `Model`.
4. `view` передає дані в `Template`.
5. `Template` рендерить HTML і повертається `HttpResponse`.

#### Висновок:

- Django не "порушує" MVC, а реалізує його у власному варіанті MVT.
- Ключова практична різниця: у Django `view` - це логіка обробки запиту, а
  візуальне представлення лежить у `template`.

<summary>4. Чому Django називають loosely coupled фреймворком?</summary>

#### Django

Django називають **loosely coupled** (слабо зв’язаним) фреймворком, тому що його
компоненти можна змінювати, розширювати або замінювати незалежно один від одного,
без переписування всього застосунку.

#### Що це означає на практиці:

1. **Компоненти відокремлені за відповідальністю:** URL-маршрути, view, шаблони,
   моделі, форми, middleware працюють разом, але не "злиті" в один монолітний
   шар логіки.
2. **Плагінна архітектура додатків:** проєкт складається з `apps`, які можна
   підключати/відключати через `INSTALLED_APPS`.
3. **Можливість заміни окремих частин:** наприклад, шаблонізатор, user model,
   бекенд кешу, email backend, storage backend.
4. **Конфігурація через settings:** поведінка системи керується налаштуваннями,
   а не жорстко зашитою логікою.
5. **Розширення через middleware і signals:** можна додавати функціональність
   без зміни ядра бізнес-логіки.

#### Типові приклади loosely coupled підходу в Django:

- Замінити SQLite на PostgreSQL лише через налаштування `DATABASES`.
- Додати Redis-кеш, не переписуючи view і моделі.
- Перейти з локального файлового сховища на S3-compatible storage через backend.
- Винести auth у кастомну модель користувача (`AUTH_USER_MODEL`) без зламу
  архітектури всього проєкту.

#### Чому це важливо для команди:

- Простіше масштабувати продукт по модулях.
- Легше тестувати окремі частини системи.
- Менше ризиків при рефакторингу.
- Легше підтримувати код у довгому життєвому циклі проєкту.

#### Висновок:

Loosely coupled у Django - це про інженерну гнучкість: ти можеш еволюціонувати
проєкт поступово, змінюючи конкретні підсистеми, а не перебудовуючи все одразу.

</details>

<summary>5. У чому різниця між Django і FastAPI?</summary>

#### Django

Django і FastAPI вирішують різні задачі. Обидва фреймворки сильні, але їх
вибирають під різні типи продуктів і команди.

#### Ключова різниця:

- **Django** - повнофункціональний вебфреймворк "з коробки" (ORM, admin, auth,
  templates, forms, sessions, middleware).
- **FastAPI** - легкий і дуже швидкий ASGI-фреймворк, орієнтований насамперед
  на побудову API з типізацією та автоматичною документацією.

#### Порівняння Django vs FastAPI:

1. **Призначення**
- Django: повноцінні вебсистеми й API в одному проєкті.
- FastAPI: API-first, мікросервіси, high-concurrency сервіси.

2. **Швидкість старту проєкту**
- Django: дуже швидко для CRUD/адмінок/кабінетів.
- FastAPI: швидко для API, але більше рішень треба зібрати самостійно.

3. **Продуктивність**
- Django: достатня для більшості бізнес-сценаріїв; класично WSGI, але підтримує
  ASGI-режим.
- FastAPI: зазвичай вища продуктивність на I/O-навантаженні завдяки async-first
  підходу.

4. **ORM і робота з БД**
- Django: зрілий вбудований ORM + міграції.
- FastAPI: ORM не вбудований; зазвичай SQLAlchemy/SQLModel/Tortoise.

5. **Адмінка і back-office**
- Django: потужна Django Admin "з коробки".
- FastAPI: адмінку потрібно будувати або інтегрувати окремо.

6. **Валідація і схеми**
- Django: форми/DRF serializers.
- FastAPI: Pydantic-моделі як стандарт де-факто.

7. **Документація API**
- Django: зазвичай через DRF + Swagger/OpenAPI пакети.
- FastAPI: OpenAPI/Swagger UI генерується автоматично.

8. **Екосистема**
- Django: дуже широка для вебпродуктів (CMS, auth, адмін-патерни).
- FastAPI: сильна в API/мікросервісах, сучасний async-стек.

#### Коли обирати Django:

- Потрібен "all-in-one" бекенд із швидким бізнес-результатом.
- Є складна рольова модель, адмінка, багато CRUD-сутностей.
- Команді важливі стабільні конвенції й довгострокова підтримуваність.

#### Коли обирати FastAPI:

- Потрібен API-first сервіс із високою конкурентністю запитів.
- Пріоритет: async інтеграції (зовнішні API, черги, стрімінг, websocket-сценарії).
- Команда хоче максимально легкий і контрольований стек.

#### Висновок:

- **Django** - оптимальний для швидкого запуску бізнес-функціоналу і комплексних
  вебсистем.
- **FastAPI** - оптимальний для продуктивних сучасних API та мікросервісних
  сценаріїв.
- У великих продуктах ці фреймворки часто співіснують: Django як core/admin,
  FastAPI як окремі API-сервіси.

</details>

<summary>6. Які недоліки та обмеження Django порівняно з іншими фреймворками?</summary>

#### Django

Django - сильний фреймворк, але не універсально найкращий для всіх типів
проєктів. Його обмеження зазвичай проявляються в специфічних технічних сценаріях.

#### Основні недоліки та обмеження Django:

1. **Важчий "вхід" для дуже простих сервісів**
- Для маленького мікросервісу Django може бути надлишковим через обсяг
  вбудованих компонентів.

2. **Менш "native async-first" UX, ніж у FastAPI**
- Django підтримує async, але історично багато частин екосистеми орієнтовані
  на sync-підхід.
- Для високонавантажених I/O API FastAPI/Starlette інколи дають простіший шлях.

3. **ORM не ідеальний для складного SQL**
- Django ORM потужний для більшості задач, але для дуже складних аналітичних
  запитів або DB-специфічних оптимізацій усе одно доводиться писати raw SQL.

4. **Жорсткі конвенції**
- Конвенції Django - це плюс для команди, але для інженерів, які хочуть
  максимально "тонкий" контроль, вони можуть здаватися обмеженням.

5. **Монолітний стиль за замовчуванням**
- Django природно штовхає до модульного моноліту, а не до дрібних мікросервісів.
- Для microservice-first архітектури потрібна чітка дисципліна розмежування.

6. **Додаткові кроки для сучасного API-first DX**
- Для повноцінного REST API зазвичай додають DRF, для OpenAPI - додаткові
  інструменти, для realtime - окремий стек (наприклад, Channels).

7. **Продуктивність "із коробки" не завжди максимальна**
- Без кешування, індексів БД, оптимізації ORM-запитів і правильного профілювання
  великі системи можуть працювати повільно.

#### Типові помилки, які плутають із "мінусами Django":

- N+1 запити через неправильне використання ORM.
- Відсутність кеш-стратегії.
- Невдала модульна декомпозиція додатків.
- Змішування бізнес-логіки у view без сервісного шару.

Це частіше питання архітектури команди, а не слабкість самого фреймворку.

#### Коли Django може бути не найкращим вибором:

- Дуже легкий API-сервіс з мінімальною бізнес-логікою.
- High-throughput async API як основна вимога.
- Система, де потрібен максимально кастомний low-level контроль без "важкого"
  framework core.

#### Висновок:

Обмеження Django реальні, але прогнозовані. Для більшості бізнес-застосунків
його переваги (швидкість розробки, безпека, підтримуваність) переважають.
Головне - обирати фреймворк під контекст задачі, а не "за трендом".

</details>

<summary>7. Як працює Django (життєвий цикл запиту–відповіді)?</summary>

#### Django

Життєвий цикл у Django - це послідовність кроків від моменту, коли HTTP-запит
приходить на сервер, до формування фінальної HTTP-відповіді для клієнта.

#### Послідовність обробки запиту:

1. **Клієнт надсилає HTTP-запит**
- Браузер або API-клієнт звертається до Django-застосунку.

2. **WSGI/ASGI сервер передає запит у Django**
- Наприклад: Gunicorn/Uvicorn + Nginx.

3. **Request проходить через middleware (request phase)**
- Middleware можуть змінювати/збагачувати запит:
  аутентифікація, сесії, locale, security-перевірки тощо.

4. **URL resolver шукає відповідний маршрут**
- Django зіставляє URL із правилами в `urls.py`.
- Якщо маршрут не знайдено -> `404 Not Found`.

5. **Викликається view**
- View отримує `request`, виконує бізнес-логіку:
  читає/записує дані, перевіряє доступ, формує контекст.

6. **Робота з моделями та БД (за потреби)**
- Через ORM (`Model.objects...`) або raw SQL для складних кейсів.

7. **Формування відповіді**
- HTML через `render(...)` + template,
- або JSON через `JsonResponse`,
- або redirect / file response.

8. **Response проходить middleware (response phase)**
- Middleware можуть додати заголовки, cookies, кеш-контроль, логування.

9. **Відповідь повертається клієнту**
- Браузер рендерить сторінку або клієнт обробляє API-відповідь.

#### Що відбувається при помилках:

- `Http404` або відсутній маршрут -> 404-сторінка.
- Неперехоплені винятки -> 500-сторінка.
- Обробка винятків теж проходить через middleware ланцюжок.

#### Стислий ланцюжок:

`Client -> Server (WSGI/ASGI) -> Middleware (request) -> URLconf -> View -> Model/ORM -> Template/Serializer -> Middleware (response) -> Client`

#### Висновок:

Сила Django в передбачуваному request/response pipeline: легко дебажити,
тестувати і масштабувати, бо кожен етап має чітку відповідальність.

</details>

<details>
<summary>8. Як створити новий проєкт Django?</summary>

#### Django

Новий Django-проєкт створюють у кілька стандартних кроків: підготовка
середовища, встановлення Django, генерація проєкту, міграції та запуск сервера.

#### Покроково:

1. **Створи директорію проєкту**

```bash
mkdir my_django_project
cd my_django_project
```

2. **Створи та активуй віртуальне середовище**

```bash
python -m venv .venv
source .venv/bin/activate        # Linux/macOS
# .venv\Scripts\activate         # Windows (PowerShell/CMD)
```

3. **Встанови Django**

```bash
pip install django
```

4. **Згенеруй проєкт**

```bash
django-admin startproject config .
```

Після цього з’являться ключові файли:
- `manage.py`
- `config/settings.py`

5. **Застосуй початкові міграції**

<summary>9. Чому важливо використовувати віртуальне середовище у роботі з Django?</summary>

#### Django

Віртуальне середовище (`venv`) ізолює залежності конкретного проєкту від
глобального Python у системі. Для Django це критично, бо проєкти часто мають
різні версії бібліотек і різні вимоги до сумісності.

#### Чому це важливо:

1. **Ізоляція залежностей**
- Кожен проєкт використовує власні версії `Django`, `djangorestframework`,
  `psycopg`, `celery` тощо, без конфліктів з іншими проєктами.

2. **Відтворюваність середовища**
- Легко зафіксувати стан залежностей у `requirements.txt` і відтворити його на
  іншій машині або сервері.

3. **Стабільність деплою**
- Менше шансів отримати "works on my machine", коли локально одна версія пакета,
  а в CI/production - інша.

4. **Безпечні оновлення**
- Можна тестувати апгрейд Django чи сторонніх пакетів без ризику зламати інші
  локальні проєкти.

5. **Чистота системного Python**
- Системні інструменти ОС не залежать від пакетів, які ти ставиш для розробки.

#### Що трапляється без `venv`:

- Конфлікти версій між проєктами.
- Непередбачувані імпорт-помилки після `pip install` нового пакета.
- Складний дебаг проблем у CI/CD через різні середовища.
- Зламані локальні інструменти через глобальні оновлення.

#### Мінімальна практика для команди:

1. Створювати `.venv` у корені кожного репозиторію.
2. Завжди активувати середовище перед `pip install` і `manage.py` командами.
3. Фіксувати залежності в `requirements.txt` (або `pyproject.toml` + lock file).
4. Додавати `.venv/` у `.gitignore`.

#### Висновок:

`venv` у Django - не опція, а базовий стандарт інженерної дисципліни. Він
забезпечує передбачуваність, стабільність і нормальний командний workflow.

<summary>10. Яка структура проєкту Django і призначення основних файлів: manage.py, settings.py, urls.py?</summary>

#### Django

Типовий Django-проєкт має дві ключові частини: корінь репозиторію та пакет
конфігурації (часто `config/`), де лежать глобальні налаштування застосунку.

#### Базова структура проєкту:

```text
my_project/
├── manage.py
├── config/
│   ├── __init__.py
│   ├── settings.py
│   ├── urls.py
│   ├── asgi.py
│   └── wsgi.py
├── app1/
├── app2/
└── requirements.txt
```

#### Призначення основних файлів:

1. **manage.py**
- CLI-точка входу для локальної роботи з проєктом.
- Через нього запускають команди:
  `runserver`, `migrate`, `makemigrations`, `createsuperuser`, `test`, `shell`.
- Автоматично вказує Django, який settings-модуль використовувати.

2. **settings.py**
- Центральна конфігурація проєкту.
- Тут задаються:
  `INSTALLED_APPS`, `MIDDLEWARE`, `DATABASES`, `TEMPLATES`, `CACHES`,
  `STATIC_URL`, `MEDIA_URL`, `AUTH_PASSWORD_VALIDATORS`, `TIME_ZONE` тощо.
- Для production зазвичай розділяють налаштування на модулі:
  `settings/base.py`, `settings/dev.py`, `settings/prod.py`.

3. **urls.py**
- Глобальна карта маршрутів (URL dispatcher).
- Приймає запитаний шлях і делегує його у відповідний app через `include()`.
- Також тут часто підключають `admin/`, health-check маршрути та API роутінг.

#### Як це працює разом:

1. `manage.py` запускає Django-команду.
2. Django завантажує `settings.py`.
3. HTTP-запит проходить через middleware.
4. `urls.py` визначає, яке `view` обробить запит.
5. `view` повертає відповідь (HTML/JSON/redirect/file).

#### Практичні рекомендації техліда:

- Тримай `urls.py` "тонким": основна маршрутизація має жити на рівні app.
- Не перетворюй `settings.py` на "звалище" - використовуй модульну структуру.
- Усі секрети (ключі, паролі, токени) зберігай у змінних середовища, не в коді.
- Винось бізнес-логіку з `views.py` у сервісний шар, щоб спростити тести.

#### Висновок:

`manage.py`, `settings.py` і `urls.py` - це операційне ядро Django-проєкту:
керування, конфігурація і маршрутизація. Чітка організація цих файлів напряму
впливає на масштабованість і підтримуваність системи.

</details>

<summary>11. Для чого використовуються Django-додатки (apps) і як організувати проєкт із кількома додатками?</summary>

#### Django

У Django `app` - це ізольований функціональний модуль із власними моделями,
view, URL, шаблонами, тестами та (за потреби) API-шаром. Apps потрібні для
розділення доменів і контрольованого росту коду.

#### Для чого використовуються Django apps:

1. **Модульність**
- Кожен app відповідає за конкретний бізнес-контекст (наприклад: `users`,
  `billing`, `orders`, `notifications`).

2. **Повторне використання**
- App можна перенести в інший проєкт або перевикористати внутрішньо.

3. **Локалізація змін**
- Зміни в одному домені менше впливають на інші модулі.

4. **Краща підтримуваність**
- Команда легше орієнтується в коді, коли відповідальність чітко розділена.

5. **Простіший тестинг**
- Тести пишуться й запускаються по доменах, а не по хаотичному моноліту.

#### Як організувати проєкт із кількома apps:

1. **Діли за доменом, а не за технічним типом**
- Добре: `users`, `catalog`, `checkout`, `payments`.
- Погано: `models_app`, `views_app`, `utils_app`.

2. **Кожен app має власні `urls.py`**
- У кореневому `config/urls.py` лише підключення через `include()`.

3. **Тримай app автономним**
- В app мають бути свої `models.py`, `views.py`, `tests.py`, `admin.py`,
  `apps.py`, `migrations/`.

4. **Уникай циклічних залежностей між apps**
- Для міжмодульної взаємодії краще використовувати сервісний шар або чіткі
  інтерфейси, а не прямі імпорти "все в усе".

5. **Винось спільний код у окремий `common/core` модуль**
- Але тільки справді спільний; не роби з цього "смітник".

#### Приклад структури багатомодульного проєкту:

```text
project/
├── config/
│   ├── settings/
│   └── urls.py
├── apps/
│   ├── users/
│   ├── products/
│   ├── orders/
│   └── payments/
└── common/
    ├── db/
    ├── exceptions/
    └── utils/
```

#### Підключення app-маршрутів:

```python
# config/urls.py
from django.urls import include, path

urlpatterns = [
    path("users/", include("apps.users.urls")),
    path("orders/", include("apps.orders.urls")),
]
```

#### Типові анти-патерни:

- Один "гігантський" app на весь проєкт.
- Розділення по технічних шарах замість бізнес-домену.
- Масовий shared-код без власника та контрактів.
- Крос-імпорти між apps без контролю залежностей.

#### Висновок:

Django apps - це базовий інструмент масштабування коду. Якщо розкладати систему
по доменах із чіткими межами, проєкт довше залишається керованим і стабільним.

<summary>12. Як працює маршрутизація URL у Django?</summary>

#### Django

Маршрутизація URL у Django визначає, яке `view` повинно обробити конкретний
HTTP-запит. За це відповідає URL dispatcher, налаштований у `urls.py`.

#### Як працює URL routing по кроках:

1. **Запит приходить у Django**
- Наприклад: `GET /articles/42/`.

2. **Django читає кореневий `urlpatterns`**
- Зазвичай у `config/urls.py`.

3. **Послідовний матчинг маршрутів**
- Django перевіряє патерни зверху вниз.
- Перший збіг перемагає.

4. **За потреби викликається `include()`**
- Частина маршруту делегується в `urls.py` конкретного app.

5. **Параметри URL передаються у view**
- Наприклад `articles/<int:id>/` передає `id` як аргумент.

6. **View формує response**
- HTML, JSON, redirect або файл.

7. **Якщо збігу немає -> 404**
- Django повертає `Not Found`.

#### Базовий приклад:

```python
# config/urls.py
from django.contrib import admin
from django.urls import include, path

urlpatterns = [
    path("admin/", admin.site.urls),
    path("articles/", include("apps.articles.urls")),
]
```

```python
# apps/articles/urls.py
from django.urls import path
from . import views

urlpatterns = [
    path("", views.article_list, name="article_list"),
    path("<int:article_id>/", views.article_detail, name="article_detail"),
]
```

#### Конвертери в `path()`:

- `<int:id>` -> ціле число
- `<str:slug>` -> рядок без `/`
- `<slug:slug>` -> slug-формат
- `<uuid:id>` -> UUID
- `<path:subpath>` -> довільний шлях, включно зі `/`

#### Практичні правила для підтримуваності:

1. Тримай кореневий `urls.py` коротким, делегуй у apps.
2. Використовуй `name=` для кожного маршруту (зручно для `reverse()` і шаблонів).
3. Дотримуйся стабільної структури API/web URL (версіонування, префікси).
4. Порядок маршрутів має значення: специфічні патерни став вище за загальні.

#### Висновок:

URL routing у Django - це керований механізм диспетчеризації запитів. Якісно
спроєктовані маршрути роблять код передбачуваним, зручним для масштабування і
безпечним для рефакторингу.

<summary>13. Різниця між path() і re_path().</summary>

#### Django

`path()` і `re_path()` - це два способи описувати URL-маршрути в Django.
Вони вирішують одну задачу, але мають різний рівень складності й гнучкості.

#### `path()`:

- Використовує зрозумілий, декларативний синтаксис з конвертерами.
- Підходить для більшості звичайних маршрутів.
- Краще читається і легше підтримується командою.

Приклад:

```python
from django.urls import path
from . import views

urlpatterns = [
    path("posts/<int:post_id>/", views.post_detail, name="post_detail"),
    path("users/<slug:username>/", views.user_profile, name="user_profile"),
]
```

#### `re_path()`:

- Використовує регулярні вирази (regex).
- Дає максимальну гнучкість для нестандартних URL-патернів.
- Складніше читається, вищий ризик помилок у regex.

Приклад:

```python
from django.urls import re_path
from . import views

urlpatterns = [
    re_path(r"^archive/(?P<year>[0-9]{4})/$", views.archive_year),
    re_path(r"^file/(?P<path>.+\.(pdf|docx))/$", views.file_view),
]
```

#### Коли що використовувати:

1. **Використовуй `path()` за замовчуванням**
- Це стандартний і рекомендований підхід у сучасному Django.

2. **Використовуй `re_path()` тільки коли `path()` недостатньо**
- Наприклад, складна валідація формату URL на рівні маршруту або legacy URL-схеми.

#### Порівняння:

- Простота: `path()` > `re_path()`
- Гнучкість: `re_path()` > `path()`
- Підтримуваність у команді: зазвичай `path()` кращий

#### Практична порада:

Не ускладнюй URL regex-ами без потреби. Якщо можна розв’язати задачу через
конвертери `path()` + перевірку у view/serializer/form, це зазвичай кращий шлях.

#### Висновок:

`path()` - основний інструмент для 90% маршрутів. `re_path()` - точкове рішення
для особливих випадків, де потрібна regex-гнучкість.

<summary>14. Як працюють URL-конфігурації та include()?</summary>

#### Django

URL-конфігурація в Django - це набір правил (`urlpatterns`), які зіставляють
шлях запиту з конкретним `view`. Функція `include()` дозволяє розбивати велику
маршрутизацію на модулі рівня app.

#### Що таке URL-конфігурація:

- Це Python-модуль (зазвичай `urls.py`) зі списком маршрутів.
- Кожен маршрут описується через `path()` або `re_path()`.
- Django проходить список зверху вниз і бере перший збіг.

#### Для чого потрібен `include()`:

1. **Декомпозиція маршрутів**
- Кореневий `urls.py` лишається коротким.

2. **Модульність по apps**
- Кожен додаток має власні маршрути у своєму `urls.py`.

3. **Краща підтримуваність**
- Команда працює з локальним routing-контекстом конкретного домену.

#### Приклад організації:

```python
# config/urls.py
from django.contrib import admin
from django.urls import include, path

urlpatterns = [
    path("admin/", admin.site.urls),
    path("blog/", include("apps.blog.urls")),
    path("accounts/", include("apps.accounts.urls")),
    path("api/v1/", include("apps.api.urls")),
]
```

```python
# apps/blog/urls.py
from django.urls import path
from . import views

app_name = "blog"

urlpatterns = [
    path("", views.post_list, name="list"),
    path("<slug:slug>/", views.post_detail, name="detail"),
]
```

#### Як працює ланцюжок з `include()`:

1. Запит `/blog/django-routing/` приходить у кореневий `config/urls.py`.
2. Префікс `blog/` збігається.
3. Django передає залишок шляху (`django-routing/`) у `apps.blog.urls`.
4. Там знаходиться відповідний маршрут і викликається потрібний `view`.

#### Namespace і `app_name`:

- `app_name` + `name` маршруту дає іменований namespace.
- Це дозволяє безконфліктно робити reverse:
  `reverse("blog:detail", kwargs={"slug": "django-routing"})`.

#### Практичні рекомендації:

1. Завжди використовуй `include()` для app-рівня.
2. Додавай `app_name` у кожен app `urls.py`.
3. Тримай префікси стабільними (`api/v1/`, `admin/`, `auth/`).
4. Не дублюй бізнес-логіку в URL-шарі - лише маршрутизація.

#### Висновок:

URL-конфігурації + `include()` - це основа масштабованої маршрутизації в Django:
чіткі межі між додатками, чистий кореневий routing і передбачувана навігація.

</details>

<summary>15. Що таке view у Django?</summary>

#### Django

`View` у Django - це обробник HTTP-запиту. Саме view приймає `request`,
виконує потрібну логіку і повертає `HttpResponse` (HTML, JSON, redirect, файл).

#### Роль view у request/response циклі:

1. Отримати запит після URL-matching.
2. Перевірити вхідні параметри (path/query/body/files).
3. Викликати бізнес-логіку та/або ORM.
4. Підготувати дані для відповіді.
5. Повернути валідний HTTP response.

#### Які бувають view:

1. **FBV (Function-Based View)**
- Звичайна Python-функція.
- Підходить для простої або кастомної логіки.

2. **CBV (Class-Based View)**
- Клас із методами `get()`, `post()` тощо.
- Краще масштабується для типових CRUD-сценаріїв, особливо з Generic Views.

#### Приклад FBV:

```python
from django.http import HttpResponse, JsonResponse
from django.shortcuts import render

def healthcheck(request):
    return HttpResponse("OK")

def home(request):
    return render(request, "home.html", {"title": "Головна"})

def api_status(request):
    return JsonResponse({"status": "ok"})
```

#### Що важливо памʼятати:

- View не повинно ставати "god function".
- Складну бізнес-логіку краще виносити у сервісний шар.
- View має бути тонким: orchestration, валідація, виклик сервісів, response.

#### Типові помилки:

- Прямо у view писати важку доменну логіку на сотні рядків.
- Робити багато дубльованої логіки в різних view.
- Ігнорувати перевірки доступу/авторизації на рівні endpoint.

#### Висновок:

View - це точка, де web-протокол зустрічається з бізнес-логікою. Якісно
спроєктовані view роблять код передбачуваним, тестованим і легким у підтримці.

</details>

<details>
<summary>16. Різниця між функціональними (FBV) і класовими (CBV) представленнями.</summary>

#### Django

У Django є два основні стилі представлень: **FBV** (function-based views) і
**CBV** (class-based views). Обидва підходи валідні, різниця - у способі
організації логіки та масштабуванні коду.

#### FBV (Function-Based Views):

- View описується як функція.
- Логіка запиту/відповіді читається зверху вниз.
- Простий і прозорий підхід для невеликих endpoint.

Приклад:

```python
from django.http import JsonResponse

def profile_detail(request, user_id):
    if request.method == "GET":
        return JsonResponse({"user_id": user_id})
    return JsonResponse({"error": "Method not allowed"}, status=405)
```

#### CBV (Class-Based Views):

- View описується як клас (методи `get`, `post`, `put`, `delete` тощо).
- Можна перевикористовувати логіку через наслідування і mixins.
- Добре підходить для CRUD, списків, форм, detail/create/update/delete сценаріїв.

Приклад:

```python
from django.http import JsonResponse
from django.views import View

class ProfileDetailView(View):
    def get(self, request, user_id):
        return JsonResponse({"user_id": user_id})
```

#### Ключові відмінності:

1. **Простота входу**
- FBV: простіше для старту.
- CBV: потребує розуміння ієрархії класів і dispatch-механіки.

2. **Масштабування**
- FBV: добре для кастомних, коротких обробників.
- CBV: краще для повторюваних патернів і великого проєкту.

3. **Перевикористання коду**
- FBV: через helper-функції/декоратори.
- CBV: через inheritance, mixins, generic views.

4. **Читабельність**
- FBV: часто очевидніший локально.
- CBV: може бути складнішим через "приховану" поведінку базових класів.

5. **Тестування**
- Обидва підходи добре тестуються, якщо логіка не "зашита" у view.

#### Коли обирати FBV:

- Endpoint простий і сильно кастомний.
- Потрібна максимальна прозорість коду для швидкого дебагу.
- Немає вигоди від class hierarchy.

#### Коли обирати CBV:

- Багато однотипних CRUD-екранів/endpoint.
- Потрібні mixins, permission-класи, повторювані патерни.
- Команда працює з Generic CBV і підтримує єдиний стиль.

#### Висновок:

FBV і CBV - це не "правильно/неправильно", а питання контексту. У production
проєктах часто використовують обидва підходи: FBV для точкових задач, CBV для
структурованих повторюваних сценаріїв.

</details>

<summary>17. Коли краще використовувати CBV та generic class-based views?</summary>

#### Django

CBV і generic CBV варто використовувати тоді, коли в проєкті багато повторюваних
web/API-патернів і потрібна стандартизація поведінки між endpoint.

#### Коли CBV справді виграють:

1. **CRUD-сценарії**
- Списки, детальні сторінки, створення, оновлення, видалення.
- Типові класи: `ListView`, `DetailView`, `CreateView`, `UpdateView`, `DeleteView`.

2. **Багато однотипних endpoint**
- Коли різниця між ендпоінтами лише в моделі, формі або permission-правилах.

3. **Потреба в повторному використанні логіки**
- Через mixins (`LoginRequiredMixin`, `PermissionRequiredMixin`, власні mixins).

4. **Єдиний стиль у великій команді**
- CBV полегшують уніфікацію підходів і зменшують copy-paste.

5. **Розширення базової поведінки через override**
- Наприклад `get_queryset()`, `get_context_data()`, `form_valid()`,
  `get_success_url()`.

#### Коли generic CBV особливо доречні:

- Адміноподібні інтерфейси.
- Back-office сторінки.
- Стандартні форми та контентні розділи.
- Швидка реалізація MVP без втрати структури коду.

#### Приклад generic CBV:

```python
from django.urls import reverse_lazy
from django.views.generic import ListView, CreateView
from .models import Article
from .forms import ArticleForm

class ArticleListView(ListView):
    model = Article
    template_name = "articles/list.html"
    context_object_name = "articles"
    paginate_by = 20

class ArticleCreateView(CreateView):
    model = Article
    form_class = ArticleForm
    template_name = "articles/form.html"
    success_url = reverse_lazy("articles:list")
```

#### Коли краще НЕ брати generic CBV:

1. Логіка сильно нестандартна і не вкладається в CRUD-парадигму.
2. Endpoint має складний оркестраційний сценарій (кілька сервісів/інтеграцій).
3. Через надмірний override generic клас стає менш читабельним за простий FBV.

#### Практичне правило техліда:

- Починай з generic CBV для стандартних кейсів.
- Якщо view "ламає" модель generic-поведінки, переходь на звичайний CBV або FBV.
- Не тримайся за generic-клас лише "бо так красивіше" - пріоритетом є
  підтримуваність і прозорість логіки.

#### Висновок:

CBV/generic CBV - це інструмент прискорення і стандартизації. Вони найкращі для
повторюваних сценаріїв, але для нетипової бізнес-логіки краще обирати простіший
та явніший підхід.

</details>

<summary>18. Що таке mixins у CBV і як їх використовувати?</summary>

#### Django

`Mixins` у CBV - це допоміжні класи з повторно використовуваною поведінкою, які
додаються до view через множинне наслідування. Вони дозволяють будувати гнучкі
представлення без дублювання коду.

#### Навіщо потрібні mixins:

1. **Повторне використання логіки**
- Авторизація, permissions, фільтрація queryset, загальний контекст тощо.

2. **Композиція поведінки**
- View складається з кількох маленьких "цеглинок", а не одного великого класу.

3. **Зменшення copy-paste**
- Один mixin можна використовувати в десятках view.

#### Найчастіші built-in mixins у Django:

- `LoginRequiredMixin` - доступ лише для авторизованих.
- `PermissionRequiredMixin` - перевірка конкретних permission.
- `UserPassesTestMixin` - кастомна перевірка доступу.
- `ContextMixin` - додавання даних у контекст шаблону.

#### Приклад використання built-in mixins:

```python
from django.contrib.auth.mixins import LoginRequiredMixin, PermissionRequiredMixin
from django.views.generic import ListView
from .models import Order

class OrderListView(LoginRequiredMixin, PermissionRequiredMixin, ListView):
    model = Order
    permission_required = "orders.view_order"
    template_name = "orders/list.html"
```

#### Приклад кастомного mixin:

```python
class StaffRequiredMixin:
    def dispatch(self, request, *args, **kwargs):
        if not request.user.is_staff:
            from django.http import HttpResponseForbidden
            return HttpResponseForbidden("Доступ заборонено")
        return super().dispatch(request, *args, **kwargs)
```

```python
class DashboardView(StaffRequiredMixin, TemplateView):
    template_name = "dashboard.html"
```

#### Порядок mixins має значення:

- Django використовує MRO (Method Resolution Order).
- Зазвичай "обмежуючі" mixins (`LoginRequiredMixin`) ставлять лівіше.
- `View`/generic class ідуть правіше.

#### Практичні правила:

1. Один mixin - одна відповідальність.
2. Не перевантажуй mixins бізнес-логікою.
3. Уникай глибокої ієрархії наслідування, яка ускладнює дебаг.
4. Для складного доменного коду використовуй сервіси, а не mixins.

#### Типові помилки:

- Занадто "розумні" mixins, які змінюють поведінку непередбачувано.
- Конфлікти методів через неправильний порядок наслідування.
- Дублювання однакових перевірок у багатьох view замість спільного mixin.

#### Висновок:

Mixins - потужний інструмент у CBV для композиції і повторного використання.
Вони дають максимальну користь, коли маленькі, чіткі за відповідальністю і
використовуються дисципліновано.

</details>

<summary>19. Як обробляти HTTP-методи (GET, POST)?</summary>

#### Django

У Django HTTP-методи обробляють у view: `GET` зазвичай читає/показує дані,
`POST` - створює або змінює стан системи. Правильне розділення методів - основа
безпечного й передбачуваного API та web-flow.

#### Обробка у FBV:

```python
from django.http import JsonResponse

def contact_view(request):
    if request.method == "GET":
        return JsonResponse({"form": "render contact form"})

    if request.method == "POST":
        # читання request.POST, валідація, збереження
        return JsonResponse({"status": "created"}, status=201)

    return JsonResponse({"error": "Method not allowed"}, status=405)
```

#### Обробка у CBV:

```python
from django.http import JsonResponse
from django.views import View

class ContactView(View):
    def get(self, request):
        return JsonResponse({"form": "render contact form"})

    def post(self, request):
        # валідація та збереження
        return JsonResponse({"status": "created"}, status=201)
```

#### Обмеження дозволених методів (декоратори):

```python
from django.views.decorators.http import require_GET, require_POST, require_http_methods

@require_GET
def list_articles(request):
    ...

@require_POST
def create_article(request):
    ...

@require_http_methods(["GET", "POST"])
def article_endpoint(request):
    ...
```

#### Практичні правила:

1. **GET**
- Не має змінювати стан системи.
- Безпечний і кешований у багатьох сценаріях.

2. **POST**
- Використовується для створення/дії, що змінює дані.
- Для HTML-форм обовʼязково CSRF-токен.

3. **Валідація**
- Завжди валідуй вхідні дані (Forms/Serializer/сервісний шар).

4. **Коди відповідей**
- `200` для успішного читання,
- `201` для створення,
- `400` для помилок валідації,
- `405` для недозволеного методу.

#### Типові помилки:

- Робити зміну даних через GET (порушення HTTP-семантики).
- Змішувати логіку GET і POST в один "хаотичний" блок.
- Не повертати коректні status code.

#### Висновок:

Обробка GET/POST у Django має бути явною: кожен метод - окрема відповідальність,
чітка валідація і правильні HTTP-коди. Це покращує безпеку, інтеграції та
підтримуваність коду.

</details>

<summary>20. Як реалізувати перенаправлення (redirect)?</summary>

#### Django

Перенаправлення (`redirect`) у Django використовується, коли після дії потрібно
відправити користувача на іншу сторінку або endpoint.

#### Основний спосіб:

```python
from django.shortcuts import redirect
```

`redirect(...)` повертає HTTP-відповідь із кодом 302 (тимчасове
перенаправлення) за замовчуванням.

#### Варіанти використання:

1. **На абсолютний/відносний URL**

```python
return redirect("/dashboard/")
```

2. **На іменований маршрут (рекомендовано)**

```python
return redirect("orders:list")
```

3. **На маршрут із параметрами**

```python
return redirect("orders:detail", order_id=order.id)
```

4. **На model instance (якщо є `get_absolute_url`)**

```python
return redirect(order)
```

#### Приклад у FBV (Post/Redirect/Get):

```python
from django.shortcuts import redirect, render

def create_order(request):
    if request.method == "POST":
        # збереження даних
        return redirect("orders:list")
    return render(request, "orders/form.html")
```

#### Приклад у CBV:

```python
from django.urls import reverse_lazy
from django.views.generic import CreateView

class OrderCreateView(CreateView):
    model = Order
    fields = ["title", "amount"]
    success_url = reverse_lazy("orders:list")
```

#### Тимчасове vs постійне перенаправлення:

- `302 Found` - тимчасове (типово в Django).
- `301 Moved Permanently` - постійне (для SEO/канонічних URL).

За потреби постійного редіректу:

```python
from django.http import HttpResponsePermanentRedirect

return HttpResponsePermanentRedirect("/new-url/")
```

#### Практичні поради:

1. Використовуй іменовані маршрути замість "жорстких" URL-рядків.
2. Після успішного POST роби redirect (патерн PRG), щоб уникнути повторної
   відправки форми при refresh.
3. Для редіректу після логіну/логауту використовуй `next` і безпечні перевірки
   redirect target.

#### Типові помилки:

- Редірект на URL, який теж редіректить назад (redirect loop).
- Хардкод URL замість `name=` маршруту.
- Повернення 200 після POST-форми без PRG у flow, де це небажано.

#### Висновок:

`redirect` у Django - базовий інструмент керування user flow. Найкраща практика:
іменовані URL + PRG-патерн + контроль безпеки цільового маршруту.

</details>

<summary>21. Що таке JsonResponse і коли його використовують?</summary>

#### Django

`JsonResponse` - це HTTP-відповідь Django для повернення даних у форматі JSON.
Найчастіше використовується в API endpoint, AJAX-запитах і інтеграціях між
сервісами.

#### Що робить `JsonResponse`:

1. Серіалізує Python-дані у JSON.
2. Ставить правильний `Content-Type: application/json`.
3. Дозволяє явно вказувати HTTP status code.

#### Базовий приклад:

```python
from django.http import JsonResponse

def health(request):
    return JsonResponse({"status": "ok"})
```

#### Приклад зі статус-кодом:

```python
def create_item(request):
    # ... логіка створення
    return JsonResponse({"id": 123, "message": "created"}, status=201)
```

#### Важливий нюанс `safe`:

- За замовчуванням `JsonResponse` очікує **dict** (`safe=True`).
- Якщо треба повернути список, встанови `safe=False`.

```python
def tags(request):
    return JsonResponse(["python", "django", "api"], safe=False)
```

#### Коли використовують `JsonResponse`:

1. Простий API без DRF.
2. Внутрішні endpoint для frontend (fetch/AJAX).
3. Webhook-відповіді та службові інтеграції.
4. Health-check/diagnostic endpoint.

#### Коли краще DRF, а не "чистий" `JsonResponse`:

- Потрібні serializers, валідація схем, пагінація, throttling, auth-політики.
- Є багато API endpoint і потрібна єдина архітектура API-шару.

#### Практичні рекомендації:

1. Завжди повертай зрозумілу JSON-структуру (наприклад, `{"data": ...}`,
   `{"error": ...}`).
2. Використовуй коректні статус-коди (`200`, `201`, `400`, `404`, `500`).
3. Не повертай "сирі" exception trace у production.
4. Узгодь формат помилок для всього API.

#### Висновок:

`JsonResponse` - легкий і зручний інструмент для JSON-відповідей у Django.
Для простих API його достатньо, для великих API-систем зазвичай переходять на DRF.

</details>

<summary>22. Як повертати файли (CSV, PDF) із view?</summary>

#### Django

У Django файли з view повертають через `HttpResponse` або `FileResponse` з
правильними HTTP-заголовками (`Content-Type`, `Content-Disposition`).

#### Базові принципи:

1. **`Content-Type`** має відповідати типу файлу.
2. **`Content-Disposition`** визначає поведінку браузера:
- `attachment` -> завантажити файл
- `inline` -> показати у браузері (якщо підтримується)

#### Приклад: повернення CSV

```python
import csv
from django.http import HttpResponse

def export_users_csv(request):
    response = HttpResponse(content_type="text/csv")
    response["Content-Disposition"] = 'attachment; filename="users.csv"'

    writer = csv.writer(response)
    writer.writerow(["id", "email", "is_active"])
    writer.writerow([1, "user@example.com", True])

    return response
```

#### Приклад: повернення PDF з файлу

```python
from django.http import FileResponse

def download_invoice_pdf(request, invoice_id):
    file_path = f"/tmp/invoice-{invoice_id}.pdf"
    return FileResponse(
        open(file_path, "rb"),
        as_attachment=True,
        filename=f"invoice-{invoice_id}.pdf",
        content_type="application/pdf",
    )
```

#### Якщо PDF генерується динамічно:

- Генеруй PDF у памʼять (`io.BytesIO`) або тимчасовий файл.
- Потім віддай через `HttpResponse`/`FileResponse`.
- Поширені бібліотеки: `reportlab`, `weasyprint`, `xhtml2pdf`.

#### Для великих файлів:

1. Віддавай через `FileResponse` (стрімінг, менше RAM).
2. Не читай весь файл у памʼять (`read()` великих файлів небажаний).
3. За потреби використовуй Nginx/X-Accel-Redirect або CDN/object storage.

#### Практичні поради:

1. Додавай зрозумілі імена файлів у `filename=`.
2. Контролюй доступ до файлів (перевірка owner/permissions).
3. Для чутливих документів додавай аудит доступу.
4. В API повертай прогнозовані помилки (`404` якщо файл не знайдено, `403` якщо
   доступ заборонений).

#### Висновок:

Повернення CSV/PDF у Django - це насамперед правильний response-тип і заголовки.
Для малих файлів вистачає `HttpResponse`, для великих і production-навантаження
краще використовувати `FileResponse` та стрімінг.

</details>

<summary>23. Як реалізувати пагінацію?</summary>

#### Django

Пагінація в Django потрібна, щоб ділити великі списки даних на сторінки й не
рендерити/передавати сотні або тисячі записів за один запит.

#### Базовий інструмент:

- `django.core.paginator.Paginator`

#### Приклад у FBV:

```python
from django.core.paginator import Paginator
from django.shortcuts import render
from .models import Article

def article_list(request):
    queryset = Article.objects.order_by("-created_at")
    paginator = Paginator(queryset, 10)  # 10 елементів на сторінку
    page_number = request.GET.get("page")
    page_obj = paginator.get_page(page_number)  # безпечний варіант

    return render(request, "articles/list.html", {"page_obj": page_obj})
```

#### Рендер пагінації в шаблоні:

```django
{% for article in page_obj %}
  <h3>{{ article.title }}</h3>
{% endfor %}

<div class="pagination">
  {% if page_obj.has_previous %}
    <a href="?page=1">Перша</a>
    <a href="?page={{ page_obj.previous_page_number }}">Назад</a>
  {% endif %}

  <span>Сторінка {{ page_obj.number }} з {{ page_obj.paginator.num_pages }}</span>

  {% if page_obj.has_next %}
    <a href="?page={{ page_obj.next_page_number }}">Вперед</a>
    <a href="?page={{ page_obj.paginator.num_pages }}">Остання</a>
  {% endif %}
</div>
```

#### CBV-варіант (ще простіше):

```python
from django.views.generic import ListView
from .models import Article

class ArticleListView(ListView):
    model = Article
    paginate_by = 10
    ordering = ["-created_at"]
```

#### Практичні рекомендації:

1. Використовуй стабільне сортування (`order_by`) для передбачуваної навігації.
2. Для великих списків оптимізуй queryset:
- `select_related`, `prefetch_related`, індекси в БД.
3. Зберігай інші query-параметри (фільтри, пошук) при переході сторінок.
4. Не роби занадто великий page size без потреби.

#### Типові помилки:

- Пагінація без сортування (дані "стрибають" між сторінками).
- Використання сирого `page()` без обробки некоректного номера сторінки.
- Відсутність індексів на колонках, за якими сортують/фільтрують.

#### Висновок:

У Django пагінація реалізується швидко і чисто через `Paginator` або `ListView`.
Ключ до продуктивності - правильний queryset, стабільне сортування і контроль
параметрів сторінки.

<summary>24. Як обробляти помилки 404 і 500?</summary>

#### Django

Помилки `404` і `500` у Django обробляються через стандартний механізм
error-handling: шаблони помилок, handler-функції і глобальні налаштування.

#### Що означають ці помилки:

- **404 Not Found** - маршрут або ресурс не знайдено.
- **500 Internal Server Error** - неперехоплена помилка на сервері.

#### Базова обробка через шаблони:

У production (`DEBUG = False`) Django автоматично шукає:

- `templates/404.html`
- `templates/500.html`

Якщо ці файли існують, вони будуть показані користувачу.

#### Кастомні handlers у `urls.py`:

```python
# config/urls.py
from django.contrib import admin
from django.urls import path
from . import views

urlpatterns = [
    path("admin/", admin.site.urls),
]

handler404 = "config.views.custom_404"
handler500 = "config.views.custom_500"
```

```python
# config/views.py
from django.shortcuts import render

def custom_404(request, exception):
    return render(request, "errors/404.html", status=404)

def custom_500(request):
    return render(request, "errors/500.html", status=500)
```

#### Важлива різниця між dev і production:

1. **DEBUG=True (локально)**
- Django показує детальний debug traceback.

2. **DEBUG=False (production)**
- Користувач бачить кастомну 404/500 сторінку.
- Технічні деталі помилки не повинні витікати назовні.

#### Практичні рекомендації:

1. Завжди мати дружні кастомні сторінки 404/500.
2. Логувати помилки (Sentry/ELK/OpenTelemetry), а не показувати stack trace користувачу.
3. Додати correlation/request ID у логи для швидкого розслідування.
4. Повернути коректні статус-коди (не 200 на сторінках помилок).
5. Для API віддавати JSON-помилки з єдиним форматом.

#### Тестування:

- Перевіряй 404 для неіснуючих URL.
- Тестуй 500-хендлери на staging із `DEBUG=False`.
- Переконайся, що шаблони помилок доступні в зібраному середовищі.

#### Висновок:

Якісна обробка 404/500 у Django - це поєднання UX і операційної зрілості:
користувач отримує зрозумілу сторінку, команда - повні логи для діагностики.

<summary>25. Що таке шаблони (templates) і як передати дані з view у шаблон?</summary>

#### Django

Шаблони (`templates`) у Django - це шар представлення, який відповідає за
генерацію HTML на основі даних, підготовлених у `view`.

#### Що таке template:

- HTML-файл із Django template syntax (`{{ }}` і `{% %}`).
- Дозволяє відображати змінні, цикли, умови, включення частин сторінки.
- Відокремлює UI від бізнес-логіки Python-коду.

#### Як передати дані з view у template:

1. У `view` сформувати `context` (словник).
2. Викликати `render(request, "template.html", context)`.

#### Приклад:

```python
from django.shortcuts import render
from .models import Article

def article_list(request):
    articles = Article.objects.filter(is_published=True).order_by("-created_at")
    context = {
        "page_title": "Статті",
        "articles": articles,
        "total": articles.count(),
    }
    return render(request, "articles/list.html", context)
```

#### Використання даних у шаблоні:

```django
<h1>{{ page_title }}</h1>
<p>Всього: {{ total }}</p>

{% if articles %}
  <ul>
    {% for article in articles %}
      <li>{{ article.title }}</li>
    {% endfor %}
  </ul>
{% else %}
  <p>Немає опублікованих статей.</p>
{% endif %}
```

#### Звідки Django бере шаблони:

- З директорій, вказаних у `TEMPLATES["DIRS"]`.
- Із папок `templates/` всередині apps (коли `APP_DIRS=True`).

#### Практичні рекомендації:

1. Тримай бізнес-логіку у Python (view/service), а не в шаблоні.
2. Передавай у контекст тільки потрібні дані.
3. Використовуй іменовані URL (`{% url 'app:view' %}`), а не хардкод посилань.
4. Для повторюваних елементів використовуй `include`/partials.

#### Безпека:

- Django автоматично екранує змінні у шаблонах (захист від XSS за замовчуванням).
- Не використовуй `|safe`, якщо не впевнений у джерелі даних.

#### Висновок:

Templates у Django - це чистий, безпечний і керований спосіб рендерити HTML.
View готує дані, template відображає їх - саме так досягається підтримуваність
і розділення відповідальностей у проєкті.

<summary>26. Як працює наслідування шаблонів (extends, block)?</summary>

#### Django

Наслідування шаблонів у Django дозволяє створити базовий layout і
перевикористовувати його у дочірніх сторінках. Це усуває дублювання HTML і
робить фронт частину проєкту керованою.

#### Ключові теги:

1. **`{% extends "base.html" %}`**
- Дочірній шаблон наслідує базовий.

2. **`{% block ... %}{% endblock %}`**
- У базовому шаблоні визначаються "слоти", які дочірній шаблон може заповнити.

#### Базовий шаблон:

```django
{# templates/base.html #}
<!doctype html>
<html lang="uk">
<head>
  <meta charset="utf-8">
  <title>{% block title %}Мій сайт{% endblock %}</title>
</head>
<body>
  <header>Header</header>

  <main>
    {% block content %}{% endblock %}
  </main>

  <footer>Footer</footer>
</body>
</html>
```

#### Дочірній шаблон:

```django
{# templates/articles/list.html #}
{% extends "base.html" %}

{% block title %}Список статей{% endblock %}

{% block content %}
  <h1>Статті</h1>
  <p>Контент сторінки...</p>
{% endblock %}
```

#### `block.super`:

Якщо потрібно доповнити, а не повністю замінити блок:

```django
{% block title %}{{ block.super }} | Статті{% endblock %}
```

#### Практичні рекомендації:

1. Тримай один головний `base.html` (layout: head, header, footer, scripts).
2. Додавай окремі блоки для `title`, `content`, `extra_css`, `extra_js`.
3. Не дублюй однакові частини в кожному шаблоні - винось у base/include.
4. Будь послідовним у назвах блоків по всьому проєкту.

#### Типові помилки:

- Відсутність єдиної структури блоків між сторінками.
- Перенесення бізнес-логіки у шаблон замість view/service.
- Дублювання layout-коду без `extends`.

#### Висновок:

`extends` і `block` - це фундамент масштабованої шаблонної архітектури в Django:
єдина оболонка сайту, мінімум дублювання, швидке внесення змін у UI.

</details>

<summary>27. Для чого використовується {% include %}?</summary>

#### Django

`{% include %}` використовується для вставки одного шаблону в інший. Це зручно
для повторюваних частин інтерфейсу: хедер, футер, картки, таблиці, пагінація.

#### Навіщо потрібен `include`:

1. **Прибирає дублювання HTML**
- Один фрагмент можна перевикористовувати в багатьох сторінках.

2. **Підвищує підтримуваність**
- Зміни в partial-файлі автоматично застосовуються всюди, де він підключений.

3. **Спрощує читання шаблонів**
- Великий шаблон розбивається на менші логічні блоки.

#### Базовий приклад:

```django
{# templates/base.html #}
<body>
  {% include "partials/header.html" %}
  <main>{% block content %}{% endblock %}</main>
  {% include "partials/footer.html" %}
</body>
```

#### Передача контексту в include:

```django
{% include "partials/user_badge.html" with user=request.user %}
```

Можна обмежити контекст лише переданими змінними:

```django
{% include "partials/user_badge.html" with user=request.user only %}
```

#### Практичні кейси:

- Компонент пагінації.
- Табличний рядок/картка товару.
- Блок алертів/повідомлень.
- Загальні елементи layout.

#### Різниця між `extends` і `include`:

- `extends` - наслідування шаблонної структури (layout-рівень).
- `include` - вставка локального фрагмента (component-рівень).

#### Типові помилки:

1. Використовувати `include` замість `extends` для базового layout.
2. Передавати занадто "широкий" неявний контекст.
3. Зашивати важку логіку в partial-шаблон.

#### Висновок:

`{% include %}` - базовий інструмент модульного template-коду в Django.
Використовуй його для дрібних перевикористовуваних фрагментів, а `extends` -
для загальної структури сторінки.

</details>

<summary>28. Що таке template partials і як працюють {% partial %} та {% partialdef %}?</summary>

#### Django

Template partials - це спосіб визначати і перевикористовувати фрагменти
шаблону прямо всередині template-файлів. Для цього використовують теги
`{% partialdef %}` (оголошення) та `{% partial %}` (виклик).

#### Як це працює:

1. **`{% partialdef name %}...{% endpartialdef %}`**
- Оголошує іменований partial-фрагмент.

2. **`{% partial name %}`**
- Рендерить partial у потрібному місці шаблону.

#### Базовий приклад:

```django
{% partialdef product_card %}
  <article class="card">
    <h3>{{ product.title }}</h3>
    <p>{{ product.price }} грн</p>
  </article>
{% endpartialdef %}

<section class="grid">
  {% for product in products %}
    {% partial product_card %}
  {% endfor %}
</section>
```

#### Чим це відрізняється від `{% include %}`:

- `include` підключає окремий файл.
- `partial/partialdef` дозволяє описати й викликати фрагмент у контексті
  поточного шаблону без створення окремого файла.

#### Коли partials особливо корисні:

1. Коли потрібно швидко перевикористати невеликий фрагмент у межах одного
   шаблону.
2. Коли важливо уникнути "роздрібнення" на десятки дрібних include-файлів.
3. Для локальних UI-блоків, які не мають сенсу виносити в окремий template.

#### Практичні рекомендації:

1. Для глобально перевикористовуваних компонентів між сторінками - лишай
   `{% include %}` або окремі partial-файли.
2. Для локальних, дрібних фрагментів - `partial/partialdef`.
3. Не перенось у partial бізнес-логіку; тільки presentation-рівень.

#### Сумісність:

- У сучасних версіях Django partial-теги доступні як частина template-системи.
- Якщо проєкт на старішій версії, аналогічну задачу зазвичай вирішують через
  `{% include %}` та кастомні template tags.

#### Висновок:

`{% partialdef %}` і `{% partial %}` - це інструмент локальної композиції
шаблонів: менше дублювання, краща читабельність і контроль структури UI.

</details>

<summary>29. Що таке template tags і filters?</summary>

#### Django

`Template tags` і `filters` - це інструменти Django template engine для
керованого рендерингу даних у шаблонах.

#### Template tags:

- Теги виконують логіку на рівні шаблону.
- Використовуються у форматі `{% ... %}`.

Типові вбудовані теги:

1. `{% if %}` / `{% elif %}` / `{% else %}` - умовний рендеринг.
2. `{% for %}` - ітерація колекцій.
3. `{% url %}` - побудова посилання за іменем маршруту.
4. `{% include %}` - вставка частини шаблону.
5. `{% csrf_token %}` - CSRF-захист у формах.
6. `{% static %}` - посилання на static-файли.

Приклад:

```django
{% if user.is_authenticated %}
  <a href="{% url 'profile' %}">Профіль</a>
{% else %}
  <a href="{% url 'login' %}">Увійти</a>
{% endif %}
```

#### Filters:

- Фільтри трансформують значення змінних.
- Використовуються через `|` у виразах `{{ ... }}`.

Типові вбудовані фільтри:

1. `|lower`, `|upper`, `|title`
2. `|length`
3. `|date:"d.m.Y"`
4. `|default:"-"`, `|default_if_none:"-"`
5. `|truncatechars:50`
6. `|safe` (обережно!)

Приклад:

```django
<p>{{ article.title|title }}</p>
<p>{{ article.created_at|date:"d.m.Y H:i" }}</p>
<p>{{ article.description|truncatechars:120 }}</p>
```

#### Як створити власний filter:

```python
# app/templatetags/text_extras.py
from django import template

register = template.Library()

@register.filter
def initials(full_name):
    parts = full_name.split()
    return "".join(p[0].upper() for p in parts if p)
```

```django
{% load text_extras %}
{{ user.full_name|initials }}
```

#### Як створити власний simple tag:

```python
@register.simple_tag
def app_version():
    return "1.0.0"
```

```django
{% load text_extras %}
<small>v{% app_version %}</small>
```

#### Практичні правила:

1. Тримай шаблони "тонкими": мінімум логіки, максимум читабельності.
2. Повторювану presentation-логіку винось у кастомні filters/tags.
3. Не переноси бізнес-логіку в template tags.
4. Використовуй `|safe` лише для перевіреного контенту.

#### Висновок:

Template tags керують структурою рендерингу, filters - форматуванням значень.
Разом вони дають гнучкий і безпечний спосіб будувати UI в Django-шаблонах.

</details>

<summary>30. Як підключати статичні файли (static) і що робить collectstatic?</summary>

#### Django

Статичні файли (`static`) - це CSS, JS, зображення, шрифти та інші ресурси, які
не генеруються динамічно для кожного запиту.

#### Базові налаштування у `settings.py`:

```python
STATIC_URL = "/static/"
STATIC_ROOT = BASE_DIR / "staticfiles"   # для production-збірки
```

За потреби додаткових директорій:

```python
STATICFILES_DIRS = [BASE_DIR / "static"]
```

#### Як підключати static у шаблоні:

```django
{% load static %}
<link rel="stylesheet" href="{% static 'css/main.css' %}">
<script src="{% static 'js/app.js' %}"></script>
<img src="{% static 'img/logo.svg' %}" alt="Logo">
```

#### Де зберігати static:

1. У кожному app:
- `app_name/static/app_name/...`

2. Або у спільній папці проєкту:
- `project_root/static/...`

#### Що робить `collectstatic`:

Команда:

```bash
python manage.py collectstatic
```

Результат:

1. Збирає всі static-файли з apps і `STATICFILES_DIRS`.
2. Копіює їх у єдину директорію `STATIC_ROOT`.
3. Готує набір файлів, який віддає вебсервер/CDN у production.

#### Dev vs Production:

1. **Локальна розробка (`DEBUG=True`)**
- Django може сам віддавати static.

2. **Production (`DEBUG=False`)**
- Static зазвичай віддає Nginx, CDN або object storage.
- Django-процес не повинен обслуговувати static напряму.

#### Часто використовувані production-практики:

- `ManifestStaticFilesStorage` для versioned/hashed файлів (cache busting).
- WhiteNoise для простого деплою без окремого Nginx.
- CDN для швидкої доставки статичних ресурсів.

#### Типові помилки:

1. Не запускати `collectstatic` перед деплоєм.
2. Хардкодити шляхи до `/static/...` без `{% static %}`.
3. Плутати `static` (assets) і `media` (user uploads).
4. Не налаштувати `STATIC_ROOT` для production.

#### Висновок:

Підключення static у Django базується на `{% static %}` і правильних settings.
`collectstatic` - обовʼязковий крок продакшн-збірки, який уніфікує всі
статичні ресурси для швидкої й стабільної роздачі.

</details>

<summary>31. Що таке MEDIA_ROOT і MEDIA_URL?</summary>

#### Django

`MEDIA_ROOT` і `MEDIA_URL` використовуються для роботи з файлами, які
завантажують користувачі (uploads): аватари, документи, фото, відео тощо.

#### Що таке `MEDIA_ROOT`:

- Абсолютний шлях у файловій системі сервера, де фізично зберігаються upload-файли.
- Це "диск", куди Django пише файли з `FileField`/`ImageField`.

#### Що таке `MEDIA_URL`:

- URL-префікс, через який ці файли доступні з браузера.
- Це "публічний шлях" до контенту з `MEDIA_ROOT`.

#### Базове налаштування:

```python
# settings.py
MEDIA_URL = "/media/"
MEDIA_ROOT = BASE_DIR / "media"
```

#### Приклад моделі:

```python
from django.db import models

class Profile(models.Model):
    avatar = models.ImageField(upload_to="avatars/")
```

Якщо користувач завантажив файл `photo.jpg`, він потрапить у:

- Файл на диску: `MEDIA_ROOT/avatars/photo.jpg`
- URL у браузері: `MEDIA_URL + "avatars/photo.jpg"` -> `/media/avatars/photo.jpg`

#### Налаштування URL у development:

```python
# config/urls.py
from django.conf import settings
from django.conf.urls.static import static

urlpatterns = [
    # ...
]

if settings.DEBUG:
    urlpatterns += static(settings.MEDIA_URL, document_root=settings.MEDIA_ROOT)
```

#### Production-підхід:

1. Django зберігає файл (локально або в cloud storage backend).
2. Віддачу файлів робить Nginx/CDN/S3, а не Django-процес.
3. Доступ контролюється через permissions або pre-signed URL (для приватних файлів).

#### Важлива різниця між `static` і `media`:

- `static` - файли розробника (CSS/JS/logo), збираються через `collectstatic`.
- `media` - файли користувачів, не проходять через `collectstatic`.

#### Типові помилки:

1. Змішувати `STATIC_ROOT` і `MEDIA_ROOT`.
2. Зберігати user uploads у git-репозиторії.
3. Віддавати media через Django в production без потреби.
4. Не валідувати тип/розмір завантажених файлів.

#### Висновок:

`MEDIA_ROOT` - де файл лежить фізично, `MEDIA_URL` - як до нього звертається
клієнт. Чітке розділення media/static - обовʼязковий стандарт у Django-проєктах.

<summary>32. Які є підходи до кешування шаблонів?</summary>

#### Django

Кешування шаблонів у Django зменшує навантаження на CPU/БД і прискорює
відповідь сторінок за рахунок повторного використання вже згенерованого
контенту.

#### Основні підходи:

1. **Template fragment caching (рекомендовано для частин сторінки)**
- Кешує тільки фрагмент шаблону, а не всю сторінку.

```django
{% load cache %}
{% cache 300 sidebar user.id %}
  {# важкий блок: меню, рекомендації, віджети #}
  ...
{% endcache %}
```

2. **Per-view caching**
- Кешує повну відповідь конкретного view на заданий TTL.

```python
from django.views.decorators.cache import cache_page

@cache_page(60 * 5)  # 5 хв
def article_list(request):
    ...
```

3. **Site-wide caching (через middleware)**
- Кешує майже весь сайт за глобальними правилами.
- Підходить для контентних сайтів із низькою персоналізацією.

4. **Low-level cache API**
- Кешування результатів запитів/обчислень у Python-коді.

```python
from django.core.cache import cache

data = cache.get("popular_articles")
if data is None:
    data = expensive_query()
    cache.set("popular_articles", data, 300)
```

#### Налаштування бекенду кешу:

У production зазвичай використовують Redis або Memcached:

```python
CACHES = {
    "default": {
        "BACKEND": "django.core.cache.backends.redis.RedisCache",
        "LOCATION": "redis://127.0.0.1:6379/1",
    }
}
```

#### Коли що обирати:

- Фрагменти UI з дорогим рендером -> template fragment cache.
- Стабільна публічна сторінка -> per-view cache.
- Переважно статичний сайт -> site-wide cache.
- Дорога бізнес-логіка/запити -> low-level cache API.

#### Практичні правила:

1. Починай із найменш агресивного кешу (фрагменти), потім масштабуй.
2. Обирай TTL за бізнес-вимогами до "свіжості" даних.
3. Для персоналізованих блоків додавай ключі (user id, locale, permissions).
4. Передбачай інвалідацію кешу після змін даних.

#### Типові помилки:

1. Кешувати персоналізований контент без user-ключа.
2. Не інвалідовувати кеш після update/delete.
3. Надто довгий TTL для динамічних даних.
4. Вважати кеш "лікуванням" неефективних SQL-запитів.

#### Висновок:

Найбезпечніший старт у Django - fragment caching + Redis. Далі додають per-view
або site-wide кеш, коли це виправдано профілем навантаження й UX-вимогами.

<summary>33. Що таке Django Forms і ModelForm?</summary>

#### Django

`Django Forms` - це механізм для обробки HTML-форм: валідація, очищення даних,
рендер полів і обробка помилок.  
`ModelForm` - спеціальний тип форми, який автоматично будується на основі
Django-моделі.

#### Django Form:

- Описується через `forms.Form`.
- Поля задаються вручну.
- Підходить для форм, не повʼязаних напряму з моделлю.

Приклад:

```python
from django import forms

class ContactForm(forms.Form):
    name = forms.CharField(max_length=100)
    email = forms.EmailField()
    message = forms.CharField(widget=forms.Textarea)
```

#### ModelForm:

- Описується через `forms.ModelForm`.
- Поля генеруються з моделі.
- Легко зберігає дані в БД через `form.save()`.

Приклад:

```python
from django import forms
from .models import Article

class ArticleForm(forms.ModelForm):
    class Meta:
        model = Article
        fields = ["title", "content", "is_published"]
```

#### Основна різниця:

1. **Джерело полів**
- Form: ти описуєш поля вручну.
- ModelForm: поля беруться з моделі.

2. **Збереження даних**
- Form: ручна обробка і збереження.
- ModelForm: `form.save()` "з коробки".

3. **Сценарій використання**
- Form: contact form, пошук, фільтри, login/reset, інтеграційні форми.
- ModelForm: CRUD для моделей, адміноподібні інтерфейси.

#### Приклад обробки у view:

```python
def create_article(request):
    if request.method == "POST":
        form = ArticleForm(request.POST)
        if form.is_valid():
            form.save()
            return redirect("articles:list")
    else:
        form = ArticleForm()

    return render(request, "articles/form.html", {"form": form})
```

#### Переваги Django Forms загалом:

- Централізована валідація на сервері.
- Захист від помилок введення і некоректних типів.
- Зручна робота з повідомленнями про помилки.
- Легка інтеграція з шаблонами.

#### Висновок:

`Form` бери для кастомних сценаріїв без прямої моделі.  
`ModelForm` - найшвидший і найчистіший шлях для CRUD-операцій з моделями Django.

</details>

<summary>34. У чому різниця між HTML-формою і Django-формою?</summary>

#### Django

HTML-форма і Django-форма вирішують одну задачу (ввід даних), але на різних
рівнях. HTML-форма - це лише розмітка в браузері, а Django-форма - серверний
механізм валідації та обробки цих даних.

#### HTML-форма:

- Описує поля, кнопки, `method`, `action`.
- Працює на клієнті (браузер).
- Не гарантує безпечність і коректність даних на сервері.

Приклад:

```html
<form method="post" action="/contact/">
  <input type="text" name="name" />
  <input type="email" name="email" />
  <button type="submit">Надіслати</button>
</form>
```

#### Django-форма:

- Описується в Python (`forms.Form` або `forms.ModelForm`).
- Виконує серверну валідацію, очищення даних (`cleaned_data`), формує помилки.
- Легко інтегрується з шаблонами і CSRF-захистом.

Приклад:

```python
from django import forms

class ContactForm(forms.Form):
    name = forms.CharField(max_length=100)
    email = forms.EmailField()
```

#### Ключові відмінності:

1. **Рівень**
- HTML-форма: presentation layer.
- Django-форма: backend validation layer.

2. **Валідація**
- HTML: базова клієнтська (легко обійти).
- Django Form: обовʼязкова серверна, надійна.

3. **Безпека**
- HTML сам по собі не захищає від підроблених запитів.
- Django форма працює з `{% csrf_token %}` і вбудованою обробкою помилок.

4. **Підтримуваність**
- HTML-only підхід часто веде до дублювання валідації.
- Django-форма централізує правила і спрощує підтримку.

5. **Робота з моделями**
- HTML: збереження вручну.
- ModelForm: `form.save()` "з коробки".

#### Практичне правило:

У production не можна покладатися лише на HTML-валідацію.  
Клієнтська валідація - для UX, Django-форма - для correctness і безпеки.

#### Висновок:

HTML-форма відповідає за інтерфейс, Django-форма - за якість даних і серверну
надійність. У реальних проєктах ці підходи працюють разом, але критична
перевірка завжди має бути на бекенді.

</details>

<summary>35. Як рендерити форми в шаблоні?</summary>

#### Django

Форми в Django рендерять у шаблоні через обʼєкт `form`, який передається з view.
Є кілька підходів: швидкий автоматичний рендер або контрольований ручний рендер.

#### 1. Швидкий автоматичний рендер:

```django
<form method="post">
  {% csrf_token %}
  {{ form.as_p }}
  <button type="submit">Зберегти</button>
</form>
```

Доступні варіанти:

- `{{ form.as_p }}` - поля обгорнуті в `<p>`
- `{{ form.as_ul }}` - поля як список `<li>`
- `{{ form.as_table }}` - поля у табличному форматі

#### 2. Ручний рендер (рекомендовано для production UI):

```django
<form method="post" novalidate>
  {% csrf_token %}

  <div>
    {{ form.title.label_tag }}
    {{ form.title }}
    {% for error in form.title.errors %}
      <p class="error">{{ error }}</p>
    {% endfor %}
  </div>

  <div>
    {{ form.content.label_tag }}
    {{ form.content }}
    {% for error in form.content.errors %}
      <p class="error">{{ error }}</p>
    {% endfor %}
  </div>

  {% if form.non_field_errors %}
    <div class="error">
      {% for error in form.non_field_errors %}
        <p>{{ error }}</p>
      {% endfor %}
    </div>
  {% endif %}

  <button type="submit">Зберегти</button>
</form>
```

#### Що обовʼязково додати:

1. `{% csrf_token %}` для POST-форм.
2. Відображення помилок полів (`field.errors`) і загальних помилок (`non_field_errors`).
3. `enctype="multipart/form-data"` для форм із файлами.

#### Приклад для upload-форми:

```django
<form method="post" enctype="multipart/form-data">
  {% csrf_token %}
  {{ form.as_p }}
  <button type="submit">Завантажити</button>
</form>
```

#### Практичні рекомендації:

1. Для MVP можна почати з `form.as_p`.
2. Для реального продукту краще ручний рендер для повного контролю над UX/UI.
3. Не приховуй помилки валідації - показуй їх біля відповідних полів.
4. Перевагу віддавай іменованим кнопкам дій і чітким submit-flow.

#### Висновок:

Рендер форм у Django може бути дуже швидким або дуже контрольованим. Для
production-якості зазвичай обирають ручний рендер з явним відображенням помилок,
CSRF-захистом і керованою версткою.

<summary>36. Як працює валідація форм і як створити власну?</summary>

#### Django

Валідація форм у Django відбувається на сервері під час виклику `form.is_valid()`.
У результаті Django або повертає очищені дані (`cleaned_data`), або наповнює
`form.errors`.

#### Як працює валідація по кроках:

1. Django перевіряє типи і базові правила полів (`required`, `max_length`,
   `EmailField`, `IntegerField` тощо).
2. Викликає кастомні валідатори поля (якщо додані).
3. Викликає методи `clean_<field>()` для конкретних полів.
4. Викликає загальний `clean()` для міжпольової логіки.
5. Формує `cleaned_data` або список помилок.

#### Валідація одного поля (`clean_<field>()`):

```python
from django import forms
from django.core.exceptions import ValidationError

class RegisterForm(forms.Form):
    username = forms.CharField(max_length=50)

    def clean_username(self):
        value = self.cleaned_data["username"].strip()
        if "admin" in value.lower():
            raise ValidationError("Імʼя користувача містить заборонене слово.")
        return value
```

#### Міжпольова валідація (`clean()`):

```python
class PasswordForm(forms.Form):
    password = forms.CharField(widget=forms.PasswordInput)
    confirm_password = forms.CharField(widget=forms.PasswordInput)

    def clean(self):
        cleaned_data = super().clean()
        p1 = cleaned_data.get("password")
        p2 = cleaned_data.get("confirm_password")

        if p1 and p2 and p1 != p2:
            raise forms.ValidationError("Паролі не співпадають.")
        return cleaned_data
```

#### Кастомні валідатори:

```python
from django.core.exceptions import ValidationError

def validate_company_email(value):
    if not value.endswith("@company.com"):
        raise ValidationError("Дозволено лише корпоративний email.")
```

```python
email = forms.EmailField(validators=[validate_company_email])
```

#### Для `ModelForm`:

- Працює валідація форми + валідація моделі.
- У моделі можна додатково використовувати:
  `clean()`, `unique=True`, `UniqueConstraint`, `validators`.

#### Як обробляти у view:

```python
def submit_form(request):
    form = RegisterForm(request.POST or None)
    if request.method == "POST" and form.is_valid():
        # працюємо з form.cleaned_data
        ...
    return render(request, "form.html", {"form": form})
```

#### Практичні правила:

1. Валідація завжди має бути на сервері (клієнтська - лише для UX).
2. Локальна логіка поля -> `clean_<field>()`.
3. Логіка, що залежить від кількох полів -> `clean()`.
4. Повторно використовувані правила -> окремі валідатори.
5. Бізнес-правила рівня домену дублюй на рівні БД (constraints), а не лише у формі.

#### Висновок:

Сильна сторона Django Forms - структурована валідація на кількох рівнях.
Це дає контроль якості даних, зрозумілі помилки для користувача і стабільний
бекенд-потік обробки форм у production.

<summary>37. Що таке кастомні валідатори в Django і як їх застосовувати повторно?</summary>

#### Django

Кастомні валідатори в Django - це функції або класи, які перевіряють значення
поля за твоїми правилами і кидають `ValidationError`, якщо дані не проходять
перевірку.

#### Навіщо вони потрібні:

1. **Повторне використання правил**
- Одна перевірка застосовується в багатьох формах/моделях.

2. **Єдине джерело правди**
- Бізнес-правило не дублюється по всьому коду.

3. **Чистіша архітектура**
- Логіка валідації винесена з view/forms у окремий шар.

#### Приклад функціонального валідатора:

```python
from django.core.exceptions import ValidationError

def validate_corporate_email(value):
    if not value.endswith("@company.com"):
        raise ValidationError("Дозволено лише корпоративний email.")
```

Використання в моделі:

```python
from django.db import models
from .validators import validate_corporate_email

class Employee(models.Model):
    email = models.EmailField(validators=[validate_corporate_email])
```

Використання у формі:

```python
from django import forms
from .validators import validate_corporate_email

class InviteForm(forms.Form):
    email = forms.EmailField(validators=[validate_corporate_email])
```

#### Приклад class-based валідатора:

```python
from django.core.exceptions import ValidationError

class MinWordsValidator:
    def __init__(self, min_words=3):
        self.min_words = min_words

    def __call__(self, value):
        if len(value.split()) < self.min_words:
            raise ValidationError(f"Мінімум {self.min_words} слова(ів).")
```

```python
description = models.TextField(validators=[MinWordsValidator(5)])
```

#### Де краще розміщувати:

- `app/validators.py` або `app/domain/validators.py`
- Для великих проєктів - окремі модулі за доменами.

#### Практичні правила:

1. Валідатор має робити одну конкретну перевірку.
2. Повідомлення про помилки мають бути зрозумілими для користувача.
3. Для критичних правил дублюй контроль на рівні БД (`UniqueConstraint`, check constraints).
4. Пиши тести для валідаторів окремо від view.

#### Висновок:

Кастомні валідатори - це правильний спосіб централізувати бізнес-правила
валідації в Django. Вони роблять код чистішим, більш повторно використовуваним
і знижують ризик розсинхрону правил у різних частинах системи.

<summary>38. Як обробляти завантаження файлів?</summary>

#### Django

Завантаження файлів у Django обробляється через `request.FILES`, поля
`FileField`/`ImageField` і правильно налаштовані `MEDIA_ROOT`/`MEDIA_URL`.

#### Що потрібно для upload:

1. Поле файлу у формі або моделі.
2. `enctype="multipart/form-data"` у HTML-формі.
3. Передача `request.FILES` у форму.
4. Налаштовані media-параметри в `settings.py`.

#### Приклад моделі:

```python
from django.db import models

class Document(models.Model):
    title = models.CharField(max_length=200)
    file = models.FileField(upload_to="documents/%Y/%m/")
```

#### Приклад ModelForm:

```python
from django import forms
from .models import Document

class DocumentForm(forms.ModelForm):
    class Meta:
        model = Document
        fields = ["title", "file"]
```

#### Приклад view:

```python
from django.shortcuts import redirect, render
from .forms import DocumentForm

def upload_document(request):
    if request.method == "POST":
        form = DocumentForm(request.POST, request.FILES)
        if form.is_valid():
            form.save()
            return redirect("documents:list")
    else:
        form = DocumentForm()

    return render(request, "documents/upload.html", {"form": form})
```

#### Шаблон:

```django
<form method="post" enctype="multipart/form-data">
  {% csrf_token %}
  {{ form.as_p }}
  <button type="submit">Завантажити</button>
</form>
```

#### Налаштування media:

```python
# settings.py
MEDIA_URL = "/media/"
MEDIA_ROOT = BASE_DIR / "media"
```

#### Безпека і production-практика:

1. Валідуй тип, розмір і розширення файлу.
2. Не довіряй оригінальному filename від клієнта.
3. Обмежуй доступ до приватних файлів через permissions.
4. Для production використовуй зовнішнє сховище (S3-compatible) або окремий
   файловий сервер.
5. Скануй потенційно небезпечні файли (якщо бізнес-кейс ризиковий).

#### Типові помилки:

1. Забути `enctype="multipart/form-data"` -> `request.FILES` порожній.
2. Передати лише `request.POST` без `request.FILES`.
3. Змішати static і media.
4. Не перевіряти розмір/тип і отримати ризики безпеки або перевантаження.

#### Висновок:

У Django upload працює надійно, якщо дотриматись базових правил: правильна форма,
`request.FILES`, валідатори та контроль доступу до файлів у production.

<summary>39. Що таке Django ORM?</summary>

#### Django

Django ORM (Object-Relational Mapping) - це шар доступу до даних, який дозволяє
працювати з базою через Python-обʼєкти замість ручного написання SQL у більшості
типових сценаріїв.

#### Основна ідея ORM:

1. **Модель Python = таблиця БД**
2. **Екземпляр моделі = рядок таблиці**
3. **Поле моделі = колонка таблиці**
4. **QuerySet = SQL-запит(и), сформовані ORM**

#### Приклад:

```python
# models.py
class Article(models.Model):
    title = models.CharField(max_length=200)
    is_published = models.BooleanField(default=False)
    created_at = models.DateTimeField(auto_now_add=True)
```

```python
# створення
Article.objects.create(title="Django ORM", is_published=True)

# читання
articles = Article.objects.filter(is_published=True).order_by("-created_at")

# оновлення
article = Article.objects.get(pk=1)
article.title = "Updated title"
article.save()

# видалення
article.delete()
```

#### Переваги Django ORM:

1. **Швидка розробка** - менше boilerplate SQL.
2. **Читабельний код** - запити описуються декларативно.
3. **Безпека** - параметризація запитів знижує ризик SQL injection.
4. **Переносимість** - той самий код працює з різними СУБД (переважно).
5. **Інтеграція з Django** - forms, admin, migrations, validators.

#### Важливі концепти:

- `QuerySet` лінивий: запит виконується лише коли потрібні дані.
- `select_related` і `prefetch_related` допомагають уникати N+1 проблем.
- `annotate`, `aggregate`, `F`, `Q`, `Subquery` - для складніших запитів.

#### Обмеження ORM:

1. Дуже складні SQL-конструкції іноді простіше писати raw SQL.
2. Без профілювання легко отримати неефективні запити.
3. Неправильне використання звʼязків може сильно вдарити по продуктивності.

#### Коли використовувати raw SQL:

- Специфічні оптимізації під конкретну СУБД.
- Складна аналітика/CTE/window-функції, де ORM-код стає нечитабельним.
- Міграційні/операційні задачі, де потрібен точний контроль SQL.

#### Висновок:

Django ORM - один із ключових плюсів фреймворку: швидко, безпечно і зручно для
більшості бізнес-кейсів. Але для production-результату важливо знати, що саме
генерує ORM, і вчасно оптимізувати запити.

</details>

<summary>40. Що таке модель і як її створити?</summary>

#### Django

Модель (`Model`) у Django - це Python-клас, який описує структуру даних і
відповідає таблиці в базі даних. Через модель ти визначаєш поля, звʼязки та
правила на рівні даних.

#### Що таке модель:

1. **Опис схеми БД у Python**
- Поля моделі (`CharField`, `IntegerField`, `DateTimeField`...) відповідають
  колонкам таблиці.

2. **Шар доменних сутностей**
- Модель представляє бізнес-обʼєкт: `User`, `Order`, `Invoice`, `Product`.

3. **Інтеграція з ORM**
- Створення, читання, оновлення, видалення записів через `Model.objects`.

#### Приклад моделі:

```python
from django.db import models

class Product(models.Model):
    name = models.CharField(max_length=150)
    sku = models.CharField(max_length=64, unique=True)
    price = models.DecimalField(max_digits=10, decimal_places=2)
    is_active = models.BooleanField(default=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    def __str__(self):
        return f"{self.name} ({self.sku})"
```

#### Як створити модель крок за кроком:

1. Додай клас у `app/models.py`.
2. Переконайся, що app підключений у `INSTALLED_APPS`.
3. Створи міграцію:

```bash
python manage.py makemigrations
```

4. Застосуй міграцію до БД:

```bash
python manage.py migrate
```

#### Базові практики:

1. Використовуй зрозумілі назви полів і обмеження (`unique`, `db_index`).
2. Додавай службові поля `created_at`, `updated_at`.
3. Задавай `__str__` для читабельності в admin і shell.
4. Доменно важливі інваріанти фіксуй у БД (constraints), не лише у view/forms.

#### Типові помилки:

1. Зберігати "все як text", ігноруючи типи даних.
2. Відсутність індексів для полів фільтрації/пошуку.
3. Змішування бізнес-логіки високого рівня прямо в моделі без структури.
4. Несвоєчасні міграції, що розходяться між середовищами.

#### Висновок:

Модель у Django - це фундамент роботи з даними. Якісно спроєктовані моделі
визначають продуктивність, надійність і масштабованість усього проєкту.

<summary>41. Що таке ForeignKey, OneToOneField, ManyToManyField?</summary>

#### Django

`ForeignKey`, `OneToOneField` і `ManyToManyField` - це типи звʼязків між
моделями в Django ORM. Вони визначають, як сутності повʼязані між собою в БД.

#### 1. ForeignKey (один-до-багатьох, 1:N)

- Один обʼєкт "батька" може мати багато "дочірніх".
- Кожен дочірній запис посилається лише на одного батька.

Приклад:

```python
class Author(models.Model):
    name = models.CharField(max_length=120)

class Book(models.Model):
    title = models.CharField(max_length=200)
    author = models.ForeignKey(Author, on_delete=models.CASCADE, related_name="books")
```

Кейс: один автор -> багато книг.

#### 2. OneToOneField (один-до-одного, 1:1)

- Кожному запису моделі A відповідає максимум один запис моделі B.
- По суті це `ForeignKey(unique=True)` зі зрозумілішою семантикою.

Приклад:

```python
from django.conf import settings

class Profile(models.Model):
    user = models.OneToOneField(settings.AUTH_USER_MODEL, on_delete=models.CASCADE)
    bio = models.TextField(blank=True)
```

Кейс: один користувач -> один профіль.

#### 3. ManyToManyField (багато-до-багатьох, M:N)

- Записи з обох таблиць можуть мати багато звʼязків між собою.
- Django створює проміжну таблицю автоматично (або через `through=`).

Приклад:

```python
class Tag(models.Model):
    name = models.CharField(max_length=50, unique=True)

class Article(models.Model):
    title = models.CharField(max_length=200)
    tags = models.ManyToManyField(Tag, related_name="articles")
```

Кейс: стаття має багато тегів, тег належить багатьом статтям.

#### Як обирати тип звʼязку:

1. `ForeignKey` - коли є чіткий "власник" і множинні дочірні записи.
2. `OneToOneField` - коли сутність-розширення має існувати в одному екземплярі.
3. `ManyToManyField` - коли обидві сторони можуть мати багато відповідників.

#### Нюанс `on_delete` для ForeignKey/OneToOne:

- `CASCADE` - видалити дочірні записи разом із батьком.
- `PROTECT` - заборонити видалення, якщо є залежності.
- `SET_NULL` - обнулити посилання (поле має бути `null=True`).

#### Висновок:

Ці три поля - основа моделювання реляційних даних у Django. Правильно обраний
тип звʼязку спрощує запити, міграції й довгострокову підтримку доменної моделі.

</details>

<summary>42. Які типи наслідування моделей існують?</summary>

#### Django

У Django є три основні типи наслідування моделей: **Abstract Base Classes**,
**Multi-table inheritance** і **Proxy models**. Вони вирішують різні задачі і
мають різний вплив на схему БД.

#### 1. Abstract Base Class (абстрактне наслідування)

- Батьківська модель не створює власну таблицю в БД.
- Її поля копіюються у дочірні моделі.

Приклад:

```python
class TimeStampedModel(models.Model):
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        abstract = True

class Article(TimeStampedModel):
    title = models.CharField(max_length=200)
```

Коли використовувати:

- Для спільних полів/методів між багатьма моделями.
- Найчастіший і найпрактичніший варіант у production.

#### 2. Multi-table inheritance (наслідування з окремими таблицями)

- Кожна модель у ланцюгу наслідування має окрему таблицю.
- Django створює `OneToOne` звʼязок між батьком і нащадком.

Приклад:

```python
class Person(models.Model):
    name = models.CharField(max_length=120)

class Employee(Person):
    salary = models.DecimalField(max_digits=10, decimal_places=2)
```

Коли використовувати:

- Коли потрібна реальна ієрархія сутностей у БД.
- Коли важливо мати окремі таблиці для базової і спеціалізованої моделі.

Мінус:

- Додаткові JOIN-и і складніші запити.

#### 3. Proxy model (проксі-модель)

- Нова таблиця не створюється.
- Працює з тією ж таблицею, що й базова модель.
- Дозволяє змінити Python-поведінку (методи, менеджер, ordering), не змінюючи схему БД.

Приклад:

```python
class Order(models.Model):
    status = models.CharField(max_length=20)

class OpenOrder(Order):
    class Meta:
        proxy = True
        ordering = ["id"]
```

Коли використовувати:

- Потрібен інший менеджер/поведінка для тієї ж моделі.
- Потрібне інше відображення в Django Admin без зміни структури таблиць.

#### Швидкий вибір:

1. Спільні поля без окремої таблиці -> **abstract**.
2. Реальна ієрархія даних із окремими таблицями -> **multi-table**.
3. Інша Python-поведінка без зміни БД -> **proxy**.

#### Практична порада:

За замовчуванням обирай **abstract base class**.  
`multi-table` використовуй обережно, тільки коли це продиктовано доменною
моделлю, а не "для краси" ієрархії.

#### Висновок:

Тип наслідування в Django обирають за метою: повторне використання полів,
фізична ієрархія таблиць або лише зміна поведінки моделі. Правильний вибір
спрощує схему БД і майбутню підтримку проєкту.

</details>

<summary>43. Для чого використовується клас Meta?</summary>

#### Django

Внутрішній клас `Meta` у моделі Django використовується для конфігурації
поведінки моделі на рівні ORM і БД без зміни бізнес-логіки самої моделі.

#### Що зазвичай задають у `Meta`:

1. **`ordering`** - сортування за замовчуванням.
2. **`db_table`** - кастомна назва таблиці в БД.
3. **`verbose_name`, `verbose_name_plural`** - читабельні назви в admin.
4. **`indexes`** - індекси для продуктивності.
5. **`constraints`** - бізнес-обмеження на рівні БД.
6. **`unique_together` / `UniqueConstraint`** - унікальність комбінацій полів.
7. **`abstract` / `proxy`** - тип наслідування моделі.

#### Приклад:

```python
from django.db import models
from django.db.models import Q

class Product(models.Model):
    sku = models.CharField(max_length=64)
    name = models.CharField(max_length=200)
    price = models.DecimalField(max_digits=10, decimal_places=2)
    is_active = models.BooleanField(default=True)

    class Meta:
        db_table = "catalog_product"
        ordering = ["name"]
        verbose_name = "Товар"
        verbose_name_plural = "Товари"
        indexes = [
            models.Index(fields=["sku"]),
            models.Index(fields=["is_active", "name"]),
        ]
        constraints = [
            models.UniqueConstraint(fields=["sku"], name="uq_product_sku"),
            models.CheckConstraint(
                check=Q(price__gte=0), name="chk_product_price_non_negative"
            ),
        ]
```

#### Чому `Meta` важливий:

1. **Продуктивність**
- Індекси та правильні обмеження зменшують вартість запитів.

2. **Цілісність даних**
- Constraints захищають від некоректних записів, навіть якщо десь пропущена
  валідація у view/form.

3. **Керованість схеми**
- Явний контроль таблиць і правил у міграціях.

4. **Зручність адмінки**
- Людинозрозумілі назви моделей у back-office.

#### Практичні рекомендації:

1. Використовуй `UniqueConstraint` замість застарілого `unique_together`.
2. Додавай індекси лише на реально "гарячі" поля фільтрації/сортування.
3. Тримай бізнес-критичні правила на рівні БД через `constraints`.
4. Не зловживай `db_table`, якщо немає потреби в явному неймінгу.

#### Висновок:

`Meta` - це центр конфігурації ORM-поведінки моделі. Через нього ти керуєш
порядком, індексами, обмеженнями і якістю схеми БД, що напряму впливає на
надійність і продуктивність Django-проєкту.

</details>

<summary>44. У чому різниця між null=True і blank=True?</summary>

#### Django

`null=True` і `blank=True` вирішують різні задачі і працюють на різних рівнях:

- `null=True` - рівень **бази даних** (чи можна зберігати `NULL`).
- `blank=True` - рівень **валідації форм/адмінки** (чи поле може бути порожнім).

#### `null=True`:

1. Дозволяє записувати `NULL` у колонку БД.
2. Впливає на схему таблиці та міграції.
3. Критично для полів, де "відсутнє значення" має зберігатися як SQL `NULL`.

#### `blank=True`:

1. Дозволяє не заповнювати поле у формах, ModelForm, Django Admin.
2. Не змінює тип колонки в БД напряму.
3. Впливає на `form.is_valid()`, а не на SQL-рівень.

#### Приклади:

```python
class Profile(models.Model):
    bio = models.TextField(blank=True)  # у формі необов'язкове
    birth_date = models.DateField(null=True, blank=True)  # можна і в БД NULL, і в формі пусто
```

#### Важливий нюанс для рядків (`CharField`, `TextField`):

- У Django зазвичай **не ставлять `null=True`** для рядкових полів.
- Рекомендована практика: використовувати порожній рядок `""`, а не `NULL`.
- Тобто зазвичай:
  `CharField(..., blank=True)` без `null=True`.

#### Швидка матриця:

1. `null=False, blank=False` -> поле обовʼязкове і в БД, і у формах.
2. `null=True, blank=False` -> у БД може бути `NULL`, але форма вимагає значення.
3. `null=False, blank=True` -> форма допускає пусто, але в БД зберігається не `NULL` (часто `""`).
4. `null=True, blank=True` -> максимально "опційне" поле (і форма, і БД).

#### Практичні рекомендації:

1. Для `DateField`, `DateTimeField`, `ForeignKey`, числових полів часто потрібні
   обидва: `null=True, blank=True`.
2. Для `CharField/TextField` зазвичай достатньо `blank=True`.
3. Тримай консистентну політику в проєкті, щоб уникнути хаосу між `NULL` і `""`.
4. Памʼятай: неправильний вибір ускладнює фільтрацію й аналітику.

#### Висновок:

`null` керує зберіганням у БД, `blank` - валідацією на рівні Django-форм.
Це різні осі поведінки, і їх потрібно виставляти усвідомлено під доменну модель.

</details>

<summary>45. У чому різниця між CharField і TextField?</summary>

#### Django

`CharField` і `TextField` обидва зберігають текст, але мають різні сценарії
використання та обмеження на рівні моделі/БД.

#### `CharField`:

1. Призначений для короткого/середнього тексту.
2. **Обовʼязково** вимагає `max_length`.
3. Добре підходить для полів, де довжина має бути контрольованою:
   імʼя, slug, title, код, email, phone.

Приклад:

```python
title = models.CharField(max_length=200)
sku = models.CharField(max_length=64, unique=True)
```

#### `TextField`:

1. Призначений для довгого вільного тексту.
2. Не вимагає `max_length` на рівні моделі (можна додати валідацією).
3. Підходить для описів, контенту статей, нотаток, коментарів.

Приклад:

```python
description = models.TextField(blank=True)
content = models.TextField()
```

#### Ключові відмінності:

1. **Обмеження довжини**
- `CharField`: завжди `max_length`.
- `TextField`: без жорсткої межі за замовчуванням.

2. **Семантика даних**
- `CharField`: структурований короткий текст.
- `TextField`: великий неструктурований текст.

3. **Форми/адмінка**
- `CharField` зазвичай рендериться як `input`.
- `TextField` зазвичай як `textarea`.

4. **Індексація і продуктивність**
- Короткі індексовані поля зазвичай логічніше тримати в `CharField`.
- Для великих текстів індексація залежить від СУБД і часто потребує окремих
  підходів (full-text search).

#### Практичні правила техліда:

1. Якщо значення має природне обмеження довжини -> обирай `CharField`.
2. Якщо текст потенційно великий/довільний -> `TextField`.
3. Не використовуй `TextField` "про всяк випадок", коли достатньо `CharField`.
4. Для пошуку по великих текстах плануй окрему search-стратегію (PostgreSQL FTS,
   Elasticsearch/OpenSearch тощо).

#### Висновок:

`CharField` - керований короткий текст, `TextField` - великий контент.
Правильний вибір поля впливає на валідацію, UX форм, індексацію та
продуктивність запитів.

</details>

<summary>46. Що таке model managers і як створити власний?</summary>

#### Django

`Model Manager` у Django - це "вхідна точка" для запитів до моделі через ORM.
За замовчуванням кожна модель має менеджер `objects`, але ти можеш створювати
власні менеджери з доменними методами.

#### Навіщо потрібні власні managers:

1. **Централізація query-логіки**
- Щоб не дублювати однакові `filter(...)` у різних view/сервісах.

2. **Читабельний API для моделі**
- Наприклад: `Article.objects.published().recent()`.

3. **Контроль "дефолтного" набору даних**
- Наприклад, приховати soft-deleted записи.

#### Базовий приклад manager:

```python
from django.db import models
from django.utils import timezone

class ArticleManager(models.Manager):
    def published(self):
        return self.get_queryset().filter(is_published=True)

    def recent(self):
        return self.get_queryset().filter(
            created_at__gte=timezone.now() - timezone.timedelta(days=7)
        )

class Article(models.Model):
    title = models.CharField(max_length=200)
    is_published = models.BooleanField(default=False)
    created_at = models.DateTimeField(auto_now_add=True)

    objects = ArticleManager()
```

Використання:

```python
Article.objects.published()
Article.objects.recent()
```

#### Комбінація Manager + QuerySet (краща масштабованість):

```python
class ArticleQuerySet(models.QuerySet):
    def published(self):
        return self.filter(is_published=True)

    def recent(self):
        return self.order_by("-created_at")

class Article(models.Model):
    ...
    objects = ArticleQuerySet.as_manager()
```

#### Практичні рекомендації:

1. Винось у manager/queryset лише логіку вибірки даних.
2. Бізнес-оркестрацію залишай у сервісному шарі.
3. Назви методів роби предметними (`active`, `paid`, `for_user`).
4. Не роби manager "магічним" - поведінка має бути очевидною.

#### Типові помилки:

1. Дублювання однакових фільтрів по всьому коду.
2. Змішування важкої бізнес-логіки в manager.
3. Перевизначення default manager так, що admin/міграції поводяться неочікувано.

#### Висновок:

Model managers - це правильне місце для повторно використовуваних ORM-запитів.
Власний manager робить код чистішим, а запити - консистентними по всьому проєкту.

<summary>47. Що таке міграції і навіщо вони потрібні?</summary>

#### Django

Міграції в Django - це контрольована історія змін схеми бази даних, яка
зберігається у коді проєкту і застосовується командами `makemigrations` та
`migrate`.

#### Що дають міграції:

1. **Версіонування БД**
- Зміни таблиць, полів, індексів, constraints фіксуються як файли-інструкції.

2. **Відтворюваність середовищ**
- Dev, staging і production отримують однакову схему БД.

3. **Безпечна еволюція схеми**
- Замість ручного SQL у "нотатках" команда має формалізований процес змін.

4. **Співпраця в команді**
- Міграції комітяться в git і проходять code review разом із кодом.

#### Як це працює:

1. Ти змінюєш модель у `models.py`.
2. `python manage.py makemigrations` генерує migration-файл.
3. `python manage.py migrate` застосовує зміни до БД.
4. Django записує стан у службову таблицю `django_migrations`.

#### Приклад типових змін через міграції:

- Додати/видалити поле.
- Змінити тип поля.
- Створити індекс.
- Додати `UniqueConstraint` або `CheckConstraint`.
- Перейменувати модель/поле.

#### Data migrations:

Окрім структури, можна робити міграції даних (`RunPython`), наприклад:
- заповнити нове поле значеннями,
- перенести дані між колонками,
- нормалізувати існуючі записи.

#### Практичні правила:

1. Коміть міграції разом із змінами моделей.
2. Не редагуй старі міграції, якщо вони вже застосовані в спільних середовищах.
3. Перевіряй міграції на staging перед production.
4. Для великих таблиць плануй backward-compatible rollout (expand/contract).

#### Типові помилки:

1. Запускати код без актуальних міграцій.
2. Забути закомітити migration-файли.
3. Робити небезпечні blocking-операції на великих таблицях без плану.
4. Редагувати застосовані міграції і ламати історію схеми.

#### Висновок:

Міграції - це "система контролю версій" для БД у Django. Вони критичні для
надійного деплою, командної роботи та передбачуваної еволюції даних.

<summary>48. У чому різниця між makemigrations і migrate?</summary>

#### Django

`makemigrations` і `migrate` - це дві різні фази роботи з міграціями:

- `makemigrations` **створює** migration-файли (план змін).
- `migrate` **застосовує** ці зміни до бази даних.

#### `makemigrations`:

1. Аналізує зміни в `models.py`.
2. Генерує Python-файли міграцій у `app/migrations/`.
3. Не змінює БД напряму.

Команда:

```bash
python manage.py makemigrations
```

#### `migrate`:

1. Читає migration-файли.
2. Виконує SQL-операції в БД (CREATE/ALTER/DROP тощо).
3. Відмічає застосовані міграції в таблиці `django_migrations`.

Команда:

```bash
python manage.py migrate
```

#### Простий workflow:

1. Змінив моделі.
2. Запустив `makemigrations`.
3. Перевірив згенеровану міграцію.
4. Запустив `migrate`.
5. Закомітив код + migration-файли.

#### Аналогія:

- `makemigrations` = "записати інструкцію".
- `migrate` = "виконати інструкцію".

#### Корисні команди:

```bash
python manage.py showmigrations
python manage.py sqlmigrate app_name 0001
```

- `showmigrations` показує, які міграції застосовані.
- `sqlmigrate` показує SQL, який буде виконано.

#### Типові помилки:

1. Запустити `migrate`, не створивши міграцій після зміни моделей.
2. Зробити `makemigrations`, але не закомітити файли.
3. Плутати локальний стан БД і стан на staging/production.
4. Ігнорувати конфлікти міграцій у командній розробці.

#### Висновок:

`makemigrations` готує зміни, `migrate` застосовує їх. Для стабільного процесу
деплою потрібні обидва кроки, виконані послідовно і контрольовано.

</details>

<summary>49. Як підключити PostgreSQL або іншу БД?</summary>

#### Django

Підключення БД у Django робиться через налаштування `DATABASES` у `settings.py`.
Для production найчастіше використовують PostgreSQL, рідше MySQL/MariaDB.

#### 1. Встанови драйвер БД

Для PostgreSQL (рекомендований сучасний варіант):

```bash
pip install psycopg[binary]
```

Альтернатива: `psycopg2-binary`.

#### 2. Налаштуй `DATABASES` у `settings.py`

```python
DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.postgresql",
        "NAME": "mydb",
        "USER": "myuser",
        "PASSWORD": "mypassword",
        "HOST": "127.0.0.1",
        "PORT": "5432",
    }
}
```

#### 3. Запусти міграції

```bash
python manage.py migrate
```

#### Підключення через env-змінні (рекомендовано):

```python
import os

DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.postgresql",
        "NAME": os.getenv("DB_NAME", "mydb"),
        "USER": os.getenv("DB_USER", "myuser"),
        "PASSWORD": os.getenv("DB_PASSWORD", ""),
        "HOST": os.getenv("DB_HOST", "127.0.0.1"),
        "PORT": os.getenv("DB_PORT", "5432"),
    }
}
```

#### Інші підтримувані движки:

1. SQLite (за замовчуванням для локалки/прототипів)
2. PostgreSQL (основний production-вибір)
3. MySQL / MariaDB
4. Oracle

#### Production-рекомендації:

1. Не зберігай паролі БД у репозиторії.
2. Увімкни SSL до БД (де актуально).
3. Налаштуй пулінг зʼєднань (PgBouncer або DB-side параметри).
4. Додай health-check і моніторинг latency/locks/slow queries.
5. Користуйся окремими ролями доступу (least privilege).

#### Типові помилки:

1. Неправильний `ENGINE` або відсутній драйвер.
2. Конфігурація локально працює, а в CI/prod - ні через env-змінні.
3. Використання SQLite у production для конкурентного навантаження.
4. Відсутність таймаутів/пулінгу під навантаженням.

#### Висновок:

Щоб підключити PostgreSQL у Django, потрібні три кроки: драйвер, `DATABASES`,
міграції. Для production критично: env-конфіг, безпечні креденшали, SSL і
контроль продуктивності зʼєднань.

<summary>50. Що таке QuerySet? Лінива (lazy) оцінка?</summary>

#### Django

`QuerySet` у Django ORM - це обʼєкт, який представляє набір записів із БД і
ланцюжок операцій над ними (`filter`, `exclude`, `order_by`, `annotate` тощо).

#### Що важливо розуміти про QuerySet:

1. Він **лінивий** (`lazy`) за замовчуванням.
2. Побудова QuerySet не виконує SQL одразу.
3. SQL виконується лише в момент, коли дані реально потрібні.

#### Приклад лінивості:

```python
qs = Article.objects.filter(is_published=True).order_by("-created_at")
# На цьому етапі запит до БД ще НЕ виконано
```

#### Коли QuerySet "materialize" (виконується SQL):

1. Ітерація:

```python
for article in qs:
    ...
```

2. Перетворення в список:

```python
items = list(qs)
```

3. Отримання довжини/булевого значення:

```python
len(qs)
bool(qs)
```

4. Виклики на кшталт:
- `first()`, `last()`, `exists()`, `count()`, `get()`
- слайсинг (часто з `LIMIT/OFFSET`), наприклад `qs[:10]`

#### Переваги lazy evaluation:

1. **Оптимізація запитів**
- Можна будувати складний queryset кроками перед виконанням.

2. **Гнучкість**
- Один базовий queryset можна перевикористовувати з різними умовами.

3. **Менше зайвих звернень до БД**
- Якщо queryset не використали, запит не виконається.

#### Типові помилки:

1. Викликати queryset у циклах і отримувати N+1 запити.
2. Кілька разів ітерувати той самий queryset без кешування результату.
3. Плутати `count()` і `len(qs)` (поведінка/вартість можуть відрізнятися).

#### Практичні поради:

1. Для звʼязків використовуй `select_related()` / `prefetch_related()`.
2. Якщо результат потрібен багато разів у межах одного flow - матеріалізуй:
   `items = list(qs)`.
3. Для перевірки наявності використовуй `exists()`, а не повну ітерацію.
4. Профілюй SQL (Django Debug Toolbar, логи БД), а не покладайся на інтуїцію.

#### Висновок:

`QuerySet` - центральна абстракція Django ORM, а lazy evaluation - його ключова
оптимізація. Розуміння моменту виконання SQL критично для продуктивності
реальних Django-проєктів.

</details>

<summary>51. Як працюють filter(), exclude(), get()?</summary>

#### Django

`filter()`, `exclude()` і `get()` - базові методи Django ORM для вибірки даних.
Вони схожі за синтаксисом, але мають різну поведінку і різні гарантії.

#### `filter()`:

- Повертає `QuerySet` з усіма записами, що відповідають умові.
- Якщо збігів немає, повертає порожній `QuerySet` (без винятку).

```python
active_users = User.objects.filter(is_active=True)
cheap_products = Product.objects.filter(price__lt=1000)
```

#### `exclude()`:

- Повертає `QuerySet` із записами, які **не** відповідають умові.

```python
non_archived = Article.objects.exclude(status="archived")
```

#### `get()`:

- Повертає **один** обʼєкт моделі.
- Очікує рівно один збіг.
- Якщо 0 або >1 результатів -> кидає виняток.

```python
user = User.objects.get(pk=1)
```

#### Винятки для `get()`:

1. `Model.DoesNotExist` - нічого не знайдено.
2. `Model.MultipleObjectsReturned` - знайдено більше одного запису.

Приклад безпечної обробки:

```python
from django.http import Http404

def profile_view(request, user_id):
    try:
        user = User.objects.get(pk=user_id)
    except User.DoesNotExist:
        raise Http404("Користувача не знайдено")
```

Або коротко:

```python
from django.shortcuts import get_object_or_404
user = get_object_or_404(User, pk=user_id)
```

#### Коли що використовувати:

1. **`filter()`** - коли результатів може бути багато або нуль.
2. **`exclude()`** - коли треба "відсіяти" частину даних.
3. **`get()`** - коли гарантуєш унікальний результат (PK, unique поле).

#### Практичні поради:

1. Не використовуй `get()` там, де унікальність не гарантована.
2. Для перевірки існування використовуй `exists()`, а не `get()+try/except`.
3. Комбінуй умови через `Q` для складної логіки.

#### Висновок:

`filter()` і `exclude()` працюють із множиною записів через QuerySet, `get()` -
із єдиним обʼєктом і строгими вимогами до унікальності. Правильний вибір методу
знижує кількість помилок і спрощує обробку edge cases.

</details>

<summary>52. Різниця між get() і filter()?</summary>

#### Django

`get()` і `filter()` обидва шукають дані в ORM, але мають принципово різну
семантику: один повертає **один обʼєкт**, інший - **набір обʼєктів (QuerySet)**.

#### `get()`:

1. Повертає один екземпляр моделі.
2. Очікує рівно один збіг.
3. Кидає винятки, якщо записів 0 або більше 1.

```python
user = User.objects.get(pk=10)
```

Можливі винятки:

- `User.DoesNotExist`
- `User.MultipleObjectsReturned`

#### `filter()`:

1. Повертає `QuerySet`.
2. Може повернути 0, 1 або багато записів.
3. Не кидає виняток, якщо нічого не знайдено.

```python
users = User.objects.filter(is_active=True)
```

#### Ключові відмінності:

1. **Тип результату**
- `get()` -> один обʼєкт.
- `filter()` -> QuerySet.

2. **Поводження при відсутності даних**
- `get()` -> виняток.
- `filter()` -> порожній QuerySet.

3. **Сценарій використання**
- `get()` -> унікальні поля (`id`, `uuid`, `email` з `unique=True`).
- `filter()` -> пошук за умовами, де збігів може бути багато.

#### Практичний приклад:

```python
# правильно: унікальний пошук
order = Order.objects.get(id=order_id)

# правильно: список замовлень користувача
orders = Order.objects.filter(user=request.user).order_by("-created_at")
```

#### Типові помилки:

1. Використовувати `get()` для неунікального поля (ризик `MultipleObjectsReturned`).
2. Очікувати один обʼєкт від `filter()` і забувати взяти `.first()` або `get()`.
3. Не обробляти винятки `get()` в API/view.

#### Висновок:

`get()` - для точкового унікального запису, `filter()` - для колекцій і гнучкого
пошуку. Якщо є сумнів у кратності результату, безпечніше починати з `filter()`.

</details>

<summary>53. Як отримати всі записи або один запис?</summary>

#### Django

У Django ORM є різні методи для отримання всіх записів або одного конкретного
запису. Вибір залежить від того, чи очікується колекція, чи одиничний результат.

#### Отримати всі записи:

1. **Усі записи моделі**

```python
articles = Article.objects.all()
```

2. **Усі записи за умовою**

```python
articles = Article.objects.filter(is_published=True)
```

3. **Оптимізований вибір конкретних полів**

```python
articles = Article.objects.values("id", "title")
```

#### Отримати один запис:

1. **Строго один запис через `get()`**

```python
article = Article.objects.get(pk=article_id)
```

Підходить, коли унікальність гарантована.

2. **Безпечний варіант у web-view**

```python
from django.shortcuts import get_object_or_404

article = get_object_or_404(Article, pk=article_id)
```

3. **Перший запис із queryset**

```python
article = Article.objects.filter(is_published=True).first()
```

Повертає обʼєкт або `None`, без винятків.

#### Коли що використовувати:

1. Колекція для списків/таблиць -> `all()` / `filter()`.
2. Один унікальний запис -> `get()`.
3. Один запис у web-flow з 404 -> `get_object_or_404()`.
4. Необовʼязковий одиничний результат -> `first()`.

#### Типові помилки:

1. Використовувати `get()` для неунікального поля.
2. Тягнути `all()` без фільтрів/пагінації на великих таблицях.
3. Не обробляти `DoesNotExist` у API/view.

#### Висновок:

Для "всіх записів" використовуй `QuerySet` (`all/filter`), для "одного запису"
використовуй `get()` або безпечні обгортки (`get_object_or_404`, `first()`)
залежно від вимог до помилок і UX.

<summary>54. Різниця між select_related() і prefetch_related()?</summary>

#### Django

`select_related()` і `prefetch_related()` використовуються для оптимізації ORM,
щоб уникати проблеми N+1 запитів при роботі зі звʼязаними моделями.

#### `select_related()`:

1. Працює для звʼязків **ForeignKey** і **OneToOne** (тобто "один" бік).
2. Робить **SQL JOIN** і дістає повʼязані дані одним запитом.
3. Ефективний для "single-valued" звʼязків.

Приклад:

```python
orders = Order.objects.select_related("customer").all()
for order in orders:
    print(order.customer.email)  # без додаткових запитів
```

#### `prefetch_related()`:

1. Працює для **ManyToMany**, reverse ForeignKey і також може для FK/O2O.
2. Робить кілька запитів (основний + додаткові), а потім "склеює" результати в Python.
3. Ефективний для "multi-valued" звʼязків.

Приклад:

```python
articles = Article.objects.prefetch_related("tags").all()
for article in articles:
    print([tag.name for tag in article.tags.all()])  # без N+1
```

#### Ключова різниця:

1. **Механізм**
- `select_related()` -> JOIN в одному SQL.
- `prefetch_related()` -> кілька SQL + обʼєднання в Python.

2. **Тип звʼязків**
- `select_related()` -> FK/O2O.
- `prefetch_related()` -> M2M, reverse FK (і за потреби інші).

3. **Обсяг даних**
- `select_related()` може "роздувати" рядок через JOIN-и.
- `prefetch_related()` краще для колекцій дочірніх обʼєктів.

#### Комбіноване використання:

```python
orders = (
    Order.objects
    .select_related("customer")
    .prefetch_related("items", "items__product")
)
```

#### Практичне правило вибору:

1. Потрібне поле з FK/O2O -> починай із `select_related`.
2. Потрібен список повʼязаних обʼєктів (M2M/reverse) -> `prefetch_related`.
3. Профілюй запити (debug toolbar/logs), не вгадуй "на око".

#### Типові помилки:

1. Не використовувати жоден із методів і отримати N+1.
2. Застосовувати `select_related()` до M2M (не вирішує задачу).
3. Викликати `.all()` на related manager у циклі без prefetch.

#### Висновок:

`select_related()` оптимізує "одиночні" звʼязки через JOIN, а
`prefetch_related()` - "колекційні" звʼязки через додаткові запити. Правильна
комбінація цих методів дає найбільший приріст продуктивності ORM-коду.

</details>

<summary>55. Як оптимізувати ORM-запити (annotate, aggregate, F(), Q())?</summary>

#### Django

Оптимізація ORM-запитів у Django базується на ідеї: переносити важкі обчислення
в БД і зменшувати кількість запитів та обсяг даних, що витягуються.

#### 1. `annotate()` - обчислення для кожного рядка

- Додає "віртуальне" поле до кожного обʼєкта queryset.
- Корисно для лічильників, сум, дат останньої активності тощо.

```python
from django.db.models import Count

authors = Author.objects.annotate(books_count=Count("books"))
```

#### 2. `aggregate()` - підсумок по всьому queryset

- Повертає словник з агрегованими значеннями.

```python
from django.db.models import Avg, Sum

stats = Order.objects.aggregate(
    total_revenue=Sum("amount"),
    avg_revenue=Avg("amount"),
)
```

#### 3. `F()` - операції над полями на рівні БД

- Дозволяє оновлювати значення без читання обʼєкта в Python.
- Зменшує race conditions у конкурентних оновленнях.

```python
from django.db.models import F

Product.objects.filter(pk=product_id).update(stock=F("stock") - 1)
```

#### 4. `Q()` - складна логіка умов (`AND/OR/NOT`)

- Коли одного `filter(a=..., b=...)` недостатньо.

```python
from django.db.models import Q

users = User.objects.filter(
    Q(is_active=True) & (Q(role="admin") | Q(role="manager"))
).exclude(Q(email__isnull=True) | Q(email=""))
```

#### Комбінований приклад:

```python
from django.db.models import Count, Q

articles = (
    Article.objects
    .filter(is_published=True)
    .annotate(comments_count=Count("comments", filter=Q(comments__is_approved=True)))
    .order_by("-comments_count")
)
```

#### Практичні правила оптимізації:

1. Обчислюй у БД (`annotate/aggregate`), а не Python-циклом.
2. Для масових оновлень використовуй `update(..., F(...))`.
3. Для складних умов використовуй `Q`, а не багаторазові запити.
4. Комбінуй із `select_related/prefetch_related` для уникнення N+1.
5. Витягуй лише потрібні поля (`only`, `values`, `values_list`).

#### Як перевіряти ефект:

1. Дивись SQL через `queryset.query` або `django-debug-toolbar`.
2. Порівнюй кількість запитів до/після.
3. Перевіряй execution plan на боєвій БД (`EXPLAIN ANALYZE`).

#### Типові помилки:

1. Робити агрегації в Python після `list(qs)` замість SQL.
2. Викликати `save()` в циклі замість одного `update()`.
3. Будувати занадто складні анотації без індексів у БД.

#### Висновок:

`annotate`, `aggregate`, `F`, `Q` - ключові інструменти продуктивного ORM-коду.
Вони дозволяють зменшити кількість запитів, прибрати зайві обчислення в Python
і зробити роботу з БД стабільною під навантаженням.

<summary>56. Як перевизначити методи save() і delete()?</summary>

#### Django

У Django можна перевизначати `save()` і `delete()` у моделі, щоб додати
додаткову логіку перед/після збереження або видалення запису.

#### Як перевизначити `save()`:

```python
from django.db import models
from django.utils.text import slugify

class Article(models.Model):
    title = models.CharField(max_length=200)
    slug = models.SlugField(unique=True, blank=True)

    def save(self, *args, **kwargs):
        if not self.slug:
            self.slug = slugify(self.title)
        super().save(*args, **kwargs)
```

Ключове правило: завжди викликати `super().save(*args, **kwargs)`.

#### Як перевизначити `delete()`:

```python
class Document(models.Model):
    file = models.FileField(upload_to="docs/")

    def delete(self, *args, **kwargs):
        storage = self.file.storage
        file_path = self.file.name
        super().delete(*args, **kwargs)
        if file_path:
            storage.delete(file_path)
```

#### Що потрібно памʼятати:

1. `QuerySet.update()` не викликає `save()`.
2. `QuerySet.delete()` не викликає індивідуальний `delete()` кожного обʼєкта в тому ж сенсі, як прямий виклик інстанса.
3. Тому логіку, критичну для цілісності даних, краще дублювати на рівні БД/сервісів.

#### Коли перевизначення доречне:

1. Генерація похідних полів (slug, normalized value).
2. Локальні інваріанти моделі, що не залежать від зовнішніх сервісів.
3. Акуратне прибирання повʼязаних локальних ресурсів.

#### Коли краще інший підхід:

1. Складна бізнес-логіка або оркестрація кількох моделей -> сервісний шар.
2. Крос-модульні побічні ефекти -> signals (обережно) або domain events.
3. Масові операції -> окремі bulk-команди замість інстанс-методів.

#### Практичні правила:

1. `save()/delete()` мають залишатися короткими і передбачуваними.
2. Не роби в них повільні зовнішні HTTP-виклики.
3. Документуй побічні ефекти, щоб команда не отримувала "магію".
4. Покривай такі місця тестами (create/update/delete сценарії).

#### Висновок:

Перевизначення `save()` і `delete()` - потужний, але чутливий інструмент.
Використовуй його для локальної моделі-логіки; складну доменну поведінку краще
тримати у сервісному шарі для прозорості та контрольованості.

</details>

<summary>57. Як працює каскадне видалення (CASCADE)?</summary>

#### Django

Каскадне видалення (`models.CASCADE`) означає: якщо видаляється "батьківський"
обʼєкт, Django автоматично видаляє всі повʼязані "дочірні" записи.

#### Де використовується:

- У `ForeignKey` та `OneToOneField` через параметр `on_delete`.

#### Приклад:

```python
class Author(models.Model):
    name = models.CharField(max_length=120)

class Book(models.Model):
    title = models.CharField(max_length=200)
    author = models.ForeignKey(Author, on_delete=models.CASCADE, related_name="books")
```

Що станеться:

- `author.delete()` -> Django видалить автора і всі його книги.

#### Коли CASCADE доречний:

1. Дочірні дані не мають сенсу без батьківських.
2. Потрібно гарантувати відсутність "сирітських" записів.
3. Простий доменний lifecycle залежних сутностей.

#### Потенційні ризики:

1. Масове випадкове видалення великої кількості даних.
2. Складний аудит/відновлення після помилки.
3. Несподівані побічні ефекти в повʼязаних доменах.

#### Альтернативи `on_delete`:

1. `PROTECT` - забороняє видалення батька, якщо є залежності.
2. `SET_NULL` - ставить `NULL` у дочірньому полі (`null=True` обовʼязково).
3. `SET_DEFAULT` - підставляє значення за замовчуванням.
4. `RESTRICT` - обмежує видалення при наявних посиланнях (схоже на суворий protect).
5. `DO_NOTHING` - відповідальність на стороні БД/розробника.

#### Практичні рекомендації:

1. Для критичних бізнес-даних частіше використовують `PROTECT/RESTRICT`.
2. Для явно залежних технічних сутностей (наприклад, тимчасові записи) часто
   підходить `CASCADE`.
3. Перед масовими delete запускай перевірку, що саме буде видалено.
4. Розглянь soft delete, якщо потрібен історичний аудит.

#### Висновок:

`CASCADE` - зручний механізм підтримки цілісності, але він агресивний за
поведінкою. Використовуй його там, де життєвий цикл дочірніх даних повністю
залежить від батьківського обʼєкта.

</details>

<summary>58. Як працюють транзакції в Django?</summary>

#### Django

Транзакції в Django гарантують, що група змін у БД виконується як єдине ціле:
або застосовуються всі зміни (`commit`), або жодна (`rollback`) при помилці.

#### Основний інструмент:

- `transaction.atomic()`

#### Базовий приклад:

```python
from django.db import transaction

@transaction.atomic
def create_order(user, items):
    order = Order.objects.create(user=user)
    for item in items:
        OrderItem.objects.create(order=order, product=item["product"], qty=item["qty"])
    return order
```

Якщо всередині станеться виняток, всі зміни в блоці будуть відкотені.

#### Контекстний менеджер:

```python
from django.db import transaction

def transfer(from_acc, to_acc, amount):
    with transaction.atomic():
        from_acc.balance -= amount
        to_acc.balance += amount
        from_acc.save()
        to_acc.save()
```

#### Вкладені транзакції і savepoints:

- Вкладені `atomic()` створюють savepoint-и.
- Помилка у внутрішньому блоці може відкотити лише частину операцій до savepoint.

#### Режим `ATOMIC_REQUESTS`:

У `settings.py` можна увімкнути транзакцію "на весь HTTP-запит":

```python
DATABASES["default"]["ATOMIC_REQUESTS"] = True
```

Це зручно, але не завжди оптимально для продуктивності на складних endpoint.

#### Практичні рекомендації:

1. Обгортай у `atomic()` лише справді звʼязані операції.
2. Тримай транзакції короткими (мінімум зовнішніх викликів усередині).
3. Не роби HTTP/API-запити всередині транзакції.
4. Для конкурентних сценаріїв використовуй `select_for_update()`.

#### Приклад блокування рядка:

```python
with transaction.atomic():
    account = Account.objects.select_for_update().get(pk=account_id)
    account.balance = F("balance") - amount
    account.save(update_fields=["balance"])
```

#### Типові помилки:

1. Частину операції виконати до `atomic()`, а частину після.
2. Довгі транзакції, що тримають locks і блокують інші запити.
3. Очікувати, що транзакція покриває зовнішні системи (черги, сторонні API).
4. Ігнорувати ізоляційні рівні БД у high-concurrency сценаріях.

#### Висновок:

Транзакції в Django - базовий механізм консистентності даних. `transaction.atomic()`
має бути стандартом для критичних багатокрокових операцій, особливо у фінансових,
складських і workflow-доменах.

</details>

<summary>59. Чи підтримує Django NoSQL?</summary>

#### Django

Коротко: Django **нативно орієнтований на реляційні БД** (PostgreSQL, MySQL,
SQLite, Oracle) через свій ORM. NoSQL підтримується не "з коробки" на тому ж
рівні, але інтеграція можлива.

#### Що це означає на практиці:

1. **Основний ORM Django = SQL-first**
- Більшість фіч (migrations, relations, admin, constraints) розраховані на SQL.

2. **NoSQL можна підключати через сторонні бібліотеки**
- Для MongoDB та інших рішень є сторонні адаптери/ODM, але сумісність із
  Django-екосистемою часто неповна.

3. **Найпоширеніший production-підхід - гібридний**
- Основна транзакційна модель у PostgreSQL.
- NoSQL-сховище для спеціалізованих задач (кеш, документи, пошук, події).

#### Типові ролі NoSQL у Django-проєктах:

1. Redis - кеш, rate limit, брокер черг, сесії.
2. Elasticsearch/OpenSearch - повнотекстовий пошук і аналітичні запити.
3. Документні БД - окремі домени з гнучкою схемою (за потреби).

#### Коли SQL у Django зазвичай кращий:

- Складні звʼязки між сутностями.
- Транзакційність і строгі гарантії консистентності.
- Багато стандартного CRUD + адмінка + чітка доменна модель.

#### Коли NoSQL може бути доречний:

- Дуже гнучка/мінлива схема даних.
- Високошвидкісний key-value доступ.
- Пошукові/лог-аналітичні сценарії, де SQL неефективний.

#### Практичні рекомендації техліда:

1. Не замінюй реляційну БД на NoSQL "за трендом".
2. Починай з PostgreSQL як надійного default для Django.
3. Додавай NoSQL точково під конкретний нефункціональний запит (latency, search, scale).
4. Окремо проектуй консистентність між SQL і NoSQL (outbox/events/retry).

#### Висновок:

Django не є NoSQL-first фреймворком, але добре працює в гібридній архітектурі.
Оптимальна стратегія для більшості продуктів: SQL як основа доменних даних +
NoSQL як спеціалізоване доповнення під конкретні навантаження.

</details>

<summary>60. Які типові винятки Django варто знати?</summary>

#### Django

У Django є набір типових винятків, які регулярно трапляються в реальних
проєктах. Знання цих винятків прискорює дебаг і дозволяє правильно будувати
обробку помилок у view/API.

#### Найважливіші винятки:

1. **`Model.DoesNotExist`**
- Виникає при `Model.objects.get(...)`, якщо запис не знайдено.

2. **`Model.MultipleObjectsReturned`**
- `get()` повернув більше одного запису.

3. **`ValidationError`**
- Помилки валідації форм, моделей, кастомних валідаторів.

4. **`IntegrityError`**
- Порушення обмежень БД (`UNIQUE`, `FK`, `NOT NULL`, `CHECK`).

5. **`ObjectDoesNotExist`**
- Базовий виняток для всіх `*.DoesNotExist`.

6. **`PermissionDenied`**
- Немає доступу до ресурсу (повертається 403).

7. **`SuspiciousOperation`**
- Підозріла/небезпечна активність (наприклад, невалідний host/header).

8. **`Http404`**
- Використовується для явного повернення 404.

9. **`ImproperlyConfigured`**
- Помилка конфігурації settings, URL, middleware, apps.

10. **`OperationalError` (db backend)**
- Проблеми з підключенням/доступністю БД, таймаути, мережеві збої.

#### Приклади обробки:

```python
from django.core.exceptions import ValidationError
from django.db import IntegrityError
from django.http import Http404

try:
    user = User.objects.get(pk=user_id)
except User.DoesNotExist:
    raise Http404("Користувача не знайдено")

try:
    create_order(...)
except ValidationError as exc:
    return JsonResponse({"errors": exc.message_dict}, status=400)
except IntegrityError:
    return JsonResponse({"error": "Порушено цілісність даних"}, status=409)
```

#### Практичні правила:

1. Для 404 у view використовуй `get_object_or_404`.
2. Не ковтай винятки без логування.
3. Розділяй користувацькі помилки (4xx) і системні (5xx).
4. У production не віддавай stack trace клієнту.
5. Для API стандартизуй формат помилок (код, message, details).

#### Типові анти-патерни:

1. `except Exception:` без конкретики.
2. Ігнорування `IntegrityError` і повторні неконсистентні операції.
3. Змішування доменних помилок і технічних винятків в один "400 unknown".

#### Висновок:

Грамотна робота з винятками в Django - це про контроль якості API, стабільність
сервісу і швидке розслідування проблем у production. Базові винятки треба знати
й обробляти системно, а не точково.

</details>

<summary>61. Що таке Django Admin? Як зареєструвати модель?</summary>

#### Django

`Django Admin` - це вбудована адміністративна панель для керування даними моделі

```python
# app/admin.py
from django.contrib import admin
from .models import Article
```

```python
from django.contrib import admin
from .models import Article

@admin.register(Article)
class ArticleAdmin(admin.ModelAdmin):

    list_filter = ("is_published", "created_at")
    search_fields = ("title",)
    ordering = ("-created_at",)
```

1. Міграції застосовані:

```bash
python manage.py migrate
```

```bash
python manage.py createsuperuser
```

2. Обмежуй доступ до admin за ролями/permissions.

4. Не використовуй admin як публічний інтерфейс для кінцевих користувачів.

#### Висновок:

Django Admin - потужний інструмент для внутрішнього керування даними. Реєстрація
моделі займає хвилини, а з базовим `ModelAdmin` адмінка швидко стає справжнім
продуктивним back-office для команди.

<summary>62. Що таке superuser?</summary>

#### Django

`Superuser` у Django - це користувач із максимальними адміністративними правами
в системі, зазвичай для доступу до Django Admin та повного керування даними.

#### Ключові характеристики superuser:

1. `is_superuser = True`
2. `is_staff = True`
3. Має всі permissions без потреби окремо призначати кожен дозвіл.
4. Може керувати користувачами, групами, моделями і контентом через `/admin/`.

#### Як створити superuser:

```bash
python manage.py createsuperuser
```

Далі Django попросить:
- username
- email (залежно від моделі)
- password

#### Де використовується:

1. Початкове адміністрування проєкту.
2. Налаштування ролей/груп і доступів.
3. Операційна підтримка через Django Admin.

#### Важливі зауваження з безпеки:

1. Не використовуй superuser-акаунт для щоденної операційної роботи.
2. Кількість superuser в production має бути мінімальною.
3. Використовуй сильні паролі + 2FA (якщо доступно через SSO/адмін-рішення).
4. Логуй дії адміністратора і контролюй доступ за IP/VPN.
5. Не створюй "тимчасових" суперкористувачів без процесу видалення.

#### Практична роль у команді:

- Superuser потрібен для bootstrap етапу.
- У сталому процесі більшість задач краще виконувати через `staff` + групи +
  granular permissions, а не через повний доступ.

#### Висновок:

Superuser - це "root-користувач" у Django. Інструмент критично важливий для
адміністрування, але його доступ повинен бути максимально обмежений і контрольований.

</details>

<summary>63. Як кастомізувати Django Admin?</summary>

#### Django

Кастомізація Django Admin виконується через клас `ModelAdmin` і повʼязані
механізми (inlines, actions, permissions, form overrides). Це дозволяє перетворити
базову адмінку на ефективний back-office інструмент.

#### Базова кастомізація `ModelAdmin`:

```python
from django.contrib import admin
from .models import Order

@admin.register(Order)
class OrderAdmin(admin.ModelAdmin):
    list_display = ("id", "user", "status", "total", "created_at")
    list_filter = ("status", "created_at")
    search_fields = ("id", "user__email")
    ordering = ("-created_at",)
    list_per_page = 50
```

#### Найпоширеніші інструменти:

1. `list_display` - які колонки видно в списку.
2. `list_filter` - фільтри справа.
3. `search_fields` - пошук по полях/звʼязках.
4. `readonly_fields` - поля лише для читання.
5. `fieldsets` - групування полів у формі редагування.
6. `actions` - масові дії над вибраними записами.

#### Inline-редагування повʼязаних моделей:

```python
class OrderItemInline(admin.TabularInline):
    model = OrderItem
    extra = 0

@admin.register(Order)
class OrderAdmin(admin.ModelAdmin):
    inlines = [OrderItemInline]
```

#### Кастомні дії (actions):

```python
@admin.action(description="Позначити як опубліковані")
def make_published(modeladmin, request, queryset):
    queryset.update(is_published=True)
```

#### Обмеження доступу:

- `has_view_permission`, `has_change_permission`, `has_delete_permission`
- `get_queryset()` для data-scoping (наприклад, менеджер бачить лише свої обʼєкти).

#### Оптимізація продуктивності адмінки:

1. Використовуй `list_select_related` для FK у `list_display`.
2. Обережно з `search_fields` на великих таблицях (індекси обовʼязкові).
3. Зменшуй "важкі" обчислення в list view.
4. Для дуже великих довідників використовуй `autocomplete_fields`.

#### Додаткова кастомізація:

- Перевизначення `form`/`get_form`.
- Перевизначення `save_model`/`delete_model`.
- Кастомні admin templates для специфічного UI.

#### Висновок:

Django Admin добре масштабується через `ModelAdmin`: від простої реєстрації до
повноцінного внутрішнього кабінету з ролями, фільтрами, масовими діями та
контрольованою продуктивністю.

<summary>64. Які є компоненти Django Admin, індивідуальні налаштування?</summary>

#### Django

Django Admin складається з кількох ключових компонентів, які можна гнучко
налаштовувати під бізнес-процеси команди: список обʼєктів, форма редагування,
фільтри, пошук, дії, права доступу, inlines і глобальна конфігурація admin site.

#### Основні компоненти Django Admin:

1. **Admin Site (`admin.site`)**
- Глобальна адмін-панель, її header/title/index можна кастомізувати.

2. **`ModelAdmin`**
- Конфігурація відображення конкретної моделі в admin.

3. **Change List**
- Сторінка списку обʼєктів (`list_display`, `list_filter`, `search_fields`).

4. **Change Form**
- Форма створення/редагування (`fields`, `fieldsets`, `readonly_fields`).

5. **Inlines**
- Редагування повʼязаних моделей на одній сторінці.

6. **Actions**

7. **Permissions**
- Доступ до перегляду/редагування/видалення за ролями.

#### Приклад індивідуальних налаштувань:

```python
from django.contrib import admin
from .models import Customer

@admin.register(Customer)
class CustomerAdmin(admin.ModelAdmin):
    list_display = ("id", "email", "is_active", "created_at")
    list_filter = ("is_active", "created_at")
    search_fields = ("email", "first_name", "last_name")
    readonly_fields = ("created_at", "updated_at")
    fieldsets = (
        ("Основне", {"fields": ("email", "first_name", "last_name")}),
        ("Статус", {"fields": ("is_active",)}),
        ("Службове", {"fields": ("created_at", "updated_at")}),

```

#### Глобальні налаштування admin site:

```python
from django.contrib import admin

admin.site.site_header = "Admin Panel"
admin.site.site_title = "Backoffice"
admin.site.index_title = "Керування системою"
```

#### Індивідуальна поведінка за ролями:

- `get_queryset()` - обмежити, що бачить конкретний staff-користувач.
- `get_readonly_fields()` - зробити частину полів read-only залежно від ролі.
- `has_change_permission()` / `has_delete_permission()` - granular контроль прав.

#### Компоненти, які часто додають у production:

1. `autocomplete_fields` для важких FK-полів.
2. `list_select_related` для оптимізації list-view.
3. Кастомні `form` і `formfield_overrides` для кращого UX.
4. Кастомні templates для специфічних сценаріїв бек-офісу.

#### Висновок:

Сила Django Admin у тому, що з базових компонентів можна зібрати керований
внутрішній інтерфейс під конкретні процеси команди: швидкий, безпечний і
операційно зручний.

<summary>65. Що таке Django Admin actions та кастомізація?</summary>

#### Django

`Admin actions` у Django - це масові операції над вибраними обʼєктами в списку
адмінки. Вони дозволяють швидко виконувати типові дії без ручного редагування
кожного запису.

#### Що таке action:

- Функція або метод, який приймає:
  `modeladmin`, `request`, `queryset`.
- Викликається для набору обʼєктів, які admin-користувач відмітив у списку.

#### Базовий приклад action:

```python
from django.contrib import admin
from .models import Article

@admin.action(description="Опублікувати вибрані статті")
def make_published(modeladmin, request, queryset):
    queryset.update(is_published=True)

@admin.register(Article)
class ArticleAdmin(admin.ModelAdmin):
    list_display = ("id", "title", "is_published")
    actions = [make_published]
```

#### Приклад action як методу `ModelAdmin`:

```python
@admin.register(Order)
class OrderAdmin(admin.ModelAdmin):
    actions = ["mark_as_paid"]

    @admin.action(description="Позначити як оплачені")
    def mark_as_paid(self, request, queryset):
        updated = queryset.update(status="paid")
        self.message_user(request, f"Оновлено замовлень: {updated}")
```

#### Кастомізація actions:

1. Власний `description` для UI.
2. Умови доступності дії залежно від ролі/стану.
3. Повернення проміжної сторінки підтвердження (custom admin view).
4. Логування виконаних масових дій.

#### Практичні рекомендації:

1. Для масових змін віддавай перевагу `queryset.update()` (швидше, менше SQL).
2. Якщо потрібна бізнес-логіка на кожному обʼєкті, ітеруй обережно в транзакції.
3. Показуй підсумок через `message_user`.
4. Обмежуй небезпечні actions для неадміністративних ролей.

#### Важливі нюанси:

1. `queryset.update()` не викликає `save()` і сигнали `post_save`.
2. Масове видалення через стандартну дію може бути ризиковим без додаткового контролю.
3. Для критичних операцій краще додавати крок підтвердження.

#### Типові кейси actions:

- Publish/Unpublish контенту.
- Batch-призначення статусу.
- Експорт вибраних записів у CSV.
- Запуск фонової обробки для selected IDs.

#### Висновок:

Admin actions - це швидкий інструмент операційної автоматизації в Django Admin.
За правильної кастомізації вони суттєво прискорюють роботу команди і зменшують
кількість ручних рутинних дій.

</details>

<summary>66. Що таке middleware і як воно працює?</summary>

#### Django

`Middleware` у Django - це шар обробки між HTTP-запитом/відповіддю і view.
Воно дозволяє виконувати код **до** того, як запит потрапить у view, і **після**
того, як сформована відповідь повертається клієнту.

#### Як працює middleware-пайплайн:

1. Запит приходить у Django.
2. Проходить послідовно через middleware (request phase).
3. Потрапляє в URL resolver і view.
4. Відповідь повертається назад через middleware у зворотному порядку (response phase).
5. Відправляється клієнту.

#### Важлива деталь:

- Порядок middleware в `settings.py` (`MIDDLEWARE = [...]`) критично важливий.
- Верхні middleware бачать запит першими, а відповідь - останніми.

#### Типові задачі middleware:

1. Безпека (security headers, host checks).
2. Сесії та автентифікація.
3. CSRF-захист.
4. Локалізація (locale/language).
5. Логування, request-id, метрики.
6. Кешування на рівні запиту/відповіді.

#### Скелет middleware:

```python
class SimpleMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        # до view
        response = self.get_response(request)
        # після view
        return response
```

#### Де підключається:

```python
MIDDLEWARE = [
    "django.middleware.security.SecurityMiddleware",
    "django.contrib.sessions.middleware.SessionMiddleware",
    "django.middleware.common.CommonMiddleware",
    # ...
]
```

#### Практичні правила:

1. Тримай middleware легким і швидким.
2. Не клади туди важку бізнес-логіку домену.
3. Уникай повільних зовнішніх викликів у middleware.
4. Уважно контролюй порядок middleware у списку.

#### Типові помилки:

1. Занадто "розумне" middleware з побічними ефектами.
2. Неправильний порядок, що ламає auth/csrf/session flow.
3. Дублювання логіки між middleware і view.

#### Висновок:

Middleware - це глобальний перехоплювач запитів/відповідей у Django. Його
сильна сторона - централізовані cross-cutting задачі (безпека, сесії, логування),
а не бізнес-логіка конкретних endpoint.

</details>

<summary>67. Які існують вбудовані middleware?</summary>

#### Django

У Django є набір вбудованих middleware, які покривають базові потреби вебдодатка:
безпека, сесії, аутентифікація, CSRF, кешування, локалізація.

#### Найпоширеніші built-in middleware:

1. **`django.middleware.security.SecurityMiddleware`**
- Security headers, HTTPS-related налаштування, базові захисні механізми.

2. **`django.contrib.sessions.middleware.SessionMiddleware`**
- Підтримка серверних сесій користувача.

3. **`django.middleware.common.CommonMiddleware`**
- Загальні HTTP-поведінки (наприклад, `APPEND_SLASH`, `PREPEND_WWW`).

4. **`django.middleware.csrf.CsrfViewMiddleware`**
- Захист від CSRF-атак для state-changing запитів.

5. **`django.contrib.auth.middleware.AuthenticationMiddleware`**
- Додає `request.user` і інтегрує auth-систему у request flow.

6. **`django.contrib.messages.middleware.MessageMiddleware`**
- Підтримка flash-повідомлень (`messages.success`, `messages.error`).

7. **`django.middleware.clickjacking.XFrameOptionsMiddleware`**
- Захист від clickjacking через заголовок `X-Frame-Options`.

8. **`django.middleware.locale.LocaleMiddleware`**
- Визначення мови/локалі для i18n.

9. **`django.middleware.gzip.GZipMiddleware`**
- Стиснення відповідей (обережно з уже стисненим контентом).

10. **`django.middleware.cache.UpdateCacheMiddleware` / `FetchFromCacheMiddleware`**
- Site-wide кешування на рівні middleware.

#### Типовий порядок у `MIDDLEWARE` (скорочено):

```python
MIDDLEWARE = [
    "django.middleware.security.SecurityMiddleware",
    "django.contrib.sessions.middleware.SessionMiddleware",
    "django.middleware.common.CommonMiddleware",
    "django.middleware.csrf.CsrfViewMiddleware",
    "django.contrib.auth.middleware.AuthenticationMiddleware",
    "django.contrib.messages.middleware.MessageMiddleware",
    "django.middleware.clickjacking.XFrameOptionsMiddleware",
]
```

#### Практичні зауваження:

1. Порядок middleware важливий і впливає на поведінку всього застосунку.
2. Не видаляй без розуміння наслідків `SecurityMiddleware`, `CsrfViewMiddleware`,
   `AuthenticationMiddleware`.
3. Для production перевіряй заголовки безпеки та cache-policy після змін middleware.

#### Висновок:

Вбудовані middleware Django покривають більшість інфраструктурних потреб "з коробки".
Правильний набір і порядок middleware напряму впливають на безпеку, стабільність
і поведінку застосунку в production.

</details>

<summary>68. Як створити власне middleware?</summary>

#### Django

Власне middleware в Django створюють як клас, який приймає `get_response` і
перехоплює `request/response` у глобальному пайплайні обробки запитів.

#### Мінімальний шаблон middleware:

```python
# app/middleware.py
class RequestIdMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        # логіка до view
        response = self.get_response(request)
        # логіка після view
        return response
```

#### Приклад із request-id:

```python
import uuid

class RequestIdMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        request.request_id = str(uuid.uuid4())
        response = self.get_response(request)
        response["X-Request-ID"] = request.request_id
        return response
```

#### Підключення у `settings.py`:

```python
MIDDLEWARE = [
    # ...
    "app.middleware.RequestIdMiddleware",
]
```

#### Де ставити у списку:

1. Middleware безпеки/сесій - зазвичай вище.
2. Кастомне логування/request-id - часто після базових security/session middleware.
3. Порядок важливий: впливає на те, що middleware бачить у `request`.

#### Що можна робити у middleware:

- Додавати headers.
- Логувати запит/відповідь.
- Додавати tracing-контекст.
- Відхиляти запит за глобальними правилами.

#### Чого не варто робити:

1. Складна бізнес-логіка конкретного endpoint.
2. Повільні зовнішні HTTP-виклики.
3. Важкі операції, що блокують кожен запит.

#### Висновок:

Кастомне middleware - інструмент для cross-cutting задач на рівні всього
застосунку. Роби його легким, передбачуваним і правильно розташовуй у
`MIDDLEWARE`, щоб уникнути побічних ефектів.

</details>

<summary>69. Що таке signals і коли їх використовувати?</summary>

#### Django

`Signals` у Django - це механізм подій, який дозволяє запускати код у відповідь
на певні дії (наприклад, збереження або видалення моделі) без прямого виклику
цієї логіки у view чи сервісі.

#### Як це працює:

1. Виникає подія (signal), наприклад `post_save`.
2. Receiver-функція, підписана на цю подію, виконується автоматично.

#### Найуживаніші сигнали:

1. `pre_save`, `post_save`
2. `pre_delete`, `post_delete`
3. `m2m_changed`
4. `request_started`, `request_finished` (рідше для app-логіки)

#### Приклад `post_save`:

```python
# app/signals.py
from django.db.models.signals import post_save
from django.dispatch import receiver
from django.contrib.auth import get_user_model
from .models import Profile

User = get_user_model()

@receiver(post_save, sender=User)
def create_profile(sender, instance, created, **kwargs):
    if created:
        Profile.objects.create(user=instance)
```

Імпорт/реєстрація в `apps.py`:

```python
from django.apps import AppConfig

class UsersConfig(AppConfig):
    name = "users"

    def ready(self):
        import users.signals  # noqa
```

#### Коли signals доречні:

1. Слабкозвʼязані побічні дії (наприклад, створити профіль після створення користувача).
2. Технічні hooks, що не повинні ускладнювати view.
3. Події, де важлива розвʼязка між модулями.

#### Коли краще не використовувати:

1. Критична бізнес-логіка, де потрібна явна послідовність кроків.
2. Складні оркестрації між кількома сервісами.
3. Логіка, яка повинна бути прозорою в одному явному use-case.

#### Типові проблеми signals:

1. "Магія" та неочікувані побічні ефекти.
2. Складний дебаг у великих проєктах.
3. Подвійні виклики при неакуратній реєстрації.
4. Важко контролювати порядок виконання багатьох receivers.

#### Практичні правила:

1. Сигнали мають бути короткими і швидкими.
2. Довгі/важкі задачі відправляй у чергу (Celery/RQ).
3. Документуй на які події підписаний модуль.
4. Не ховай у signal головну бізнес-логіку транзакції.

#### Висновок:

Signals у Django корисні для помірних побічних подій і слабкого звʼязування
модулів. Для core-бізнес процесів зазвичай надійніше використовувати явний
сервісний шар, де flow видно прямо в коді.

<summary>70. Різниця між signals і middleware.</summary>

#### Django

`Middleware` і `signals` вирішують різні задачі, хоча обидва виглядають як
"місце для додаткової логіки". Ключова різниця - **рівень подій**, на якому вони
працюють.

#### Middleware:

1. Працює на рівні **HTTP request/response**.
2. Виконується для кожного запиту в глобальному пайплайні.
3. Використовується для cross-cutting вебзадач.

Типові кейси:

- security headers
- auth/session flow
- logging/tracing/request-id
- locale/caching/compression

#### Signals:

1. Працюють на рівні **подій у моделях/фреймворку**.
2. Спрацьовують при діях типу `post_save`, `post_delete`, `m2m_changed`.
3. Використовуються для реакції на події доменної моделі.

Типові кейси:

- створення профілю після створення користувача
- синхронізація допоміжних сутностей
- lightweight hooks після змін даних

#### Головна відмінність:

- Middleware відповідає на запит/відповідь.
- Signals відповідають на події даних/моделей.

#### Коли що обирати:

1. Потрібна глобальна HTTP-поведінка -> **middleware**.
2. Потрібна реакція на зміну моделі -> **signals**.
3. Потрібен явний контроль бізнес-flow -> **сервісний шар**, не signals/middleware.

#### Таблиця-порівняння:

| Критерій | Middleware | Signals |
| --- | --- | --- |
| Рівень | HTTP pipeline | Model/framework events |
| Точка запуску | До/після view | До/після save/delete тощо |
| Прозорість flow | Вища (через `MIDDLEWARE`) | Нижча (може бути "магія") |
| Тип задач | Інфраструктурні | Подієві побічні дії |

#### Типові помилки:

1. Класти бізнес-логіку моделі в middleware.
2. Ховати критичний бізнес-flow у signals без явної оркестрації.
3. Дублювати одну й ту ж логіку в обох шарах.

#### Висновок:

Middleware - для HTTP-інфраструктури, signals - для реакції на події моделі.
Для важливої бізнес-логіки краще явні сервіси: вони найпрозоріші, найтестованіші
і найменш схильні до прихованих побічних ефектів.

</details>

<summary>71. Як працює система аутентифікації та авторизації?</summary>

#### Django

У Django система доступу складається з двох частин:

1. **Аутентифікація (authentication)** - хто ти?
2. **Авторизація (authorization)** - що тобі дозволено?

#### Як працює аутентифікація:

1. Користувач надсилає логін/пароль.
2. Django перевіряє облікові дані через `authenticate()`.
3. Якщо успішно - викликається `login()`, створюється сесія.
4. У наступних запитах `AuthenticationMiddleware` додає `request.user`.

Базовий приклад:

```python
from django.contrib.auth import authenticate, login

user = authenticate(request, username=username, password=password)
if user is not None:
    login(request, user)
```

#### Як працює авторизація:

- Django перевіряє права користувача через permissions, групи і role-based правила.
- Базові permissions для моделей:
  `add`, `change`, `delete`, `view`.

Перевірка права:

```python
if request.user.has_perm("orders.change_order"):
    ...
```

#### Інструменти контролю доступу:

1. `@login_required` - доступ лише для авторизованих.
2. `@permission_required("app.perm_codename")`
3. `LoginRequiredMixin`, `PermissionRequiredMixin` для CBV.
4. Групи (`Group`) для рольового керування permission-наборами.

#### Сесійна модель (за замовчуванням):

- Після `login()` Django створює сесію в БД/кеші.
- Ідентифікатор сесії зберігається в cookie.
- На кожен запит користувач ідентифікується через цю сесію.

#### Де це налаштовується:

- `AUTHENTICATION_BACKENDS`
- `LOGIN_URL`, `LOGIN_REDIRECT_URL`, `LOGOUT_REDIRECT_URL`
- `AUTH_USER_MODEL` (якщо кастомна модель користувача)

#### Практичні рекомендації:

1. Не перевіряй доступ лише на фронтенді - бекенд має бути джерелом істини.
2. Групуй права за ролями, а не видавай індивідуально кожному юзеру.
3. Для API окремо продумуй token/session auth і policy доступу.
4. Логуй чутливі адмін-дії для аудиту.

#### Висновок:

У Django аутентифікація відповідає за ідентифікацію користувача, авторизація -
за його права. Разом вони формують безпечну та масштабовану модель доступу для
web і API застосунків.

</details>

<summary>72. Як реалізувати реєстрацію користувачів?</summary>

#### Django

Реєстрація користувачів у Django зазвичай складається з форми, view-логіки,
створення користувача з хешованим паролем і (опційно) автоматичного входу.

#### Базовий підхід:

1. Створити форму реєстрації (`UserCreationForm` або кастомна).
2. Обробити форму у view (`GET`/`POST`).
3. Зберегти користувача.
4. Перенаправити на login або виконати автологін.

#### Приклад форми:

```python
# users/forms.py
from django.contrib.auth.forms import UserCreationForm
from django.contrib.auth import get_user_model

User = get_user_model()

class SignUpForm(UserCreationForm):
    class Meta(UserCreationForm.Meta):
        model = User
        fields = ("username", "email")
```

#### Приклад view:

```python
# users/views.py
from django.contrib.auth import login
from django.shortcuts import redirect, render
from .forms import SignUpForm

def signup_view(request):
    if request.method == "POST":
        form = SignUpForm(request.POST)
        if form.is_valid():
            user = form.save()  # пароль хешується автоматично
            login(request, user)  # опційно: одразу логінимо
            return redirect("home")
    else:
        form = SignUpForm()
    return render(request, "users/signup.html", {"form": form})
```

#### Шаблон:

```django
<form method="post">
  {% csrf_token %}
  {{ form.as_p }}
  <button type="submit">Зареєструватися</button>
</form>
```

#### URL:

```python
path("signup/", signup_view, name="signup")
```

#### Важливі правила безпеки:

1. Ніколи не зберігай пароль вручну в полі plain text.
2. Використовуй `UserCreationForm` або `set_password()`.
3. Додай захист від brute force (rate limiting / captcha, де потрібно).
4. Для production бажано підтвердження email перед активацією акаунта.

#### Розширення реєстрації:

1. Профіль користувача (через `OneToOne`).
2. Підтвердження email токеном.
3. Соціальний логін (Google/GitHub) через allauth.
4. Audit log подій реєстрації.

#### Висновок:

У Django реєстрація реалізується швидко через вбудовані auth-форми і view-flow.
Ключове - коректна робота з паролями, CSRF-захист і продуманий процес активації
акаунтів у production.

<summary>73. Як створити власну модель користувача?</summary>

#### Django

Власну модель користувача в Django створюють, коли стандартного `User` не
вистачає (наприклад, потрібен email як логін, додаткові обовʼязкові поля,
кастомна логіка ідентифікації).

#### Ключове правило:

`AUTH_USER_MODEL` потрібно визначити **на початку проєкту**, до перших міграцій.
Пізня заміна дуже дорога й ризикована.

#### Варіант 1 (рекомендований у 90% випадків): `AbstractUser`

- Наслідуєш вбудований користувач і додаєш свої поля.
- Найпростіший і найстабільніший шлях.

```python
# users/models.py
from django.contrib.auth.models import AbstractUser
from django.db import models

class User(AbstractUser):
    phone = models.CharField(max_length=30, blank=True)
    timezone = models.CharField(max_length=64, blank=True, default="UTC")
```

У `settings.py`:

```python
AUTH_USER_MODEL = "users.User"
```

#### Варіант 2: `AbstractBaseUser` + `PermissionsMixin`

- Повний контроль над моделлю і логін-полем.
- Значно складніше: потрібен кастомний `UserManager`, `USERNAME_FIELD`,
  `REQUIRED_FIELDS`, сумісність з admin/forms/auth flow.

Використовуй тільки коли справді потрібна нестандартна auth-модель.

#### Приклад посилання на кастомного користувача в моделях:

```python
from django.conf import settings

class Order(models.Model):
    user = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE)
```

#### У коді замість прямого імпорту `User`:

```python
from django.contrib.auth import get_user_model
User = get_user_model()
```

#### Практичні рекомендації:

1. Якщо не впевнений - створи кастомну модель через `AbstractUser` одразу.
2. Не використовуй `from django.contrib.auth.models import User` у доменному коді.
3. Переконайся, що форми/admin/auth backend працюють із твоєю моделлю.
4. Покрий тестами реєстрацію, логін, reset password, permissions.

#### Типові помилки:

1. Спроба змінити модель користувача після запуску проєкту в production.
2. Жорсткий імпорт стандартного `User` у моделях і міграціях.
3. Використання `AbstractBaseUser` без реальної потреби та без повного налаштування.

#### Висновок:

Правильна кастомна user-модель у Django - це стратегічне рішення на старті.
Найбезпечніший шлях: `AbstractUser` + `AUTH_USER_MODEL` + дисципліна використання
`get_user_model()` і `settings.AUTH_USER_MODEL` по всьому проєкту.

</details>

<summary>74. Для чого використовується login_required?</summary>

#### Django

`login_required` - це декоратор, який обмежує доступ до view тільки для
авторизованих користувачів.

#### Як працює:

1. Якщо користувач авторизований -> view виконується.
2. Якщо ні -> редірект на сторінку логіну (`LOGIN_URL`) з параметром `next`.

#### Приклад для FBV:

```python
from django.contrib.auth.decorators import login_required

@login_required
def dashboard(request):
    return render(request, "dashboard.html")
```

#### Кастомний login URL:

```python
@login_required(login_url="/auth/login/")
def billing_page(request):
    ...
```

#### Аналог для CBV:

```python
from django.contrib.auth.mixins import LoginRequiredMixin
from django.views.generic import TemplateView

class DashboardView(LoginRequiredMixin, TemplateView):
    template_name = "dashboard.html"
```

#### Налаштування в `settings.py`:

```python
LOGIN_URL = "login"
LOGIN_REDIRECT_URL = "home"
```

#### Для чого це використовують:

1. Захист приватних сторінок (кабінет, замовлення, налаштування профілю).
2. Базовий контроль доступу в admin-like розділах.
3. Запобігання неавторизованим POST-діям.

#### Що `login_required` НЕ робить:

1. Не перевіряє ролі/permissions (тільки факт логіну).
2. Не замінює детальний авторизаційний контроль.

Для прав використовуй:

- `permission_required`
- `user_passes_test`
- `PermissionRequiredMixin`

#### Практичні поради:

1. Став `login_required` на всі приватні endpoint за замовчуванням.
2. Не покладайся лише на приховування кнопок на фронтенді.
3. Перевіряй, що `next`-redirect безпечний (Django обробляє це стандартно).

#### Висновок:

`login_required` - базовий і обовʼязковий захист приватних view у Django.
Він вирішує "чи увійшов користувач", а деталізацію прав доступу потрібно
будувати додатковими механізмами авторизації.

</details>

<summary>75. Як працюють сесії і cookies?</summary>

#### Django

У Django сесії та cookies використовуються для збереження стану між запитами,
бо HTTP сам по собі stateless.

#### Що таке cookies:

- Невеликі дані, які браузер зберігає на клієнті.
- Надсилаються назад серверу в кожному наступному запиті до того ж домену.
- Часто містять session ID, а не самі чутливі дані.

#### Що таке сесія:

- Серверне сховище даних користувача (на БД/кеші/файлах).
- Браузер тримає лише ключ сесії в cookie.
- Через `request.session` Django дістає/оновлює дані сесії.

#### Як це працює разом:

1. Після логіну Django створює сесію на сервері.
2. Session key записується в cookie (`sessionid`).
3. На кожен запит Django читає cookie і відновлює сесію.

#### Приклад роботи з сесією:

```python
def set_theme(request):
    request.session["theme"] = "dark"
    return HttpResponse("ok")

def get_theme(request):
    theme = request.session.get("theme", "light")
    return HttpResponse(theme)
```

#### Приклад явного cookie:

```python
response = HttpResponse("ok")
response.set_cookie("promo_seen", "1", max_age=3600)
```

#### Де зберігаються сесії в Django:

- За замовчуванням: у БД (`django.contrib.sessions`).
- Можна змінити backend на cache, file, signed cookies.

#### Важливі security-налаштування:

1. `SESSION_COOKIE_HTTPONLY = True` (захист від JS-доступу до cookie).
2. `SESSION_COOKIE_SECURE = True` (лише через HTTPS).
3. `SESSION_COOKIE_SAMESITE = "Lax"` або `"Strict"` за потреби.
4. Аналогічно для CSRF cookie (`CSRF_COOKIE_SECURE`, `CSRF_COOKIE_HTTPONLY`).

#### Практичні рекомендації:

1. Не зберігай чутливі дані напряму в cookie.
2. Мінімізуй обсяг даних у сесії.
3. Для масштабування використовуй Redis-backed сесії.
4. Очищай/інвалідовуй сесію при logout і критичних змінах акаунта.

#### Висновок:

Cookies - це клієнтський механізм передачі невеликих даних, а сесії - серверне
зберігання стану користувача. У Django вони працюють разом і є основою
класичного auth-flow та персоналізованого UX.

</details>

<summary>76. Що таке групи та дозволи і як реалізувати рольову модель доступу?</summary>

#### Django

У Django **permissions** (дозволи) і **groups** (групи) - це базові механізми
авторизації для реалізації RBAC (Role-Based Access Control).

#### Що таке дозволи (permissions):

- Це права типу `app_label.codename`, наприклад:
  `orders.view_order`, `orders.change_order`.
- Django автоматично створює базові дозволи для моделей:
  `add`, `change`, `delete`, `view`.
- Можна додавати власні (custom permissions) у `Meta`.

#### Що таке групи (groups):

- Група = роль, якій призначають набір permission.
- Користувача додають у групу, і він отримує права цієї ролі.

#### Приклад custom permission у моделі:

```python
class Order(models.Model):
    ...
    class Meta:
        permissions = [
            ("approve_order", "Can approve order"),
        ]
```

#### Перевірка доступу у коді:

```python
if request.user.has_perm("orders.approve_order"):
    ...
```

Декоратор:

```python
from django.contrib.auth.decorators import permission_required

@permission_required("orders.approve_order", raise_exception=True)
def approve_view(request, order_id):
    ...
```

#### Приклад ролей (RBAC):

1. **viewer** -> лише `view_*`
2. **manager** -> `view_*`, `change_*`, `approve_order`
3. **admin** -> повний набір доступів домену

#### Типовий процес реалізації RBAC:

1. Описати ролі та їх відповідальність.
2. Визначити mapping "роль -> permission-и".
3. Створити групи та призначити permission-и.
4. Додавати користувачів у групи (через admin або seed-скрипти).
5. Перевіряти permissions у view/CBV/API.

#### Практичні рекомендації:

1. Права задавай ролями (groups), а не вручну по користувачах.
2. Тримай policy доступу централізованою й документованою.
3. Перевіряй доступ на бекенді, навіть якщо UI приховує кнопки.
4. Для складних випадків додавай object-level permissions (за потреби).

#### Типові помилки:

1. Давати `is_superuser` там, де достатньо role-permissions.
2. Змішувати ролі та бізнес-стани без чітких правил.
3. Роздавати "тимчасові" права без аудиту й терміну дії.

#### Висновок:

Групи та дозволи в Django - надійна основа для рольової моделі доступу.
RBAC через groups робить систему безпечнішою, прозорішою і легшою в підтримці
при масштабуванні команди та продукту.

</details>

<summary>77. Як реалізувати відновлення пароля?</summary>

#### Django

Відновлення пароля в Django найпростіше реалізувати через вбудовані auth-views:
вони вже містять безпечний flow з одноразовим токеном і перевіркою терміну дії.

#### Стандартний flow:

1. Користувач вводить email на сторінці "Забули пароль".
2. Django надсилає лист із посиланням (uid + token).
3. Користувач переходить за посиланням і задає новий пароль.
4. Django підтверджує успішну зміну.

#### Підключення URL:

```python
from django.urls import path
from django.contrib.auth import views as auth_views

urlpatterns = [
    path("password-reset/", auth_views.PasswordResetView.as_view(), name="password_reset"),
    path("password-reset/done/", auth_views.PasswordResetDoneView.as_view(), name="password_reset_done"),
    path(
        "reset/<uidb64>/<token>/",
        auth_views.PasswordResetConfirmView.as_view(),
        name="password_reset_confirm",
    ),
    path("reset/done/", auth_views.PasswordResetCompleteView.as_view(), name="password_reset_complete"),
]
```

#### Потрібні шаблони:

- `registration/password_reset_form.html`
- `registration/password_reset_done.html`
- `registration/password_reset_confirm.html`
- `registration/password_reset_complete.html`
- email-шаблон, наприклад `registration/password_reset_email.html`

#### Базові налаштування email:

```python
EMAIL_BACKEND = "django.core.mail.backends.smtp.EmailBackend"
EMAIL_HOST = "smtp.example.com"
EMAIL_PORT = 587
EMAIL_USE_TLS = True
EMAIL_HOST_USER = "..."
EMAIL_HOST_PASSWORD = "..."
DEFAULT_FROM_EMAIL = "noreply@example.com"
```

#### Важливі безпекові принципи:

1. Не повідомляй, чи існує email у системі (захист від user enumeration).
2. Використовуй HTTPS для reset-посилань.
3. Обмежуй частоту reset-запитів (rate limiting).
4. Після зміни пароля інвалідовуй активні сесії за політикою безпеки.
5. Логуй події відновлення пароля для аудиту.

#### Коли потрібна кастомізація:

- Власний дизайн email/сторінок.
- Кастомна user-модель з login по email.
- Інтеграція з зовнішнім auth/SSO.

#### Висновок:

Django дає готовий безпечний механізм відновлення пароля через стандартні
auth-views. У production достатньо правильно налаштувати email, шаблони,
HTTPS і anti-abuse захист.

<summary>78. Що таке CSRF і як працює {% csrf_token %}?</summary>

#### Django

**CSRF (Cross-Site Request Forgery)** - це атака, коли зловмисний сайт змушує
авторизованого користувача виконати небажану дію у твоєму застосунку
(наприклад, змінити пароль або створити платіж).

#### Як Django захищає від CSRF:

1. На стороні сервера генерується CSRF-токен.
2. Токен передається у форму (через `{% csrf_token %}`).
3. Для state-changing запитів (`POST`, `PUT`, `PATCH`, `DELETE`) Django
   порівнює токен із запиту з очікуваним токеном.
4. Якщо токен відсутній/невалідний -> `403 Forbidden`.

#### `{% csrf_token %}` у формі:

```django
<form method="post">
  {% csrf_token %}
  {{ form.as_p }}
  <button type="submit">Зберегти</button>
</form>
```

#### Що потрібне для роботи:

1. `django.middleware.csrf.CsrfViewMiddleware` у `MIDDLEWARE`.
2. Використання `{% csrf_token %}` у всіх POST-формах.
3. Коректна cookie-політика (особливо в production з HTTPS).

#### Для AJAX/fetch запитів:

- CSRF-токен передають у заголовку `X-CSRFToken`.
- Токен беруть із cookie (`csrftoken`) або з HTML (залежно від підходу).

#### Типові помилки:

1. Забути `{% csrf_token %}` у шаблоні форми.
2. Відключити `CsrfViewMiddleware` без обґрунтованої причини.
3. Неправильно налаштувати домен/secure-параметри cookie.
4. Використовувати `@csrf_exempt` "для швидкості", а потім забути повернути захист.

#### Коли `csrf_exempt` допустимий:

- Лише для спеціальних endpoint (наприклад, зовнішні webhooks), і тільки якщо
  є інший надійний механізм валідації запиту (HMAC signature, token, allowlist).

#### Практичні рекомендації:

1. Усі browser-based state-changing форми мають містити `{% csrf_token %}`.
2. Для API без cookie-сесій (Bearer/JWT) будуй окрему auth-модель.
3. У production використовуй HTTPS + `CSRF_COOKIE_SECURE = True`.

#### Висновок:

`{% csrf_token %}` - ключовий елемент захисту Django від CSRF-атак у формах.
Це не опція, а базовий стандарт безпеки для будь-яких дій, що змінюють стан системи.

<summary>79. Як Django захищає від XSS, CSRF, SQL injection?</summary>

#### Django

Django має "secure by default" підхід і вбудовані механізми захисту від
найтиповіших веб-уразливостей: XSS, CSRF і SQL injection.

#### 1. Захист від XSS (Cross-Site Scripting)

Як працює:

1. Django templates автоматично екранують змінні в `{{ ... }}`.
2. HTML-спецсимволи перетворюються у безпечний вигляд.
3. Потенційно небезпечний контент не виконується як скрипт.

Важливо:

- `|safe` вимикає екранування, використовуй лише для повністю довіреного HTML.
- Не рендерь сирий user input без санітизації.

#### 2. Захист від CSRF

Як працює:

1. `CsrfViewMiddleware` перевіряє токен у state-changing запитах.
2. `{% csrf_token %}` додає токен у HTML-форми.
3. Без валідного токена Django повертає `403`.

#### 3. Захист від SQL injection

Як працює:

1. ORM використовує параметризовані запити.
2. Дані передаються окремо від SQL-структури.
3. Це блокує класичні інʼєкції в більшості стандартних сценаріїв.

Ризик виникає, якщо:

- писати raw SQL з конкатенацією рядків вручну.

#### Додаткові вбудовані захисти Django:

1. `SecurityMiddleware` (headers, SSL-related політики).
2. `XFrameOptionsMiddleware` (захист від clickjacking).
3. Безпечне хешування паролів (`PBKDF2` за замовчуванням).
4. Перевірки host header (`ALLOWED_HOSTS`).

#### Практичні production-рекомендації:

1. `DEBUG=False` у production.
2. HTTPS всюди (`SECURE_SSL_REDIRECT`, secure cookies).
3. Виставити `SESSION_COOKIE_SECURE`, `CSRF_COOKIE_SECURE`, `HTTPONLY`.
4. Використовувати CSP (через заголовки/пакети).
5. Регулярно оновлювати Django та залежності.

#### Типові помилки розробників:

1. Безконтрольне використання `|safe`.
2. `@csrf_exempt` без альтернативної валідації.
3. Raw SQL із f-string/format.
4. Залишений `DEBUG=True` у production.

#### Висновок:

Django надає потужний базовий захист від XSS, CSRF і SQL injection. Але безпека
залежить і від дисципліни коду: правильний template-рівень, ORM-підхід,
продакшн-налаштування і постійний security-review.

</details>

<summary>80. Як Django захищає від SQL-ін’єкцій?</summary>

#### Django

Django захищає від SQL-інʼєкцій насамперед через ORM і параметризовані запити:
дані користувача передаються як параметри, а не вбудовуються в SQL-рядок.

#### Основний механізм захисту:

1. Ти пишеш ORM-запит:
```python
User.objects.filter(email=user_input)
```
2. Django генерує SQL із placeholder-параметрами.
3. Значення `user_input` не "ламає" структуру SQL.

#### Чому це безпечно:

- ORM відділяє SQL-логіку від даних.
- СУБД обробляє параметри як значення, а не як частину SQL-команди.

#### Де ризик SQL injection усе ще можливий:

1. **Raw SQL із конкатенацією рядків**
```python
# Небезпечно
sql = f"SELECT * FROM users WHERE email = '{user_input}'"
```

2. Неправильне використання `.extra()` (legacy-підхід).
3. Динамічні ORDER BY/field name без allowlist-перевірки.

#### Безпечний raw SQL (якщо справді потрібен):

```python
from django.db import connection

with connection.cursor() as cursor:
    cursor.execute("SELECT * FROM users WHERE email = %s", [user_input])
    rows = cursor.fetchall()
```

#### Практичні правила:

1. За замовчуванням використовуй ORM (`filter/get/exclude/annotate`).
2. Якщо raw SQL неминучий - тільки параметризовані запити.
3. Ніколи не вставляй user input у SQL через f-string або `format`.
4. Для динамічних полів/сортування застосовуй allowlist:
- дозволені тільки заздалегідь визначені імена колонок.

#### Додаткові кроки hardening:

1. Мінімальні права DB-користувача (least privilege).
2. Логи підозрілих запитів і моніторинг аномалій.
3. Регулярні оновлення Django/драйверів БД.

#### Висновок:

Django ORM добре захищає від SQL-інʼєкцій "з коробки". Реальні ризики
починаються там, де розробник обходить ORM і будує SQL вручну без
параметризації та валідації.

<summary>81. Що таке Content Security Policy (CSP) у Django?</summary>

#### Django

**Content Security Policy (CSP)** - це HTTP-політика безпеки, яка обмежує, з
яких джерел браузер може завантажувати скрипти, стилі, зображення, шрифти тощо.
Основна мета - зменшити ризик XSS і виконання небезпечного контенту.

#### Що робить CSP:

1. Визначає allowlist джерел (`self`, CDN, API-домени).
2. Блокує невідомі або інʼєкційні ресурси.
3. Дозволяє поступово впроваджувати політику через `Report-Only`.

#### Приклад CSP-заголовка:

```http
Content-Security-Policy:
default-src 'self';
script-src 'self';
style-src 'self' https://fonts.googleapis.com;
font-src 'self' https://fonts.gstatic.com;
img-src 'self' data:;
connect-src 'self';
frame-ancestors 'none';
base-uri 'self';
```

#### Як додати CSP у Django:

Найзручніше через пакет `django-csp`:

```bash
pip install django-csp
```

```python
# settings.py
INSTALLED_APPS += ["csp"]
MIDDLEWARE += ["csp.middleware.CSPMiddleware"]

CSP_DEFAULT_SRC = ("'self'",)
CSP_SCRIPT_SRC = ("'self'",)
CSP_STYLE_SRC = ("'self'", "https://fonts.googleapis.com")
CSP_FONT_SRC = ("'self'", "https://fonts.gstatic.com")
CSP_IMG_SRC = ("'self'", "data:")
CSP_CONNECT_SRC = ("'self'",)
CSP_FRAME_ANCESTORS = ("'none'",)
```

#### Важливі практики:

1. Починай з `Content-Security-Policy-Report-Only`, щоб зібрати порушення.
2. Мінімізуй використання `'unsafe-inline'` і `'unsafe-eval'`.
3. Для inline-скриптів використовуй nonce/hash-підхід.
4. Поступово звужуй політику до найнеобхідніших джерел.

#### Типові проблеми при впровадженні:

1. Ламаються сторонні віджети/аналітика через надто жорсткі правила.
2. Непродумані wildcard-джерела (`*`) знецінюють CSP.
3. Відсутність моніторингу CSP-порушень після релізу.

#### Висновок:

CSP у Django - важливий додатковий шар захисту від XSS і supply-chain ризиків.
Найкращий підхід: спочатку `Report-Only`, далі поступове посилення політики до
строгої allowlist-моделі.

<summary>82. Що таке custom management commands і коли їх варто використовувати?</summary>

#### Django

`Custom management commands` - це власні CLI-команди Django, які запускаються
через `python manage.py <command_name>`. Вони дозволяють оформити операційні
та технічні задачі як підтримуваний код проєкту.

#### Навіщо це потрібно:

1. Автоматизація рутинних задач (імпорт/експорт, очистка, синхронізація).
2. Операційні команди для DevOps/Support.
3. Скрипти для міграції/бекфілу даних.
4. Команди для CI/CD, staging і maintenance-вікон.

#### Структура файлів:

```text
my_app/
  management/
    __init__.py
    commands/
      __init__.py
      recalc_stats.py
```

#### Приклад команди:

```python
from django.core.management.base import BaseCommand
from orders.models import Order

class Command(BaseCommand):
    help = "Перераховує агреговану статистику замовлень"

    def add_arguments(self, parser):
        parser.add_argument("--dry-run", action="store_true")

    def handle(self, *args, **options):
        total = Order.objects.count()
        if options["dry_run"]:
            self.stdout.write(self.style.WARNING(f"DRY RUN: orders={total}"))
            return
        # реальна логіка
        self.stdout.write(self.style.SUCCESS(f"Done. orders={total}"))
```

Запуск:

```bash
python manage.py recalc_stats --dry-run
```

#### Коли custom command - правильний вибір:

1. Задача має повторюватися та бути відтворюваною.
2. Потрібен доступ до повного Django-контексту (ORM, settings, services).
3. Потрібна контрольована команда для команди/CI.

#### Практичні рекомендації:

1. Додавай `--dry-run` для безпечної перевірки.
2. Роби ідемпотентні операції, якщо можливо.
3. Використовуй транзакції для критичних пакетних змін.
4. Логуй прогрес і підсумки (`self.stdout`, structured logs).
5. Для довгих задач розглядай запуск через черги/воркери.

#### Типові помилки:

1. Тримати важливі "one-off" скрипти поза репозиторієм.
2. Запускати ризикові update/delete без dry-run і backup-плану.
3. Писати команду як нерозбірливий monolith без тестів.

#### Висновок:

Custom management commands - стандартний інструмент операційної зрілості Django.
Вони роблять технічні процедури прозорими, контрольованими й придатними для
повторного запуску в командному середовищі.

<summary>83. Що таке контекст-процесори (context processors) і як їх використовують?</summary>

#### Django

`Context processors` - це функції, які автоматично додають дані в контекст
кожного шаблону (або певних шаблонів), щоб не передавати ці змінні вручну в
кожному view.

#### Як працює context processor:

1. Django викликає функцію під час рендеру шаблону.
2. Функція повертає словник.
3. Цей словник додається до template context.

#### Приклад:

```python
# core/context_processors.py
def app_meta(request):
    return {
        "APP_NAME": "My Django App",
        "SUPPORT_EMAIL": "support@example.com",
    }
```

Підключення в `settings.py`:

```python
TEMPLATES = [
    {
        "BACKEND": "django.template.backends.django.DjangoTemplates",
        "DIRS": [],
        "APP_DIRS": True,
        "OPTIONS": {
            "context_processors": [
                "django.template.context_processors.request",
                "django.contrib.auth.context_processors.auth",
                "django.contrib.messages.context_processors.messages",
                "core.context_processors.app_meta",
            ],
        },
    },
]
```

Після цього в будь-якому шаблоні:

```django
<footer>{{ APP_NAME }} · {{ SUPPORT_EMAIL }}</footer>
```

#### Коли використовувати:

1. Глобальні дані для layout (назва застосунку, контакти, feature flags).
2. Дані, потрібні в багатьох шаблонах.
3. Легкі й швидкі значення без дорогих обчислень.

#### Коли не варто:

1. Для важких SQL-запитів на кожен рендер.
2. Для вузькоспецифічних даних одного-двох view.
3. Для складної бізнес-логіки.

#### Вбудовані корисні context processors:

1. `django.template.context_processors.request` (`request` у шаблоні)
2. `django.contrib.auth.context_processors.auth` (`user`, `perms`)
3. `django.contrib.messages.context_processors.messages`
4. `django.template.context_processors.static`
5. `django.template.context_processors.media`

#### Практичні рекомендації:

1. Тримай context processors максимально легкими.
2. Уникай дублювання ключів, які можуть конфліктувати з локальним context.
3. Якщо дані дорогі - кешуй їх або передавай точково з view.
4. Документуй глобальні ключі контексту для команди.

#### Висновок:

Context processors - зручний механізм для загальних template-даних. Вони добре
працюють для легких глобальних значень, але не повинні ставати місцем для
важкої доменної логіки.

<summary>84. Як працює кешування в Django? Основні підходи до кешування.</summary>

#### Django

Кешування в Django зменшує кількість дорогих операцій (SQL, рендер шаблонів,
обчислення) і прискорює відповіді для користувача.

#### Як це працює:

1. Дані/відповідь зберігаються у швидкому сховищі (Redis/Memcached).
2. Повторний запит отримує результат із кешу, без повного перерахунку.
3. Після TTL або інвалідації значення обчислюється заново.

#### Основні підходи кешування в Django:

1. **Site-wide cache (middleware)**
- Кешує майже весь HTTP-відгук для сторінок.
- Добре для публічного контенту з невеликою персоналізацією.

2. **Per-view cache (`cache_page`)**
- Кешує конкретний view на заданий час.

```python
from django.views.decorators.cache import cache_page

@cache_page(60 * 5)
def article_list(request):
    ...
```

3. **Template fragment cache**
- Кешує частину шаблону, а не всю сторінку.

```django
{% load cache %}
{% cache 300 homepage_sidebar user.id %}
  ...
{% endcache %}
```

4. **Low-level cache API**
- Ручне кешування результатів у коді.

```python
from django.core.cache import cache

data = cache.get("popular_products")
if data is None:
    data = expensive_query()
    cache.set("popular_products", data, 300)
```

#### Налаштування backend:

```python
CACHES = {
    "default": {
        "BACKEND": "django.core.cache.backends.redis.RedisCache",
        "LOCATION": "redis://127.0.0.1:6379/1",
    }
}
```

#### Коли який підхід обирати:

1. Публічні сторінки без персоналізації -> per-view/site-wide.
2. Частково динамічні сторінки -> fragment cache.
3. Дорогі доменні обчислення/агрегації -> low-level cache API.

#### Практичні правила:

1. Починай з точкового кешу (fragment/per-view), а не одразу site-wide.
2. Визначай TTL з урахуванням вимог до актуальності даних.
3. Для персоналізованих даних додавай ключі (user id, locale, role).
4. Плануй інвалідацію кешу після змін даних.

#### Типові помилки:

1. Кешувати персоналізований контент без user-ключа.
2. Не інвалідовувати кеш після update/delete.
3. Занадто довгий TTL для "живих" даних.
4. Покладатися на кеш замість оптимізації поганих SQL-запитів.

#### Висновок:

Кешування в Django - це багаторівнева стратегія: від фрагментів шаблону до
повних відповідей. Найкращий результат дає комбінація правильного backend
(зазвичай Redis), помірного TTL і контрольованої інвалідації.

</details>

<summary>85. Як оптимізувати запити, кешування та індексацію БД?</summary>

#### Django

Оптимізація в Django завжди комплексна:  
`ORM-запити` + `кешування` + `правильні індекси БД` + `профілювання`.
Окремо взятий прийом рідко дає стабільний результат.

#### 1. Оптимізація ORM-запитів

1. Уникай N+1 через:
- `select_related()` для FK/O2O
- `prefetch_related()` для M2M/reverse FK

2. Витягуй лише потрібні поля:
- `values()`, `values_list()`, `only()`, `defer()`

3. Для масових оновлень використовуй:
- `update()` + `F()` замість `save()` у циклі

4. Для перевірки наявності:
- `exists()` замість завантаження всього queryset

#### 2. Кешування

1. Починай із fragment/per-view кешу.
2. Для "гарячих" обчислень - low-level cache API.
3. Для продакшену зазвичай Redis backend.
4. Продумай ключі кешу (user/locale/role) та інвалідацію після змін даних.

#### 3. Індексація БД

1. Додавай індекси на поля, які часто в:
- `WHERE`
- `ORDER BY`
- `JOIN`

2. Використовуй `models.Index` і `UniqueConstraint` у `Meta`.
3. Не індексуй "все підряд": кожен індекс має вартість на `INSERT/UPDATE`.

Приклад:

```python
class Order(models.Model):
    user = models.ForeignKey("auth.User", on_delete=models.CASCADE)
    status = models.CharField(max_length=20)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        indexes = [
            models.Index(fields=["status", "-created_at"]),
            models.Index(fields=["user", "-created_at"]),
        ]
```

#### 4. Профілювання і вимірювання

1. Використовуй Django Debug Toolbar на dev/staging.
2. Логуй повільні SQL-запити на рівні БД.
3. Перевіряй `EXPLAIN ANALYZE` для критичних запитів.
4. Міряй p95/p99 latency до/після змін.

#### Типовий порядок оптимізації:

1. Знайти повільне місце (не оптимізувати "наосліп").
2. Виправити ORM (N+1, зайві поля, дубльовані запити).
3. Додати/перевірити індекси.
4. Додати кеш для стабільно гарячих сценаріїв.
5. Повторно виміряти результат.

#### Типові помилки:

1. Кешувати все підряд без стратегії інвалідації.
2. Додавати індекси без аналізу реальних запитів.
3. Оптимізувати Python-код, ігноруючи SQL-вузьке місце.
4. Робити великі schema-зміни в production без поетапного rollout.

#### Висновок:

Стабільна продуктивність Django досягається не "однією кнопкою", а системним
підходом: вимірювати, оптимізувати ORM, правильно індексувати БД і кешувати
лише те, що дає реальну вигоду під навантаженням.

</details>

<summary>86. Що таке Django testing framework?</summary>

#### Django

`Django testing framework` - це вбудований набір інструментів для тестування
моделей, view, форм, API та інтеграцій у межах Django-проєкту.

#### Що входить у framework:

1. `django.test.TestCase` - базовий клас для більшості тестів.
2. `Client` - тестовий HTTP-клієнт для запитів до view/URL.
3. Тестова база даних (створюється автоматично перед запуском).
4. Assertions для response, контексту, редіректів, шаблонів.
5. Інструменти override settings та ізоляції середовища.

#### Як запускаються тести:

```bash
python manage.py test
```

#### Як працює тестова БД:

1. Django створює окрему test-DB.
2. Накатує міграції/схему.
3. Кожен тест ізолюється транзакційно (`TestCase`).
4. Після завершення test-DB видаляється.

#### Основні класи тестів:

1. **`TestCase`**
- Найчастіший вибір; швидко й ізоляційно для ORM/view тестів.

2. **`TransactionTestCase`**
- Коли потрібен контроль транзакцій/commit behavior.

3. **`SimpleTestCase`**
- Тести без доступу до БД.

4. **`LiveServerTestCase`**
- Інтеграційні/browser тести з піднятим тестовим сервером.

#### Приклад простого тесту:

```python
from django.test import TestCase
from django.urls import reverse

class HealthViewTests(TestCase):
    def test_health_endpoint_returns_200(self):
        response = self.client.get(reverse("health"))
        self.assertEqual(response.status_code, 200)
```

#### Практичні рекомендації:

1. Пиши тести для критичних бізнес-сценаріїв, а не лише "щоб був coverage".
2. Роби тести детермінованими (без залежності від часу/мережі без mock).
3. Винось фабрики/fixtures для читабельної підготовки даних.
4. Запускай тести в CI на кожному PR.

#### Типові помилки:

1. Тести з залежністю від порядку виконання.
2. Надмірно інтеграційні тести там, де достатньо unit рівня.
3. Ігнорування edge cases і негативних сценаріїв.

#### Висновок:

Django testing framework дає готову інфраструктуру для надійного тестування
бекенду "з коробки". Це основа безпечного рефакторингу і стабільного релізного
процесу в командній розробці.

<summary>87. Чим відрізняються unit та integration тести?</summary>

#### Django

`Unit` і `integration` тести відрізняються рівнем перевірки:

- **Unit test** перевіряє ізольований шматок логіки.
- **Integration test** перевіряє взаємодію кількох компонентів разом.

#### Unit тести:

1. Тестують одну функцію/метод/клас у відриві від решти системи.
2. Часто використовують mock/stub для залежностей.
3. Дуже швидкі, точні у локалізації помилки.

Приклад у Django:

- тест валідатора,
- тест helper-функції,
- тест сервісного методу без реального HTTP/БД (або з мінімальною ізоляцією).

#### Integration тести:

1. Перевіряють роботу кількох шарів разом (view + ORM + middleware + templates).
2. Часто використовують test client і тестову БД.
3. Повільніші, але ближчі до реального поведінкового сценарію.

Приклад у Django:

- `client.post(...)` -> view -> model save -> redirect -> перевірка БД.

#### Ключові відмінності:

1. **Швидкість**
- Unit: швидкі.
- Integration: повільніші.

2. **Обсяг покриття**
- Unit: вузький фокус.
- Integration: ширший системний сценарій.

3. **Діагностика**
- Unit: простіше знайти конкретну причину.
- Integration: краще ловлять проблеми звʼязки компонентів.

#### Приклад тест-піраміди для Django:

1. Багато unit тестів (валідатори, сервіси, утиліти).
2. Менше integration тестів (ключові endpoint, критичні бізнес-флоу).
3. Ще менше end-to-end/browser тестів.

#### Практичні рекомендації:

1. Критичну доменну логіку тримай добре покритою unit тестами.
2. Для core user journeys обовʼязково додавай integration тести.
3. Не намагайся покрити все лише інтеграційними тестами - це повільно і крихко.
4. Кожен баг у production має завершуватись новим тестом відповідного рівня.

#### Висновок:

Unit і integration тести не замінюють один одного. Стабільний Django-проєкт
потребує комбінації: unit для швидкого фідбеку і integration для перевірки
реальної взаємодії компонентів.

</details>

<summary>88. Що таке Django test client?</summary>

#### Django

`Django test client` - це вбудований інструмент для симуляції HTTP-запитів до
твого застосунку без запуску реального браузера чи зовнішнього HTTP-сервера.

#### Що вміє test client:

1. Робити `GET`, `POST`, `PUT`, `PATCH`, `DELETE` запити.
2. Перевіряти status code, redirects, templates, context.
3. Працювати з сесією, cookies і авторизацією користувача.
4. Тестувати URL -> view -> response ланцюжок.

#### Базовий приклад:

```python
from django.test import TestCase
from django.urls import reverse

class HomeTests(TestCase):
    def test_home_returns_200(self):
        response = self.client.get(reverse("home"))
        self.assertEqual(response.status_code, 200)
```

#### Приклад POST-запиту:

```python
response = self.client.post(reverse("signup"), {
    "username": "testuser",
    "password1": "StrongPass123!",
    "password2": "StrongPass123!",
})
self.assertEqual(response.status_code, 302)
```

#### Логін у тестах:

1. Швидкий технічний логін:

```python
self.client.force_login(user)
```

2. Реалістичний логін через credentials:

```python
logged_in = self.client.login(username="testuser", password="secret")
self.assertTrue(logged_in)
```

#### Що можна перевіряти:

1. `response.status_code`
2. `response.context`
3. `response.templates`
4. `assertRedirects(...)`
5. вміст відповіді (`assertContains(...)`)

#### Обмеження test client:

1. Не виконує JavaScript (це не browser automation).
2. Не перевіряє реальний frontend runtime як Playwright/Selenium.
3. Не моделює всі нюанси мережевого оточення production.

#### Практичні рекомендації:

1. Використовуй test client для більшості web/API інтеграційних тестів Django.
2. Для JS-flow додавай окремі e2e/browser тести.
3. Комбінуй із фабриками/fixtures для читабельної підготовки даних.

#### Висновок:

Django test client - швидкий і потужний інструмент для перевірки HTTP-поведінки
застосунку на backend-рівні. Це стандартний вибір для тестування view, URL,
авторизації та базової інтеграції компонентів.

</details>

<summary>89. Як тестувати форми, views і ORM?</summary>

#### Django

У Django форми, views і ORM тестують на різних рівнях, але в єдиному стилі:
чіткі preconditions -> дія -> перевірка результату.

#### 1. Тестування форм

Що перевіряти:

1. валідні/невалідні дані
2. помилки по конкретних полях
3. кастомну валідацію (`clean_<field>`, `clean()`)

Приклад:

```python
from django.test import TestCase
from users.forms import SignUpForm

class SignUpFormTests(TestCase):
    def test_form_valid(self):
        form = SignUpForm(data={
            "username": "newuser",
            "email": "user@example.com",
            "password1": "StrongPass123!",
            "password2": "StrongPass123!",
        })
        self.assertTrue(form.is_valid())

    def test_form_invalid_password_mismatch(self):
        form = SignUpForm(data={
            "username": "newuser",
            "email": "user@example.com",
            "password1": "StrongPass123!",
            "password2": "WrongPass123!",
        })
        self.assertFalse(form.is_valid())
```

#### 2. Тестування views

Що перевіряти:

1. статус-код
2. шаблон/контекст
3. редіректи
4. auth/permission поведінку

Приклад:

```python
from django.test import TestCase
from django.urls import reverse

class DashboardViewTests(TestCase):
    def test_anonymous_redirected_to_login(self):
        response = self.client.get(reverse("dashboard"))
        self.assertEqual(response.status_code, 302)
```

#### 3. Тестування ORM

Що перевіряти:

1. створення/оновлення/видалення моделей
2. кастомні manager/queryset методи
3. звʼязки і обмеження (`unique`, `constraints`)

Приклад:

```python
from django.test import TestCase
from blog.models import Article

class ArticleORMTests(TestCase):
    def test_create_article(self):
        article = Article.objects.create(title="Test", is_published=True)
        self.assertEqual(Article.objects.count(), 1)
        self.assertTrue(article.is_published)
```

#### Практичні рекомендації:

1. Тримай тести незалежними один від одного.
2. Давай тестам описові назви за правилом `should_<behavior>`.
3. Для фікстурних даних використовуй factories/fixtures, а не дублювання setup.
4. Тестуй негативні сценарії так само ретельно, як позитивні.

#### Що часто забувають:

1. Перевірки permissions на view-рівні.
2. Валідацію форм із неочікуваними вхідними даними.
3. Поведінку manager/queryset методів на порожніх вибірках.

#### Висновок:

Форми тестують валідацію, views - HTTP-поведінку, ORM - дані і звʼязки.
Разом це дає надійний тестовий фундамент, який дозволяє безпечно рефакторити
Django-код і стабільно релізити зміни.

<summary>90. Як використовувати fixtures?</summary>

#### Django

`Fixtures` у Django - це заздалегідь підготовлені дані (JSON/YAML/XML), які
можна завантажувати в БД для тестів, демо або початкового seed.

#### Навіщо використовують fixtures:

1. Швидко підняти стандартний набір даних.
2. Відтворювати однаковий тестовий стан у різних середовищах.
3. Тестувати сценарії з "реалістичними" взаємоповʼязаними записами.

#### Приклад файлу fixture:

```json
[
  {
    "model": "blog.article",
    "pk": 1,
    "fields": {
      "title": "Hello Django",
      "is_published": true
    }
  }
]
```

Розміщення:

- `app/fixtures/articles.json` (типовий варіант)

#### Завантаження fixture:

```bash
python manage.py loaddata articles.json
```

#### Використання у тестах:

```python
from django.test import TestCase
from blog.models import Article

class ArticleTests(TestCase):
    fixtures = ["articles.json"]

    def test_fixture_loaded(self):
        self.assertEqual(Article.objects.count(), 1)
```

#### Коли fixtures доречні:

1. Невеликий стабільний reference-набір даних.
2. Інтеграційні тести зі складними звʼязками.
3. Демо/локальне наповнення для review середовищ.

#### Обмеження fixtures:

1. Крихкі при зміні схеми моделей.
2. Важче читати й підтримувати великі JSON-файли.
3. Складніше гнучко комбінувати дані під різні тести.

#### Практична альтернатива:

- Для більшості тестів краще factories (factory_boy, model_bakery), бо:
1. Дані створюються програмно.
2. Легше адаптувати під зміни моделі.
3. Менше "магічних" статичних файлів.

#### Рекомендований підхід техліда:

1. Невеликі shared-дані -> fixtures.
2. Більшість unit/integration тестів -> factories.
3. Не змішуй хаотично обидва підходи без чітких правил у команді.

#### Висновок:

Fixtures у Django корисні для стабільного стартового набору даних, але для
гнучких і довготривалих тестових наборів зазвичай ефективніші factory-підходи.

</details>

<summary>91. Як працюють async views і async ORM в Django?</summary>

#### Django

Django підтримує асинхронні view (`async def`) і поступово розвиває async-стек
для I/O-навантажених сценаріїв. Головна цінність async - краща масштабованість
при великій кількості одночасних очікувань (зовнішні API, мережеві операції).

#### Async view у Django:

```python
from django.http import JsonResponse

async def async_health(request):
    return JsonResponse({"status": "ok"})
```

#### Коли async справді корисний:

1. Багато I/O-очікувань (HTTP-запити до сторонніх сервісів).
2. Long-polling/websocket-патерни (разом із відповідним стеком).
3. Висока конкуренція запитів при відносно легкій CPU-логіці.

#### Sync vs Async у Django:

1. `def view(...)` -> синхронний view.
2. `async def view(...)` -> асинхронний view.
3. Django може адаптувати sync/async межі, але зайві переходи мають overhead.

#### Робота з БД (ORM):

- Історично ORM був sync-first.
- У сучасних версіях Django є async-варіанти частини операцій (`aget`, `acreate`,
  `aupdate_or_create`, async iteration тощо), але потрібно враховувати
  сумісність конкретної версії та драйвера БД.

Приклад:

```python
user = await User.objects.aget(pk=user_id)
```

#### Важливі обмеження:

1. Не всі сторонні бібліотеки мають async-safe API.
2. CPU-bound задачі async не прискорює (для цього воркери/черги/процеси).
3. Змішування sync ORM у async-коді може блокувати event loop.

#### Практичні рекомендації:

1. Використовуй async там, де є реальний I/O bottleneck.
2. Якщо стек переважно sync - не форсуй async "для моди".
3. Мінімізуй sync<->async переходи.
4. Для фонового CPU/довгого processing використовуй Celery/RQ, а не await у view.

#### Типові помилки:

1. Очікувати приріст продуктивності без зміни I/O-профілю застосунку.
2. Робити блокуючі sync-виклики всередині async view.
3. Не тестувати поведінку під конкурентним навантаженням.

#### Висновок:

Async у Django - потужний інструмент для I/O-heavy сценаріїв, але не універсальна
панацея. Ефективність залежить від узгодженості всього стека: view, ORM,
зовнішніх клієнтів і реального профілю навантаження.

</details>

<summary>92. Як створювати бекенд-задачі (@task, Celery)?</summary>

#### Django

Бекенд-задачі (background jobs) у Django використовують для важких або відкладених
операцій, які не варто виконувати прямо в HTTP-запиті: email-розсилки, імпорт,
генерація файлів, інтеграції, ретраї.

#### Найпоширеніший стек:

- **Celery** + broker (зазвичай Redis або RabbitMQ) + workers

#### Базова архітектура:

1. Django-код ставить задачу в чергу.
2. Broker зберігає повідомлення.
3. Celery worker забирає задачу і виконує її асинхронно.
4. (Опційно) результат зберігається в result backend.

#### Приклад задачі:

```python
# app/tasks.py
from celery import shared_task

@shared_task
def send_welcome_email(user_id):
    # логіка відправки email
    return {"user_id": user_id, "status": "sent"}
```

Виклик із view/сервісу:

```python
send_welcome_email.delay(user.id)
```

#### Мінімальна інтеграція Celery з Django:

1. Встановити:
```bash
pip install celery redis
```
2. Додати `celery.py` у конфіг проєкту.
3. Вказати broker URL у settings/env.
4. Запустити worker:
```bash
celery -A config worker -l info
```

#### Періодичні задачі:

- Через `celery beat` або `django-celery-beat` (cron-like scheduler).

#### Практичні рекомендації:

1. Задачі мають бути **ідемпотентними** (повторний запуск не ламає дані).
2. Додавай `retry` для тимчасових збоїв (мережа, сторонні API).
3. Не передавай у задачу "важкі" обʼєкти - передавай `id`, а дані діставай у worker.
4. Логуй статус виконання і помилки.
5. Визначай таймаути і обмеження конкурентності воркерів.

#### Типові помилки:

1. Виконувати довгі операції прямо у view замість черги.
2. Відсутність retry/policy при інтеграціях із зовнішніми сервісами.
3. Неочікувані дублікати задач через неідемпотентну логіку.
4. Відсутність моніторингу черг і "завислих" задач.

#### Висновок:

Celery в Django - стандарт для асинхронної обробки бекенд-задач. Ключ до
надійності: ідемпотентність, ретраї, моніторинг і чітке розділення HTTP-flow
та фонової обробки.

</details>

<summary>93. Що таке Django REST Framework?</summary>

#### Django

**Django REST Framework (DRF)** - це популярний фреймворк поверх Django для
побудови REST API: серіалізація даних, валідація, авторизація, маршрутизація,
пагінація та зручний API-browsable інтерфейс.

#### Навіщо використовують DRF:

1. Швидко будувати стабільні API для web/mobile клієнтів.
2. Стандартизувати валідацію і формат відповідей.
3. Гнучко керувати auth/permissions/throttling.
4. Зменшити кількість boilerplate-коду порівняно з "чистими" JsonResponse-view.

#### Ключові компоненти DRF:

1. **Serializer**
- Перетворює модель/обʼєкт <-> JSON.
- Виконує валідацію вхідних даних.

2. **APIView / GenericAPIView / ViewSet**
- Шари для побудови endpoint-логіки.

3. **Routers**
- Автоматична генерація URL для ViewSet.

4. **Authentication / Permissions**
- Контроль хто ти і що тобі дозволено.

5. **Pagination / Filtering / Ordering**
- Готові механізми для списків і пошуку.

#### Приклад serializer:

```python
from rest_framework import serializers
from .models import Article

class ArticleSerializer(serializers.ModelSerializer):
    class Meta:
        model = Article
        fields = ["id", "title", "is_published", "created_at"]
```

#### Приклад ViewSet:

```python
from rest_framework.viewsets import ModelViewSet
from .models import Article
from .serializers import ArticleSerializer

class ArticleViewSet(ModelViewSet):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer
```

#### Приклад router:

```python
from rest_framework.routers import DefaultRouter
from .views import ArticleViewSet

router = DefaultRouter()
router.register("articles", ArticleViewSet, basename="article")
urlpatterns = router.urls
```

#### Коли DRF особливо доречний:

1. Потрібен API-first або multi-client продукт.
2. Є складні permission/pagination/filtering вимоги.
3. Потрібна масштабована архітектура API і стандартизований error format.

#### Висновок:

DRF - де-факто стандарт для API в Django-екосистемі. Він додає структуровану
API-архітектуру, прискорює розробку і спрощує підтримку production endpoint-ів.

</details>

<summary>94. Що таке серіалізація і навіщо вона потрібна?</summary>

#### Django

Серіалізація - це перетворення обʼєктів Python/моделей у формат передачі даних
(зазвичай JSON), а десеріалізація - зворотне перетворення в обʼєкти з валідацією.

#### Навіщо потрібна серіалізація:

1. Передавати дані між backend і frontend/mobile клієнтами.
2. Стандартизувати формат API-відповідей.
3. Валідовувати вхідні дані перед записом у БД.
4. Відокремити модель БД від зовнішнього контракту API.

#### Серіалізація в DRF:

- Виконується через `Serializer` або `ModelSerializer`.

Приклад:

```python
from rest_framework import serializers
from .models import Article

class ArticleSerializer(serializers.ModelSerializer):
    class Meta:
        model = Article
        fields = ["id", "title", "is_published", "created_at"]
```

#### Перетворення model -> JSON:

```python
article = Article.objects.first()
data = ArticleSerializer(article).data
```

#### Перетворення JSON -> validated data/model:

```python
payload = {"title": "New", "is_published": True}
serializer = ArticleSerializer(data=payload)
serializer.is_valid(raise_exception=True)
obj = serializer.save()
```

#### Що ще дає serializer:

1. Перевірка типів і обовʼязкових полів.
2. Кастомна валідація (`validate_<field>`, `validate`).
3. Контроль read-only/write-only полів.
4. Вкладені серіалізатори для повʼязаних сутностей.

#### Практичні рекомендації:

1. Не віддавай модель "як є" без явного serializer-контракту.
2. Тримай API-схему стабільною і backward-compatible.
3. Кастомні бізнес-правила валідуй у serializer або сервісному шарі.
4. Для різних use-cases створюй окремі serializers (list/detail/create/update).

#### Висновок:

Серіалізація - ядро API-шару в Django/DRF: вона відповідає за формат даних,
валідацію і стабільність контракту між сервером і клієнтами.

</details>

<summary>95. Які особливості Django з REST-фреймворком (DRF)?</summary>

#### Django

Django + DRF - це комбінація зрілого вебфреймворку і потужного API-шару.
Вона добре підходить для production REST-сервісів із чіткою доменною моделлю.

#### Ключові особливості DRF у Django-екосистемі:

1. **Serializer-first підхід**
- Чіткий контракт API, валідація вхідних даних, контроль полів.

2. **Гнучка побудова endpoint-ів**
- `APIView`, `GenericAPIView`, `ViewSet`, `ModelViewSet`.

3. **Автоматична маршрутизація**
- Routers значно зменшують boilerplate для CRUD API.

4. **Auth + permissions**
- Session, Token, JWT (через пакети), custom permission класи.

5. **Pagination, filtering, ordering**
- Готові механізми для списків у великих системах.

6. **Throttling**
- Обмеження частоти запитів (anti-abuse / API hygiene).

7. **Content negotiation**
- Підтримка різних renderer/parser (JSON за замовчуванням).

8. **Browsable API**
- Зручний dev-інтерфейс для ручної перевірки endpoint-ів.

#### Типовий production-стек з DRF:

1. Versioned API (`/api/v1/...`).
2. Serializer-и під різні сценарії (`list/detail/create/update`).
3. Permission policy per endpoint.
4. Стандартизований формат помилок.
5. Пагінація + фільтрація + індексація БД.

#### Сильні сторони підходу:

1. Висока швидкість розробки без втрати структури.
2. Сумісність із Django auth/admin/ORM/migrations.
3. Легко розширювати кастомними класами під бізнес-вимоги.

#### На що звертати увагу:

1. Не перевантажувати `ModelViewSet` складною доменною логікою.
2. Контролювати N+1 запити в serializer/viewset.
3. Чітко розділяти API-контракт і внутрішню модель даних.

#### Висновок:

DRF робить Django сильним API-фреймворком enterprise-рівня: швидкий старт,
структурована архітектура, зріла безпека і масштабованість для реальних
мультиклієнтних продуктів.

<summary>96. Яке місце займає Django в архітектурі сервісів та REST API?</summary>

#### Django

Django у сучасній архітектурі найчастіше займає роль **основного бізнес-бекенду**:
керує доменною логікою, транзакційними даними, адмін-процесами та REST API
для web/mobile клієнтів.

#### Типові ролі Django в архітектурі:

1. **Модульний моноліт (найчастіше)**
- Один сервіс із чіткою декомпозицією на apps/домени.
- Швидкий розвиток продукту і простіший операційний контур.

2. **API-сервіс (DRF)**
- Чистий backend для SPA/mobile/партнерських інтеграцій.

3. **Core-сервіс у мікросервісній екосистемі**
- Django як "source of truth" для критичних доменів (користувачі, білінг, замовлення).

4. **Backoffice/Admin-сервіс**
- Внутрішній контур операцій і керування даними через Django Admin + custom UI.

#### Чому Django добре підходить на цю роль:

1. Швидка розробка повного backend-циклу.
2. Сильний ORM + транзакційна модель.
3. Зріла auth/permissions система.
4. DRF для стандартизованого API.
5. Надійна адмінка для операційних команд.

#### Де Django може бути не ідеальним "першим вибором":

1. Ультра-легкі high-concurrency API-only сервіси (іноді краще FastAPI).
2. Специфічні low-latency сценарії з мінімальним framework overhead.

#### Практичний архітектурний підхід:

1. Починати з Django-моноліту з чіткими доменними межами.
2. Виносити окремі сервіси лише при реальній потребі масштабування.
3. Залишати Django як core domain/API/admin вузол.

#### Інтеграція з іншими сервісами:

- Celery/queues для async processing
- Redis для кешів
- Search engine для full-text
- Object storage/CDN для медіа
- Observability stack (logs, metrics, tracing)

#### Висновок:

Django займає сильну позицію як backbone бізнес-сервісів і REST API. У більшості
продуктів це практичний компроміс між швидкістю розробки, архітектурною
дисципліною і довгостроковою підтримуваністю.

<summary>97. Як задеплоїти Django-проєкт (WSGI, ASGI, Nginx)?</summary>

#### Django

Деплой Django у production зазвичай будується так:

1. **Nginx** - reverse proxy, static/media, TLS.
2. **App server** - Gunicorn (WSGI) або Uvicorn/Daphne (ASGI).
3. **Django app** - бізнес-логіка.
4. **PostgreSQL/Redis** - дані та кеш/черги.

#### WSGI vs ASGI:

1. **WSGI (Gunicorn)**
- Класичний стабільний варіант для sync Django-застосунків.

2. **ASGI (Uvicorn/Daphne)**
- Потрібний для async views, websocket та modern async сценаріїв.

#### Мінімальний production-flow:

1. Підготувати середовище:
- `DEBUG=False`
- `ALLOWED_HOSTS`
- секрети через env

2. Встановити залежності:
```bash
pip install gunicorn
```

3. Зібрати статику:
```bash
python manage.py collectstatic --noinput
```

4. Накатити міграції:
```bash
python manage.py migrate
```

5. Запустити app server:
```bash
gunicorn config.wsgi:application --bind 127.0.0.1:8000 --workers 4
```

#### Приклад Nginx (спрощено):

```nginx
server {
    listen 80;
    server_name example.com;

    location /static/ {
        alias /srv/app/staticfiles/;
    }

    location /media/ {
        alias /srv/app/media/;
    }

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

#### Для ASGI-деплою:

```bash
uvicorn config.asgi:application --host 127.0.0.1 --port 8000 --workers 4
```

#### Обовʼязковий security/ops чекліст:

1. HTTPS (Let’s Encrypt), secure cookies.
2. `CSRF_TRUSTED_ORIGINS` і коректні proxy headers.
3. `SECURE_HSTS_SECONDS`, `SECURE_SSL_REDIRECT`, `X_FRAME_OPTIONS`.
4. Логи, моніторинг, алерти (error rate, latency, DB health).
5. Backup та перевірений rollback-план.

#### Типові помилки:

1. `DEBUG=True` у production.
2. Відсутність `collectstatic` перед релізом.
3. Невірні `ALLOWED_HOSTS` / proxy headers -> CSRF/redirect проблеми.
4. Запуск Django development server (`runserver`) у production.

#### Висновок:

Стабільний деплой Django - це звʼязка Nginx + WSGI/ASGI app server + правильні
production-налаштування безпеки й observability. Технічно нескладно, якщо є
дисципліна релізного процесу.

<summary>98. Як масштабувати Django-додаток?</summary>

#### Django

Масштабування Django - це не один крок, а послідовність рішень на рівні
коду, БД, кешу, черг і інфраструктури. Найкраща стратегія - масштабувати за
даними метрик, а не "про запас".

#### 1. Оптимізація застосунку (перший етап)

1. Прибрати N+1 запити (`select_related`, `prefetch_related`).
2. Оптимізувати важкі endpoint-и (pagination, ліміти, індекси).
3. Мінімізувати payload API/HTML.
4. Додати кешування там, де є повторювані "гарячі" запити.

#### 2. Кешування і прискорення читання

1. Redis для low-level/per-view/fragment cache.
2. CDN для static/media.
3. HTTP-cache заголовки для публічних ресурсів.

#### 3. Горизонтальне масштабування app-рівня

1. Запуск кількох інстансів Django за load balancer.
2. Stateless app-процеси (сесії в Redis/DB, без локального стану).
3. Автоскейл на основі CPU/RPS/latency.

#### 4. Винесення важких задач у фон

1. Celery/RQ для асинхронних job-ів.
2. Черги для email, звітів, інтеграцій, імпортів.
3. Retry/backoff і idempotency для надійності.

#### 5. Масштабування БД

1. Індекси та query tuning.
2. Connection pooling (PgBouncer).
3. Read replicas для read-heavy сценаріїв.
4. Partitioning/sharding - лише за реальної потреби і чіткої архітектури.

#### 6. Архітектурна еволюція

1. Від "моноліту без меж" -> до модульного моноліту.
2. За потреби виносити окремі вузькі домени в сервіси.
3. Не дробити на мікросервіси передчасно.

#### 7. Observability як умова масштабування

1. Метрики: p95/p99 latency, error rate, queue lag, DB locks.
2. Логи з request-id.
3. Tracing між сервісами/чергами.
4. Алерти на SLA/SLO порушення.

#### Типові помилки:

1. Масштабувати інфру, не оптимізувавши запити.
2. Кешувати без стратегії інвалідації.
3. Ігнорувати вузьке місце БД.
4. Передчасна мікросервісна складність.

#### Висновок:

Масштабування Django - це керований процес: спочатку оптимізувати код і БД,
потім додати кеш і черги, далі горизонтально масштабувати app-інстанси і лише
після цього ускладнювати архітектуру. Метрики мають вести всі рішення.

</details>

<summary>99. Які є підходи до деплою та моніторингу Django-додатка?</summary>

#### Django

Надійний production для Django тримається на двох стовпах:

1. **керований деплой**
2. **повноцінний моніторинг (observability)**

#### Підходи до деплою:

1. **Класичний VM/VPS деплой**
- Nginx + Gunicorn/Uvicorn + systemd + PostgreSQL/Redis.
- Підходить для контрольованого self-hosted середовища.

2. **Container-based (Docker/Kubernetes)**
- Передбачуване середовище, масштабування через репліки.
- Зручно для команди з DevOps-процесами.

3. **PaaS/Managed платформи**
- Швидший запуск, менше операційних витрат.
- Компроміс у гнучкості інфраструктури.

#### Рекомендований релізний pipeline:

1. CI: тести + лінтери + security checks.
2. Build immutable artifact (image).
3. Deploy на staging.
4. Smoke/health checks.
5. Production rollout (blue/green, rolling або canary).
6. Автоматичний rollback при деградації.

#### Що моніторити обовʼязково:

1. **Application metrics**
- RPS, p95/p99 latency, 4xx/5xx error rate.

2. **Infrastructure metrics**
- CPU, RAM, disk, network, pod/container restart rate.

3. **Database metrics**
- slow queries, locks, connections, replication lag.

4. **Queue/worker metrics**
- queue depth, task latency, retry/failure rate.

5. **Logs + tracing**
- structured logs, request-id, distributed tracing.

#### Мінімальний production-набір:

1. Health endpoints (`/health/live`, `/health/ready`).
2. Централізовані логи (ELK/Loki/Cloud logs).
3. APM/metrics (Prometheus + Grafana, Datadog, New Relic, Sentry).
4. Алерти на SLO-порушення, а не тільки на "сервер впав".

#### Практичні рекомендації:

1. Впровадь runbook для інцидентів (хто, що, як відновлює).
2. Тримай міграції backward-compatible для zero/min-downtime релізів.
3. Автоматизуй секрети й конфіг через env/secret manager.
4. Регулярно тренуй rollback і disaster recovery.

#### Типові помилки:

1. Деплой без health checks і rollback-стратегії.
2. Моніторинг лише CPU без бізнес-метрик і latency.
3. Відсутність звʼязку між логами і конкретним request/trace id.
4. Ручні "гарячі" фікси на сервері без репродукції в коді.

#### Висновок:

Успішний деплой Django - це передбачуваний CI/CD процес, а стабільна робота -
це моніторинг, який дає ранній сигнал про деградацію. Без observability навіть
хороший код швидко втрачає керованість у production.

<summary>100. Які patch-релізи зазвичай містять зміни і чому важливо оновлюватися?</summary>

#### Django

Patch-релізи (наприклад, `5.1.2 -> 5.1.3`) зазвичай містять **зворотно сумісні**
виправлення без змін публічного API, які "ламають" застосунок на рівні major/minor.

#### Що зазвичай входить у patch-реліз:

1. **Security fixes**
- Закриття вразливостей (часто критично для production).

2. **Bug fixes**
- Виправлення помилок у ORM, forms, middleware, admin, migrations тощо.

3. **Stability improvements**
- Краща сумісність із залежностями, edge-case виправлення.

4. **Незначні docs/test покращення**
- Без змін архітектурного контракту застосунку.

#### Чому важливо оновлюватися:

1. **Безпека**
- Пропущений security patch може залишити відому уразливість у продакшені.

2. **Надійність**
- Patch-релізи прибирають баги, які можуть проявлятися під навантаженням.

3. **Сумісність**
- Сучасні бібліотеки очікують актуальні patch-версії фреймворку.

4. **Простіший шлях апгрейду**
- Регулярні невеликі оновлення легші за рідкі "великі стрибки".

#### Практичний процес оновлення:

1. Оновити залежність (`pip`/`poetry`/`uv`).
2. Прогнати тести та лінтери.
3. Перевірити changelog/release notes.
4. Розкатати на staging.
5. Випустити в production через контрольований rollout.

#### Частота:

- Перевіряти patch-оновлення регулярно (мінімум щомісяця або після security advisory).
- Для LTS-версій - ставити security patch якомога швидше.

#### Типові помилки:

1. Відкладати patch-оновлення "на потім".
2. Оновлювати в production без staging/CI перевірок.
3. Ігнорувати release notes і потенційні side effects.

#### Висновок:

Patch-релізи - це не "косметика", а операційна гігієна проєкту. Регулярне
оновлення Django знижує security-ризики, підвищує стабільність і робить
майбутні апгрейди передбачуваними.

</details>
